# 2.4 Session 会话管理

> **学习目标**：掌握 Session 中消息、状态、事件、token usage 的管理方式，理解 InMemory、Redis、SQL 三种会话后端的适用场景与选型决策。

---

## 一、为什么需要 Session？

### 1.1 LLM 是无状态的

这是理解 Session 的**第一性原理**：

> LLM API 本身没有"记忆"。每一次调用，你都必须把完整的对话历史发过去，模型才知道上下文。

```
没有 Session 的世界：

第 1 轮：
  你 → LLM：[{role: user, content: "北京天气怎么样？"}]
  LLM → 你：北京 25°C，晴朗

第 2 轮（用户说"那上海呢？"）：
  你 → LLM：[{role: user, content: "那上海呢？"}]
  LLM → 你：???（不知道"那"指什么）

正确做法（带历史）：
  你 → LLM：[
    {role: user,      content: "北京天气怎么样？"},
    {role: assistant, content: "北京 25°C，晴朗"},
    {role: user,      content: "那上海呢？"}      ← 必须带上前面的全部历史
  ]
  LLM → 你：上海 26°C，多云  ✅
```

### 1.2 Session 解决的核心问题

Session 就是一个**对话容器**，帮你管理：

1. 每次调 LLM 时，**自动加载历史 messages**
2. 模型回复后，**自动追加到历史**
3. 把每个用户/每次对话**隔离开**（通过 session_id）
4. 持久化（按需）

---

## 二、Session 的四大组成部分

把 Session 想象成一个**信封**，里面装了四样东西：

```
Session（一次对话的完整容器）
│
├── 1. Messages（消息列表）     —— 对话内容
├── 2. State（状态字典）        —— 业务上下文
├── 3. Events（事件流）         —— 执行过程的全量记录
└── 4. Token Usage（令牌消耗）  —— 计费/配额/裁剪依据
```

同时 Session 还有一组**元数据**（不属于上述四部分，但用来定位 Session）：

```
Session 元数据：
  session_id: "abc123"      ← Session 的门牌号
  user_id:    "user_001"    ← 哪个用户的
  app_name:   "my_app"      ← 哪个应用的
  created_at: 2026-06-25 20:30
```

> **关键澄清**：`session_id` / `user_id` 是 Session 的**门牌号**（标识"哪个会话"），不是 State。State 是 Session 内部的**业务数据**（"这个会话目前进展到哪一步"）。

---

### 2.1 Messages — 消息列表

最熟悉的部分，就是对话的来回记录。

```python
messages = [
    {"role": "user",      "content": "北京天气怎么样？"},
    {"role": "assistant", "tool_calls": [{"name": "get_weather", "args": {"city": "北京"}}]},
    {"role": "tool",      "content": '{"temperature": "25°C"}'},
    {"role": "assistant", "content": "北京今天 25°C，天气晴朗。"},
    {"role": "user",      "content": "那上海呢？"},   # 第二轮，靠 Session 保持上下文
]
```

**Runner 每次执行的流程**：

```
1. 从 Session 读出历史 messages
2. 追加新的用户消息
3. 完整 messages 发给 LLM
4. LLM 回复（可能含 tool_calls）
5. 把回复追加回 Session
6. 如果有 tool_calls，执行工具，结果再追加，再调一次 LLM
7. 最终回复追加到 Session
```

---

### 2.2 State — 状态字典（业务数据便签纸）

State 是一个挂在 Session 上的 **key-value 字典**，用来存：

> 不属于对话内容、但这次对话需要知道的业务上下文。

#### 在你的代码里已经用到了

```python
await session_service.create_session(
    app_name=app_name,
    user_id=user_id,
    session_id=session_id,
    state={"user_name": user_id},   # ← 这里就是 State
)
```

#### 业务场景举例

| State 键 | 值 | 用途 |
|----------|-----|------|
| `user_name` | "张三" | 工具/Agent 可读取，知道当前用户是谁 |
| `current_step` | "confirm_order" | 多步骤流程中记录当前进度 |
| `selected_product` | "iPhone 16" | 用户选了商品，后续工具调用要用 |
| `retry_count` | 2 | 工具调用失败计数 |
| `language` | "zh-CN" | 用户偏好语言 |
| `vip_level` | 3 | 影响业务逻辑分支 |

#### State 的核心价值

**工具函数之间通过 State 传递业务数据，而不是让 LLM 从聊天记录里"回忆"。**

```python
# 工具函数里读写 State
def place_order(product: str, tool_context: ToolContext) -> dict:
    user = tool_context.state["user_name"]        # 读
    tool_context.state["order_placed"] = True     # 写
    return {"status": "ok", "user": user}
```

#### Messages vs State 的区别

| | Messages | State |
|--|----------|-------|
| **内容** | 对话说了什么 | 对话的背景数据 |
| **发给 LLM？** | 是（占 token） | 否（不占 token） |
| **谁来读写** | Runner 自动维护 | 工具函数、Agent 主动读写 |
| **形式** | 顺序的消息列表 | 无序的 key-value |

---

### 2.3 Events — 事件流（执行过程的全量记录）

#### Event 不只是 "AI 的思考输出"

Event 是 Runner 把一次执行过程拆成的**细粒度结构化通知**。**任何**在执行过程中发生的事都可能是 Event：

| Event 类型 | 内容 | 来源 |
|-----------|------|------|
| 流式文本 chunk | `partial=True, text="北京"` | LLM 流式输出 |
| 思考过程 | `thought=True` | LLM 的推理思考 |
| 工具调用 | `function_call={name, args}` | Runner 调度 |
| 工具结果 | `function_response={...}` | 本地工具函数返回 |
| Agent 切换 | `transfer_to_agent="specialist"` | 多 Agent 编排 |
| 错误事件 | `error="工具执行超时"` | 任何环节出错 |
| 最终完整回复 | `partial=False, text="完整内容"` | 整合后的最终结果 |

#### 在你的代码里已经在消费 Event 了

```python
async for event in runner.run_async(
    user_id=user_id,
    session_id=session_id,
    new_message=user_content,
):
    if event.partial:
        # 流式文本 chunk
        for part in event.content.parts:
            if part.text:
                print(part.text, end="", flush=True)
    else:
        # 非流式事件
        for part in event.content.parts:
            if part.function_call:    # 工具调用
                print(...)
            elif part.function_response:  # 工具结果
                print(...)
```

#### 一次 `run_async` 的 Event 流示例

```
用户输入："北京天气怎么样？"

Event 1: partial=True,  text="正在"          ← 流式 chunk
Event 2: partial=True,  text="查询..."       ← 流式 chunk
Event 3: partial=False, function_call={name: "get_weather", args: {city: "北京"}}
Event 4: partial=False, function_response={temperature: "25°C", ...}
Event 5: partial=True,  text="北京"          ← 整合后的流式输出
Event 6: partial=True,  text="今天 25°C..."
Event 7: partial=False, text="北京今天 25°C，天气晴朗。"   ← 最终完整回复
```

#### Event vs Message 的区别

|  | Message | Event |
|--|---------|-------|
| **粒度** | 一整条消息 | 一个执行瞬间的切片 |
| **持久化** | 必须存（对话历史） | 可选存（用于审计/回放） |
| **发给 LLM？** | 是 | 否 |
| **用途** | 给 LLM 看的上下文 | 给前端展示 / 监控审计 |

#### Event 的实际用途

1. **流式输出体验**：打字机效果（前端逐字显示）
2. **可观测性**：每次调用了哪个工具、耗时多少、出错没有
3. **审计回放**：完整还原一次对话的执行轨迹
4. **业务统计**：比如你 uiagent 项目里"每个用户用了多少次某个工具"

> **回到你之前的问题**：skills 评测中，"调用了哪个 skill、入参出参、执行耗时" → 这是 Event；"这个会话用过某个 skill 的标记" → 可以存到 State；"用户问的内容和 AI 的最终回复" → 这是 Message。

### 2.4 Token Usage — 令牌消耗

LLM API 是**按 token 计费**的，每次调用都有消耗。

#### 一次对话的 Token Usage 累积

```
第 1 轮：
  input_tokens:  500    ← system prompt + tools 定义 + 用户消息
  output_tokens: 80     ← 模型回复

第 2 轮：
  input_tokens:  650    ← 上一轮全部历史 + 新消息（历史越长越多）
  output_tokens: 120

第 3 轮（触发工具调用，2 次 API 调用）：
  input_tokens:  850 + 950
  output_tokens: 60  + 150

会话累计：input = 2950, output = 410, total = 3360 tokens
```

#### 为什么 Session 要管 Token Usage？

| 场景 | 用途 |
|------|------|
| **成本控制** | 一个用户/一个会话消耗了多少钱 |
| **历史裁剪** | 消息超过 context window（如 128k）前，需要裁剪或摘要 |
| **配额管理** | 给每个用户/部门设 token 额度，超了拒绝 |
| **计费审计** | 企业场景按部门/项目统计消耗 |
| **性能分析** | 哪些 prompt 设计过于浪费 token |

#### 历史裁剪策略（重要）

随着对话变长，messages 越来越多，迟早会超出模型的 context window。常见处理：

```
策略 1：滑动窗口
  只保留最近 N 轮对话，老的丢弃
  优点：简单
  缺点：丢失早期上下文

策略 2：摘要压缩
  把老的对话用 LLM 总结成一段摘要，替代原始消息
  优点：保留语义
  缺点：多一次 LLM 调用，有信息损失

策略 3：分级保留
  system prompt + 重要消息（如用户偏好声明）+ 最近 N 轮
  优点：兼顾上下文和成本
  缺点：要判断"哪条消息重要"
```

> Token Usage 就是判断"该不该裁剪"的依据。

## 三、三种 Session 后端

理解了 Session 装什么之后，再看"装在哪里"。

### 3.1 InMemory（内存）

```python
from google.adk.sessions import InMemorySessionService
session_service = InMemorySessionService()
```

```
┌─────────────────────────────────┐
│       Python 进程内存            │
│                                  │
│  session_abc123:                 │
│    messages: [...]               │
│    state:    {...}               │
│    events:   [...]               │
│    token_usage: {...}            │
│                                  │
│  session_def456:                 │
│    ...                           │
└─────────────────────────────────┘
       ↑
   进程一关，全没了
```

| 维度 | 说明 |
|------|------|
| **延迟** | 极低（直接内存读写） |
| **持久化** | ❌ 进程重启全丢 |
| **多实例** | ❌ 各进程数据隔离 |
| **容量** | 受限于进程内存 |
| **适用场景** | 本地开发、跑 demo、单元测试 |

---

### 3.2 Redis

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   实例 1      │     │   实例 2      │     │   实例 3      │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │                    │                    │
       └────────────────────┼────────────────────┘
                            ▼
                  ┌──────────────────┐
                  │      Redis       │
                  │                  │
                  │  sess:abc → {..} │
                  │  sess:def → {..} │
                  │                  │
                  │  TTL: 2h         │ ← 自动过期
                  └──────────────────┘
```

| 维度 | 说明 |
|------|------|
| **延迟** | 低（< 1ms） |
| **持久化** | ⚠️ 有限（RDB/AOF，但不适合长期审计） |
| **多实例** | ✅ 天然共享 |
| **容量** | 受限于 Redis 集群规模 |
| **TTL** | ✅ 可设过期，自动清理 |

#### 适合的业务场景

| 场景 | 为什么选 Redis |
|------|--------------|
| **多实例部署** | 用户请求可能落到不同实例，需要共享存储 |
| **在线客服 Agent** | 需要毫秒级读取历史 |
| **短生命周期对话** | TTL 自动过期，不用手动清理 |
| **流量高峰** | Redis 抗并发能力强 |
| **你的 uiagent 多 Pod 部署** | Session 必须共享存储，不能存进程内存 |

#### 不适合

- 长期审计回溯（数据会过期）
- 复杂的关联查询（Redis 不擅长 SQL 类查询）

---

### 3.3 SQL（MySQL / PostgreSQL）

```
sessions 表：
┌────────┬──────────┬──────────┬─────────────────┐
│ id     │ user_id  │ state    │ created_at      │
├────────┼──────────┼──────────┼─────────────────┤
│ abc123 │ user_001 │ {json}   │ 2026-06-25 20:30│
└────────┴──────────┴──────────┴─────────────────┘

messages 表：
┌────┬──────────┬──────┬────────┬────────┐
│ id │ sess_id  │ role │content │ tokens │
├────┼──────────┼──────┼────────┼────────┤
│ 1  │ abc123   │ user │ 你好   │ 5      │
│ 2  │ abc123   │ asst │ 你好！ │ 8      │
└────┴──────────┴──────┴────────┴────────┘

events 表：
┌────┬──────────┬───────────┬────────────────┐
│ id │ sess_id  │ type      │ payload        │
├────┼──────────┼───────────┼────────────────┤
│ 1  │ abc123   │ tool_call │ {name: "..."}  │
└────┴──────────┴───────────┴────────────────┘
```

| 维度 | 说明 |
|------|------|
| **延迟** | 中（10-100ms 量级） |
| **持久化** | ✅ 长期可靠 |
| **多实例** | ✅ 共享 |
| **容量** | 大（可扩展到 TB） |
| **查询能力** | ✅ 强（聚合、关联、复杂条件） |

#### 适合的业务场景

| 场景 | 为什么选 SQL |
|------|------------|
| **合规审计** | 保留所有对话，可追溯、可导出 |
| **会话分析** | "过去 30 天 get_weather 被调用了多少次" |
| **skills 评测** | 每个会话用了哪些 skill、效果如何 |
| **计费系统** | 按用户/部门统计 token 消耗 |
| **长期对话** | 跨天、跨周的项目助手 |

#### 不适合

- 极致延迟要求（< 5ms）
- 海量短生命周期数据（用 Redis 更合适）

### 3.4 生产环境常见组合：Redis + SQL 分层

```
              用户请求
                  │
                  ▼
            ┌───────────┐
            │   App     │
            └─────┬─────┘
                  │
       ┌──────────┴──────────┐
       ▼ 热数据（最近 2h）      ▼ 冷数据（历史）
  ┌─────────┐            ┌─────────┐
  │  Redis  │ ──归档──▶  │   SQL   │
  └─────────┘            └─────────┘
   毫秒级读写              持久化 + 可查询
```

**职责划分**：
- **Redis**：承载活跃对话的实时读写
- **SQL**：归档历史对话，支持分析、审计、计费

这和你在 uiagent-management 项目里"MongoDB 存全量 + 后续考虑 Redis 缓存 pureUsernames"的思路是一样的：**热数据缓存，冷数据持久化**。

## 四、Session 的完整生命周期（结合你的代码）

```
你的 main.py 执行流程中 Session 的生命周期：

1. 创建 Session
   session_service.create_session(session_id, state={"user_name": ...})
   → 分配 ID，初始化空 messages，设置初始 state
   
2. 用户输入 "北京天气怎么样？"
   → runner.run_async(session_id, new_message=...)
   
3. Runner 内部一次循环：
   ┌──────────────────────────────────────────────┐
   │ a. 从 Session 加载历史 messages                │
   │ b. 追加用户消息                                │
   │ c. 读取 state（如果工具需要）                  │
   │ d. 调 LLM API（带完整 messages + tools）      │
   │ e. 模型返回 tool_calls → 执行工具              │
   │ f. 工具结果追加到 messages                     │
   │ g. 再调 LLM → 生成最终回复                    │
   │ h. 回复追加到 messages                         │
   │ i. 累加 token_usage                            │
   │ j. 产出 events（yield 给 async for）           │
   │ k. 保存 Session（messages + state + usage）   │
   └──────────────────────────────────────────────┘

4. 用户输入 "那上海呢？"
   → 重复步骤 3，但 messages 已带上一轮历史
   → 模型看到历史，理解"那"指上海的天气

5. 用户输入 /clear
   → 创建新 session_id，旧 Session 不再使用
   → 新 Session messages 为空 → 上下文重置
```

## 五、后端选型决策树

```
是不是只做 demo / 本地开发 / 单元测试？
├── 是 → InMemory（最简单）
└── 否
    │
    需不需要长期保留对话（审计、分析、计费）？
    ├── 不需要（只是活跃会话） → Redis
    │                            （快、TTL 自动清理、多实例共享）
    │
    └── 需要
        │
        对实时性要求高吗？
        ├── 极高（毫秒级） → Redis + SQL 分层
        │                   （热数据 Redis，冷数据归档 SQL）
        │
        └── 一般 → SQL
                  （持久化、可查询、可审计）
```

## 六、关键认知总结

| 维度 | 核心要点 |
|------|---------|
| **Session 是什么** | 一次对话的容器，装着 Messages、State、Events、Token Usage |
| **为什么需要** | LLM 无状态，每次调用必须带完整历史 |
| **Messages** | 对话来回内容，发给 LLM，占 token |
| **State** | 业务上下文便签纸，不发给 LLM，工具读写 |
| **Events** | 执行过程的细粒度记录，用于流式输出/审计/可观测 |
| **Token Usage** | 计费/配额/裁剪的依据 |
| **InMemory** | 开发测试用，重启即丢 |
| **Redis** | 活跃会话热数据，毫秒级，多实例共享 |
| **SQL** | 长期持久化，可审计、可分析 |
| **生产组合** | Redis（热）+ SQL（冷）分层 |

---

## 七、常见认知误区澄清

| ❌ 误区 | ✅ 正解 |
|---------|--------|
| State 就是 user_id / session_id | 那是 Session 的元数据（门牌号），State 是业务数据（便签纸） |
| Event 就是 AI 的思考输出 | Event 是整个执行管道的全量记录，包括工具调用、Agent 切换、错误等 |
| Session 主从同步是大问题 | 每个用户操作自己的会话，天然隔离，主从同步不是主要矛盾 |
| 历史消息可以无限增长 | 受 context window 限制，需要裁剪/摘要 |
| InMemory 完全没用 | 开发测试/单进程短生命周期场景下最简单实用 |
