# 2.3 Function Calling 与工具调用

> 从底层 HTTP 协议到框架封装的完整拆解 —— 理解 LLM 如何"调用"本地函数，以及 `FunctionTool` 在其中的抽象边界。

---

## 一、核心认知：LLM 不会"调用"工具

这是理解 Function Calling 的起点：

> **LLM 永远不可能自动调用工具。** 它只能根据你给它的工具定义，**输出一段结构化 JSON**，告诉你："我想调用 `xxx` 函数，参数是 `yyy`"。真正执行函数的，永远是你的代码。

所以 Function Calling 的本质是：

```
LLM 输出格式化 JSON  →  你的代码解析 JSON  →  你的代码调用本地函数  →  你的代码把结果回传 LLM
```

整个流程是一个**循环**，至少需要 **2 次 API 调用**：

1. 第 1 次：发请求带上 `tools` 定义，模型返回 `tool_calls`
2. 第 2 次：你执行完工具，把结果塞回 `messages`，模型整合结果生成自然语言

---

## 二、第一层：裸金属层 — Function Calling 的真实面目

不使用任何框架，仅用 OpenAI SDK 写完整流程：

```
┌────────────────────────────────────────────────────────────────────┐
│                    Function Calling 的本质                          │
│                                                                    │
│   你的代码                        LLM API                          │
│   ┌─────┐                        ┌─────┐                          │
│   │  1  │──── messages + tools ──▶│     │                          │
│   │     │                        │  2  │ 模型看到 tools 定义       │
│   │     │                        │     │ 判断"需要调用 get_weather"│
│   │     │◀── tool_calls JSON ────│  3  │ 返回结构化 JSON           │
│   │  4  │ 解析 JSON              │     │                          │
│   │     │ 执行本地函数            │     │                          │
│   │  5  │── tool result ────────▶│     │                          │
│   │     │                        │  6  │ 整合结果生成自然语言       │
│   │     │◀── 最终回答 ───────────│     │                          │
│   └─────┘                        └─────┘                          │
│                                                                    │
│   整个过程至少 2 次 API 调用，你的代码负责"中间调度"                │
└────────────────────────────────────────────────────────────────────┘
```

裸 OpenAI SDK 实现：

```python
import json
from openai import OpenAI

client = OpenAI()

# ========== 第 1 步：告诉模型"你有哪些工具" ==========
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取指定城市的天气信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "城市名称，如 '北京'"
                    }
                },
                "required": ["city"]
            }
        }
    }
]

# 本地真实函数
def get_weather(city: str) -> dict:
    return {"city": city, "temperature": "25°C", "condition": "晴"}

messages = [{"role": "user", "content": "北京天气怎么样？"}]

# ========== 第 2-3 步：模型返回 tool_calls ==========
response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    tools=tools,
)

choice = response.choices[0]

# ========== 第 4 步：解析并执行 ==========
if choice.finish_reason == "tool_calls":
    for tool_call in choice.message.tool_calls:
        func_name = tool_call.function.name                    # "get_weather"
        func_args = json.loads(tool_call.function.arguments)   # {"city": "北京"}

        # 函数路由：本质就是 if/elif 或 dict dispatch
        if func_name == "get_weather":
            result = get_weather(**func_args)

        # ========== 第 5 步：把结果塞回 messages ==========
        messages.append(choice.message)
        messages.append({
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": json.dumps(result),
        })

# ========== 第 6 步：再请求一次，模型整合结果 ==========
final_response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    tools=tools,
)
print(final_response.choices[0].message.content)
# → "北京今天天气晴，气温25°C。"
```

**关键事实**：

- LLM 不可能自动调用工具 —— 它只输出 JSON
- 本质是格式化提取 —— 但现在 API 已经定义了 `tool_calls` 结构化格式
- 给 LLM 的 `tools` 定义就是 prompt 的一部分 —— 会被序列化进 context，计入 token
- 循环判断 + 调本地方法 —— 就是 `if/elif` 分发

## 三、第二层：API 帮你做了什么

早期（GPT-3.5 之前），需要在 prompt 里手写"请按 JSON 格式输出"，然后用正则解析。现在 API 层面已经把这个问题解决了：

| 演进阶段 | 你需要做的 | API 帮你做的 |
|---------|-----------|-------------|
| **史前时代** | prompt 里写格式要求 + 正则解析输出 | 无 |
| **Function Calling v1** | 定义 tools JSON Schema | 模型输出结构化 `tool_calls`，保证是合法 JSON |
| **Strict Mode** | 加 `strict: true` | 模型输出 **100% 符合 Schema**（用 CFG 约束解码） |
| **Parallel Tool Calls** | 无需额外配置 | 模型一次返回多个 `tool_calls`，你并行执行 |

关键变化：**从"求模型按格式输出"变成了"模型保证按格式输出"**。但"解析 → 路由 → 执行 → 回传"这个循环始终需要你写。

## 四、第三层：框架封装 — FunctionTool 做了什么

这是框架的核心价值。看 `quickstart_python/agent/tools.py` 里的写法：

```python
def get_weather(city: str) -> dict:
    """获取指定城市的天气信息

    Args:
        city: 城市名称，如 "北京"、"Beijing"
    """
    return {"city": city, "temperature": "25°C", "condition": "晴"}
```

这就是全部 —— 一个普通的 Python 函数。剩下的所有事都被 `FunctionTool` 接管：

```
你写的普通函数 get_weather(city: str) -> dict
        │
        ▼
┌─ FunctionTool(get_weather) ─────────────────────────────────────────┐
│                                                                     │
│  ① __init__: 自动提取元数据                                          │
│     • name ← func.__name__ → "get_weather"                          │
│     • description ← func.__doc__ → "获取指定城市的天气信息"          │
│                                                                     │
│  ② _get_declaration: 自动生成 JSON Schema                            │
│     • inspect.signature(func) 反射 → 参数 city: str                  │
│     • 自动生成符合 OpenAI/Gemini 规范的 FunctionDeclaration:         │
│       {                                                             │
│         "name": "get_weather",                                      │
│         "description": "获取指定城市的天气信息",                     │
│         "parameters": {                                             │
│           "type": "object",                                         │
│           "properties": {                                           │
│             "city": {"type": "string", "description": "城市名称"}   │
│           },                                                        │
│           "required": ["city"]                                      │
│         }                                                           │
│       }                                                             │
│     • 自动过滤 tool_context / input_stream 等框架内部参数            │
│                                                                     │
│  ③ process_request: 注册到 LLM 请求                                  │
│     • FunctionDeclaration → llm_request.config.tools                │
│     • self → llm_request.tools_dict[name]（用于路由）                │
│                                                                     │
│  ④ run_async → _run_async_impl: 执行调用                             │
│     • 拿到模型返回的 args: {"city": "北京"}                          │
│     • 检查必填参数 → 缺了返回 {"error": "..."}                       │
│     • 自动注入 tool_context（如果函数签名有的话）                    │
│     • dict → Pydantic Model 自动转换                                │
│     • 同步函数 → asyncio.to_thread | 异步函数 → await               │
│     • 返回结果（Pydantic → JSON 自动序列化）                         │
│                                                                     │
│  ⑤ 错误处理: 软错误策略                                              │
│     • 不抛异常，返回 {"error": "缺少参数 xxx, 请重试"}               │
│     • 模型收到错误后可以自动修正参数重新调用                          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**对照表：裸写 vs 框架**

| 你要做的事 | 裸写 OpenAI SDK | tRPC-Agent FunctionTool |
|-----------|----------------|------------------------|
| 定义 JSON Schema | 手写 JSON，容易漏/错 | **自动从函数签名 + docstring 生成** |
| 函数路由（name → 函数） | `if/elif` 或 dict dispatch | **框架内部 `tools_dict[name]` 自动路由** |
| 参数解析 | `json.loads()` + 手动拆 | **自动解析 + Pydantic 类型转换** |
| 必填参数校验 | 自己写 | **自动检测 + 返回引导错误信息** |
| 结果回传 | 手动组装 `{"role": "tool", ...}` | **框架自动塞回 messages** |
| 异步处理 | 自己管 | **自动判断 sync/async，sync 用 to_thread** |
| 错误处理 | 自己 try/catch | **软错误，不中断 Agent，模型可重试** |
| 循环（可能多次工具调用） | 自己写 while loop | **Runner 的 while 循环自动处理** |

## 五、第四层：Runner 把这一切串起来

Runner 的 `run_async` 内部实际上就是工具调用循环的封装：

```
Runner.run_async() 内部：

while True:  # ← 这就是你说的"循环"
    │
    ├── 1. 收集所有 tool.process_request()
    │      → 每个 FunctionTool 的 Schema 注入 llm_request.config.tools
    │      → 同时把 tool 注册到 tools_dict（用于后续路由）
    │
    ├── 2. 调用 model.generate()
    │      → 发送 messages + tools 给 LLM API
    │
    ├── 3. 检查返回
    │      ├── 有 function_call?
    │      │     → 从 tools_dict[name] 找到 FunctionTool
    │      │     → 调用 tool.run_async(args=模型给的参数)
    │      │     → 把结果塞回 messages（function_response）
    │      │     → continue（回到循环顶部，再次调用模型）
    │      │
    │      ├── 有 transfer_to_agent?
    │      │     → 切换 agent，continue
    │      │
    │      └── 是文本回答?
    │            → yield Event，break
    │
    └── （循环结束 → post_turn_processing → 摘要/记忆）
```

**所以"在循环中判断对应的工具，然后调本地方法"这个过程，Runner 已经替你写完了**。

## 六、四层抽象的完整视图

```
┌─────────────────────────────────────────────────────────────────┐
│  第 4 层：Runner                                                 │
│  └─ 把整个工具调用循环（while loop）封装成一个 run_async()        │
├─────────────────────────────────────────────────────────────────┤
│  第 3 层：FunctionTool                                           │
│  └─ Schema 自动生成、路由、参数解析、错误处理、sync/async 适配    │
├─────────────────────────────────────────────────────────────────┤
│  第 2 层：LLM API                                                │
│  └─ 提供 tool_calls 结构化输出、strict mode 保证格式             │
├─────────────────────────────────────────────────────────────────┤
│  第 1 层：裸金属                                                 │
│  └─ messages + tools → tool_calls JSON → 你解析 → 你调函数        │
└─────────────────────────────────────────────────────────────────┘
```

每往上一层，你少写一些代码，但应该清楚每一层都帮你做了什么。

## 七、关键概念回顾

| 概念 | 说明 |
|------|------|
| **工具 Schema** | JSON Schema 格式的工具签名（name + description + parameters），是 LLM 看到的"工具说明书" |
| **入参解析** | LLM 返回的是 JSON 字符串，需要 `json.loads` 反序列化，框架还会做 Pydantic 类型转换 |
| **返回值** | 工具结果会被塞回 messages，作为 `role=tool` 或 `function_response` 的内容 |
| **错误处理** | 框架采用**软错误**：不抛异常，把错误信息作为结果返回，让模型有机会自我修正 |
| **触发过程** | 模型基于 system prompt + 用户意图 + 工具 description 决定调用哪个工具 |

## 八、实操建议

理解四层抽象后，最有效的学习路径是：

1. **用裸 OpenAI SDK 手写一遍完整的工具调用循环**（不用任何框架），真正感受那个 while loop —— 这一步会让你彻底理解"LLM 不调用工具，只输出 JSON"
2. **再用 `FunctionTool` 重写**，对比代码量差异，体会框架的抽象边界
3. **阅读 Runner 的 `run_async` 源码**，看框架是如何把循环封装起来的

## 九、常见误区

| 误区 | 真相 |
|------|------|
| "LLM 直接调用了我的函数" | 错。LLM 只输出 JSON，函数永远是你的代码调用的 |
| "Function Calling 是新出的特性" | 协议是新的，但本质上和早期 prompt + 正则提取格式化输出是同一件事，只是 API 把这个能力标准化了 |
| "工具定义不算 token" | 错。tools 定义会被序列化进 context，每次请求都消耗 token |
| "FunctionTool 是魔法" | 不是魔法，就是反射 + JSON Schema 生成 + 字典路由的组合 |
| "工具调用只调一次" | 一轮对话可能触发多次工具调用，Runner 内部就是 while 循环驱动的 |

## 十、与上一节的衔接

回顾 `2.2 跑通 Quickstart` 中的代码结构：

- `agent/tools.py` 里写的普通函数 → 被 `FunctionTool` 包装
