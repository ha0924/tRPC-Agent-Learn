# 2.1 Agent 工程化核心概念

> 理解 Agent、Runner、Model、Tool、Session、Memory、Graph 等基础抽象，以及一次 Agent 调用从用户输入到事件流输出的完整链路。

---

## 一、核心认知：Agent = LLM + Harness

Agent 的本质是 **LLM + 工程化治理（Harness/Orchestration）**，通过工程手段将 LLM 的不稳定输出变得稳定、可控、可靠。

治理手段包括：
- **结构化输出约束**（JSON Schema / Function Calling）—— 让 LLM 输出可解析
- **重试与纠错**（Agent Loop 的循环机制）—— 结果不对就再来
- **工具调用验证**（参数校验、权限检查）—— 工程侧的防护
- **上下文管理**（Session / Memory）—— 避免 LLM "失忆" 或 "幻觉"
- **护栏（Guardrails）**—— 安全、合规、质量的兜底机制

完整表述：

> **Agent = LLM + Orchestration（编排） + Tool Use + Memory + Guardrails（护栏）**

---

## 二、Agent vs 工作流

核心区别在于 **谁做决策**：

| 维度 | 工作流 (Workflow) | Agent |
|------|------------------|-------|
| **决策者** | 开发者预定义的流程 | LLM 在运行时动态决定 |
| **路由方式** | if/else、状态机、DAG 固定路径 | 模型根据上下文自主选择 |
| **灵活性** | 低，新需求要改代码 | 高，换 prompt 或加工具就行 |
| **可控性** | 高，行为可预测 | 较低，需要 guardrails |
| **典型实现** | BPMN、Airflow、传统工作流引擎 | ReAct Loop、tRPC-Agent |

**关键洞察**：GraphAgent 是两者的融合 —— 用图的结构约束 Agent 的自由度，兼顾灵活性和可控性。

## 三、七大核心抽象

### 3.1 Agent —— 智能体

Agent 是**声明式**的核心执行单元，它定义 "我是谁"：

- **我的指令是什么**（System Instruction / Prompt）
- **我有什么工具**（Tools 列表）
- **我用什么模型**（Model 配置）

Agent 本身**不关心怎么运行**，它只是一份"能力声明"。运行的事交给 Runner。

### 3.2 Runner —— 运行引擎

Runner 是**独立于 Agent 的运行时引擎**，负责驱动 Agent 的完整执行循环。

类比后端概念：

```
Agent  ≈  Handler（业务逻辑定义）
Runner ≈  HTTP Server / Event Loop（运行时引擎）
```

Runner 的完整职责：

```
┌─────────────────────────────────────────────────────┐
│                     Runner                          │
│                                                     │
│  1. 接收用户输入                                      │
│  2. 加载/创建 Session                                 │
│  3. 召回 Memory                                      │
│  4. 组装完整 Prompt（system + history + memory + user）│
│  5. 调用 Model 推理                                   │
│  6. 解析模型输出：                                     │
│     ├─ 纯文本 → 直接输出                               │
│     └─ Tool Call → 执行工具 → 结果回填 → 再次推理       │
│  7. 循环步骤 5-6 直到终止（无 tool_call 或达 max_turns）│
│  8. 发射事件流（Event Stream）                         │
│  9. 保存 Session 状态                                  │
│  10. 返回结果                                         │
└─────────────────────────────────────────────────────┘
```

**为什么 Runner 和 Agent 分离？**

- 同一个 Agent 可被不同 Runner 驱动（同步 / 异步 / 流式 / 批量）
- 同一个 Runner 可驱动不同 Agent
- **关注点分离** —— Agent 管 "做什么"，Runner 管 "怎么跑"

### 3.3 Model —— 模型抽象

Model 决定了 Agent 的能力上限和下限。在 tRPC-Agent 中，Model 是一个**接口抽象**，不绑定具体模型：

- **可替换**：换模型不改业务代码（混元、DeepSeek、GPT、Claude）
- **可 Fallback**：主模型挂了自动切备用模型
- **可 Routing**：简单问题用小模型省 token，复杂问题用大模型保质量

### 3.4 Tool —— 工具体系

MCP 只是协议，Tool 是抽象概念。工具的分层架构：

```
Tool（抽象概念）
├── FunctionTool     ← 本地函数封装
├── MCPTool          ← 通过 MCP 协议调用远程工具服务
├── Agent-as-Tool    ← 把另一个 Agent 当工具用
└── SkillTool        ← 通过 SKILL.md 封装的可复用能力
```

无论哪种实现，对 LLM 来说看到的都是一样的 **JSON Schema**（名称、描述、参数定义）。模型不知道也不关心工具是本地函数还是远程 API —— 这就是抽象的力量。

### 3.5 Session —— 会话（单次对话的记忆）

```
┌────────────────────────────────────────────────────────┐
│                    Session（会话）                       │
│                                                        │
│  范围：单次会话（一个 session_id 的生命周期内）            │
│  内容：                                                 │
│    • 消息历史（user / assistant / tool 的完整对话）       │
│    • 会话状态（key-value，如当前任务进度）                │
│    • 事件记录（本次会话产生的事件）                       │
│    • Token Usage（本次会话的 token 消耗统计）             │
│  存储：InMemory / Redis / SQL                           │
│  生命周期：会话开始 → 会话结束（可持久化可恢复）          │
│                                                        │
│  类比：浏览器的 Session / HTTP 会话                      │
└────────────────────────────────────────────────────────┘
```

### 3.6 Memory —— 记忆（跨会话的持久记忆）

```
┌────────────────────────────────────────────────────────┐
│                    Memory（记忆）                        │
│                                                        │
│  范围：跨会话（用户级别，不随单次对话消亡）               │
│  内容：                                                 │
│    • 用户偏好（"我喜欢简洁的回答"）                      │
│    • 历史事实（"用户是 Go 开发者"）                      │
│    • 决策记录（"上次选择了方案 A"）                      │
│  操作：                                                 │
│    • 写入：Agent 运行中自动 / 手动沉淀                   │
│    • 召回：下次会话时 Runner 自动检索相关记忆             │
│  存储：向量数据库 / 结构化存储 / Agent DB                 │
│  生命周期：持久化，跨多次会话存在                        │
│                                                        │
│  类比：人的长期记忆 / CRM 里的客户画像                   │
└────────────────────────────────────────────────────────┘
```

**Session vs Memory 一句话区分**：Session 是 "这次聊天记得什么"，Memory 是 "永远记得什么"。

### 3.7 Graph —— 图编排

Graph 不等于传统工作流，它比工作流多了关键能力：

- **节点可以是 Agent**（不只是函数），节点内部自带 Agent Loop 的自主性
- **条件路由可以基于 LLM 判断**（不只是 if/else）
- **支持 Checkpoint + Interrupt/Resume**（可暂停等人审批后继续）
- **状态 Reducer** —— 多个并行分支执行完后的结果合并策略

```
GraphAgent = 工作流的结构性 + Agent 的自主性
```

## 四、一次 Agent 调用的完整链路

以 "查看今天的天气" 为例，展示从用户输入到事件流输出的完整执行链路：

```
                        一次 Agent 调用的完整链路

 ① 用户输入："查看今天的天气"
                    │
                    ▼
 ② Runner.run(user_id, session_id, message)
                    │
                    ▼
 ③ Session.load() ─── 加载历史消息（如果有的话）
                    │
                    ▼
 ④ Memory.recall() ── 召回跨会话记忆（如"用户在深圳"）
                    │
                    ▼
 ⑤ 构造完整 Prompt:
    ┌──────────────────────────────────────┐
    │ System: "你是天气助手..."             │
    │ Memory: "用户常在深圳"                │
    │ History: [之前的对话...]              │
    │ User: "查看今天的天气"                │
    │ Tools: [get_weather(city, date)]     │
    └──────────────────────────────────────┘
                    │
                    ▼
 ⑥ Model.generate() ── 第一次推理
    │
    │ 模型输出（结构化 JSON，不是自然语言）：
    │ {
    │   "tool_calls": [{
    │     "name": "get_weather",
    │     "arguments": {"city": "深圳", "date": "2026-06-25"}
    │   }]
    │ }
    │
    ▼
 ⑦ Runner 检测到 tool_call → 执行工具
    │  get_weather("深圳", "2026-06-25")
    │  → 返回: {"temp": "32°C", "weather": "多云", ...}
    │
    ▼
 ⑧ 工具结果回填到消息历史，再次调用 Model
    │
    │ 模型输出（自然语言）：
    │ "今天深圳天气多云，气温 32°C，建议带伞。"
    │
    ▼
 ⑨ Runner 检测到无 tool_call → 判定为最终回答
                    │
                    ▼
 ⑩ 事件流输出（SSE / Streaming）：
    Event: TextChunk("今天")
    Event: TextChunk("深圳")
    Event: TextChunk("天气多云...")
    Event: TurnComplete
                    │
                    ▼
 ⑪ Session.save() ── 保存本次对话到会话历史
    Memory.update() ── 如果有值得记住的信息则沉淀
```

### 关键机制解析

| 步骤 | 机制 | 说明 |
|------|------|------|
| ⑥ 模型输出结构化 JSON | **Function Calling** | 模型不是输出自然语言，而是输出符合 JSON Schema 的工具调用指令 |
| ⑦ Runner 执行工具 | **Agent Loop 核心** | Runner 解析 tool_call → 调用工具 → 结果回填 → 再推理 |
| ⑨ 终止判断 | **循环退出条件** | 无 tool_call（模型认为任务完成）或达到 max_turns 阈值 |
| ⑩ 事件流输出 | **SSE（Server-Sent Events）** | 流式输出实现打字机效果，提升用户体验 |

## 五、各抽象之间的关系总览

```
┌─────────────────────────────────────────────────────────────┐
│                         Runner（运行引擎）                    │
│                                                             │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐              │
│   │  Agent   │    │  Model   │    │  Tools   │              │
│   │ (声明式)  │───▶│ (推理)   │───▶│ (执行)   │              │
│   │ 指令+工具 │    │ 能力上限  │    │ 能力扩展  │              │
│   └──────────┘    └──────────┘    └──────────┘              │
│         │                                                   │
│         │         ┌──────────┐    ┌──────────┐              │
│         └────────▶│ Session  │    │  Memory  │              │
│                   │ (单次会话)│    │ (跨会话)  │              │
│                   │ 消息历史  │    │ 长期记忆  │              │
│                   └──────────┘    └──────────┘              │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                   Graph（图编排）                     │   │
│   │   [Agent A] ──▶ [Agent B] ──▶ [Tool Node]          │   │
│   │       │              │                               │   │
│   │       └──── 条件路由 ─┘                               │   │
│   └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

## 六、小结

| 概念 | 一句话定义 | 类比 |
|------|----------|------|
| **Agent** | 封装了指令、模型、工具的声明式能力单元 | Handler / Controller |
| **Runner** | 驱动 Agent 执行循环的运行时引擎 | HTTP Server / Event Loop |
| **Model** | 统一的 LLM 接口抽象，决定能力天花板 | 数据库驱动（MySQL/PG 可切换） |
| **Tool** | Agent 可调用的外部能力，对模型透明 | RPC 接口 / 微服务 |
| **Session** | 单次会话的消息历史与状态管理 | HTTP Session / Cookie |
| **Memory** | 跨会话的持久记忆，用户级别 | CRM 客户画像 / 用户偏好表 |
| **Graph** | 以 DAG 编排多个 Agent/节点的执行流 | 工作流引擎 + 动态路由 |
