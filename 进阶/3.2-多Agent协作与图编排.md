# 3.2 多 Agent 协作与图编排

> **学习目标**：理解多 Agent 系统的两大范式（Graph-based / Agent-based）；掌握 Chain、Parallel、Cycle 三种图拓扑；区分 GraphAgent 与 TeamAgent 的本质差异；理解节点、边、条件路由、状态 Reducer、Checkpoint、Interrupt / Resume 等编排机制；建立从"传统工作流"到"Agent 工作流"的概念迁移路径。

---

## 一、为什么需要多 Agent 协作？

### 1.1 单 Agent 的能力天花板

经过前面的学习，一个单 Agent 已经能做很多事：

```
单 Agent 的能力清单：
  ✅ 对话（LLM）
  ✅ 记住上下文（Session）
  ✅ 调用工具（Function Calling）
  ✅ 检索知识（RAG）
  ✅ 长期记忆（Memory）
```

但当任务复杂到一定程度，单 Agent 开始力不从心：

```
场景：用户上传一份 50 页的财报 PDF，要求生成投资分析报告

单 Agent 的尴尬：
  - 一个 Prompt 既要解析数据，又要做行业对标，又要写报告
  - System Prompt 里塞了 5 类指令，LLM 注意力被稀释
  - 任何一步出错，整条链路就乱
  - 工具列表越塞越多（数据库查询、网页搜索、表格解析、图表生成...）
    → LLM 容易在工具选择上出错
  - 调试时无法定位"是哪一环出问题"
```

**根本矛盾**：一个 LLM 调用 = 一个职责的最佳实践。任务越复杂，越需要拆解。

### 1.2 多 Agent 的核心思想

```
让多个"小而专"的 Agent 协作，
代替一个"大而全"的 Agent 单打独斗。

类比：
  传统软件：一个 1000 行的函数 vs 拆成 10 个 100 行的小函数
  团队管理：一个全栈打十个 vs 前端/后端/测试分工协作
  多 Agent：一个 Agent 做所有 vs 多个专职 Agent 分工 + 协作
```

拆完之后立刻面临一个新问题：**多个 Agent 怎么协作？** 这就是多 Agent 编排（Multi-Agent Orchestration）要解决的核心问题。

---

## 二、两大范式：Graph-based vs Agent-based

多 Agent 协作的所有具体方案，本质上都落在这两种范式里。先把这个分类讲清楚，后面所有概念都是它们的展开。

### 2.1 范式 1：Graph-based（图编排）—— 流程在代码里

```
开发者在代码里画一张"流程图"：
  - 哪些节点（每个节点是一个 Agent 或一个函数）
  - 节点之间怎么连（边）
  - 满足什么条件走哪条边（条件路由）

运行时严格按图执行，不需要"指挥官"。
```

**典型示意**：

```
[需求分析Agent] → [代码生成Agent] → [代码审查Agent] → 输出
         ↑                                  │
         └────────── 审查不通过 ─────────────┘
```

**核心特征**：

- 流程**设计时**确定，写在代码里
- 行为**可预测**、可重放、易调试
- 适合**结构化**任务（流程相对固定）

**典型框架**：LangGraph、tRPC-Agent 的 GraphAgent

### 2.2 范式 2：Agent-based（智能体编排）—— 流程在 LLM 脑子里

```
有一个"队长 Agent"（LLM 驱动）：
  - 队长看到任务后，自己决定调用哪些"成员 Agent"
  - 决定调用顺序、几轮调用、什么时候停止
  - 整个编排逻辑由 LLM 在运行时动态生成
```

**典型示意**：

```
              用户任务
                 │
                 ▼
          ┌─ 队长 Agent ─┐  ← LLM 决策中心
          │              │
          ▼              ▼
     [研究员 Agent]  [代码员 Agent]
          │              │
          └──────┬───────┘
                 ▼
            队长汇总结果
                 │
                 ▼
              返回用户
```

**核心特征**：

- 流程**运行时**由 LLM 生成
- 行为**不可预测**但灵活
- 适合**开放式**任务（流程难以提前定义）

**典型框架**：tRPC-Agent 的 TeamAgent、Claude Code 的 SubAgent 调度、AutoGen

### 2.3 对比一句话总结

| 维度 | Graph-based | Agent-based |
|------|-------------|-------------|
| **编排逻辑写在哪** | 代码里（设计时） | LLM 脑子里（运行时） |
| **流程可预测性** | 高 | 低 |
| **灵活性** | 低（改流程要改代码） | 高（队长会动态调整） |
| **可调试性** | 强（流程图清晰） | 弱（决策黑盒） |
| **成本** | 低（无队长 LLM 开销） | 高（每次决策都要 LLM 调用） |
| **适合场景** | 结构化业务流程 | 开放式探索任务 |

**实战中的现实**：两种范式可以**嵌套**。比如一个 Graph 的某个节点里放一个 TeamAgent，反之亦然。这是后期才需要纠结的，先把两种基础范式掌握。

## 三、Graph-based 详解

### 3.1 图的三种基本拓扑

很多人误以为 Chain / Parallel / Cycle 是"三种不同的 Agent 形式"，**错**。它们是**图的三种基本结构**，可以在同一个 GraphAgent 里混合使用。

#### 3.1.1 Chain（链式）—— 串行流水线

```
A → B → C → D

上一个节点的输出 = 下一个节点的输入
```

**例子**：

```
用户需求
   │
   ▼
[需求分析 Agent]   输出：结构化需求 JSON
   │
   ▼
[代码生成 Agent]   输入：需求 JSON / 输出：代码
   │
   ▼
[代码审查 Agent]   输入：代码 / 输出：审查报告
   │
   ▼
最终交付
```

**适用场景**：流程线性，每一步依赖上一步的输出。

#### 3.1.2 Parallel（并行）—— 分支与汇聚

```
         ┌→ B ─┐
A ────→ │     │ → D（合并）
         └→ C ─┘
```

**例子**：

```
        一篇文章
           │
   ┌───────┼───────┐
   ▼       ▼       ▼
[情感   [摘要    [关键词
 分析]   生成]    提取]
   │       │       │
   └───────┼───────┘
           ▼
     [汇聚节点]
           │
           ▼
      综合分析报告
```

**适用场景**：

- 多个子任务彼此独立
- 想用并发节省总耗时
- 需要多视角综合（投票 / 多专家意见）

**关键技术点**：汇聚节点怎么"合并"多个分支的结果？ → 这就是 **Reducer** 的职责（见 3.3）。

#### 3.1.3 Cycle（循环）—— 条件回跳

```
A → B → C ─── 满足条件 ──→ 输出
        │
        └── 不满足 ─→ 回到 B（重新执行）
```

**例子 1：代码自修复循环**

```
[写代码 Agent]
       │
       ▼
[运行测试]
       │
       ▼
   测试通过？
   ┌──┴──┐
   是    否
   │     │
   ▼     ▼
 输出   回到[写代码 Agent]
        （把测试报错信息带上，让它修）
```

**例子 2：ReAct 模式的循环本质**

ReAct（Reasoning + Acting）模式其实就是一个 Cycle：

```
[思考] → [行动（调工具）] → [观察结果] → 任务完成？
   ↑                                      │
   └────────── 没完成 ────────────────────┘
```

**关键技术点**：必须有**退出条件**，否则会死循环 → 通常需要设置 **最大迭代次数** + **完成判断**。

#### 3.1.4 三种拓扑可以混合

一个完整的 GraphAgent 通常**不是**纯 Chain 或纯 Parallel，而是混合的：

```
                    输入
                     │
                     ▼
              [意图识别 Agent]
                     │
              ┌──────┴──────┐
              ▼              ▼          ← 这里是"条件路由"
        意图=技术      意图=客服
              │              │
              ▼              ▼
       ┌──[查文档]    [常见问题FAQ]    ← 技术分支用了并行
       ├──[查代码]
       └──[查论坛]
              │              │
              └──────┬───────┘
                     ▼
              [整合答案 Agent]
                     │
                     ▼
                 质量够好？
                ┌────┴────┐                ← 这里是 Cycle
                是        否
                │         │
                ▼         └→ 回到[整合答案]
              输出           带上质量反馈
```

### 3.2 节点（Node）和边（Edge）

来自你做工作流平台的老朋友。在 Agent 图编排里它们的具体含义：

#### 3.2.1 节点（Node）

**节点 = 一个可执行单元**，可以是：

| 节点类型 | 内容 | 例子 |
|---------|------|------|
| Agent 节点 | 一个完整的子 Agent | 需求分析 Agent、代码生成 Agent |
| 工具节点 | 一个纯函数调用 | 调用数据库查询、调用 HTTP 接口 |
| 处理节点 | 数据转换 / 格式化 | JSON 解析、字段裁剪 |
| 控制节点 | 流程控制 | 条件判断、起点（START）、终点（END） |

```
节点的统一抽象：
  接收：当前 state
  执行：自己的逻辑
  返回：对 state 的更新
```

#### 3.2.2 边（Edge）

**边 = 节点之间的连接关系**，决定执行顺序。两种基本形式：

```
普通边（Direct Edge）：
  A ─→ B
  A 执行完直接跳到 B，无条件

条件边（Conditional Edge）：
  A ─→ ???
       ├─ condition_1 → B
       ├─ condition_2 → C
       └─ default     → D
  A 执行完根据条件函数的返回值，路由到不同节点
```

### 3.3 条件路由（Conditional Routing）

你做过工作流平台，这个概念已经懂，只是要把它和 Agent 场景对应起来。

#### 3.3.1 一个条件路由 = 一个判断函数

```python
# 伪代码示意
def route_after_intent(state):
    """根据上游节点产出的 intent 决定下一站"""
    intent = state["intent"]
    if intent == "技术问题":
        return "tech_agent"
    elif intent == "退款申请":
        return "refund_agent"
    else:
        return "chitchat_agent"

graph.add_conditional_edge("intent_agent", route_after_intent)
```

#### 3.3.2 条件路由的几种典型场景

```
1. 意图分流
   [意图识别] → 技术问题 / 客服问题 / 闲聊 → 不同处理 Agent

2. 质量回环
   [生成答案] → 质量够 / 不够 → 输出 / 回到生成

3. 资源选择
   [问题分析] → 简单问题 / 复杂问题 → 用小模型 / 用大模型

4. 多版本 A/B
   [请求入口] → 走老版本 / 走新版本 → 不同 Agent 处理
```

#### 3.3.3 和"普通 if-else"的区别

```
普通 if-else 写在节点内部：
  def some_agent(state):
      result = process(state)
      if result.ok:
          return next_step_1(state)
      else:
          return next_step_2(state)

  问题：流程逻辑和业务逻辑耦合，画不出流程图

条件路由写在图层面：
  graph.add_conditional_edge("agent_x", router_function)

  好处：流程图清晰可视化，路由逻辑独立可测试，
        改流程不用动节点代码
```

**核心理念**：**业务逻辑写在节点里，流程逻辑写在图里**。

### 3.4 状态（State）和 Reducer

这是你最陌生的部分。

#### 3.4.1 全局状态（State）是图编排的"数据载体"

```
图执行过程中，所有节点共享一个 state 对象：

  state = {
    "user_query": "tRPC 怎么配超时？",
    "intent": "技术问题",          ← 节点A产出
    "search_results": [...],       ← 节点B产出
    "draft_answer": "你可以...",    ← 节点C产出
    "messages": [...],             ← 累积的对话历史
    ...
  }

每个节点：
  - 输入：当前的 state
  - 输出：对 state 的更新（新增字段 或 修改字段）
```

**类比**：你做工作流平台时，每个流程实例都有一个 `variables` 或 `context`，节点之间通过它传递数据。Agent 图里的 state 就是这个。

#### 3.4.2 为什么需要 Reducer？

问题来了：**并行分支汇聚时，多个节点同时修改 state，怎么合并？**

```
情景：
        ┌→ [情感Agent]   返回更新：{sentiment: "positive"}
A ────→ │
        └→ [摘要Agent]   返回更新：{summary: "..."}

        ┌→ [搜索Agent A] 返回更新：{results: [doc1, doc2]}
A ────→ │
        └→ [搜索Agent B] 返回更新：{results: [doc3, doc4]}
                                          ↑
                            两个节点都写 results 字段，冲突了！
```

**Reducer 就是定义这个"合并策略"的函数**。

#### 3.4.3 三种最常见的 Reducer 策略

**① 覆盖（Replace / Overwrite）—— 默认策略**

```
旧值：{intent: "技术"}
新值：{intent: "客服"}
结果：{intent: "客服"}   ← 直接覆盖

适用：字段是单一值（用户ID、当前意图、最终答案）
```

**② 追加（Append）—— 列表累加**

```
旧值：{messages: [msg1, msg2]}
新值：{messages: [msg3]}
结果：{messages: [msg1, msg2, msg3]}   ← 追加到末尾

适用：消息历史、操作日志、搜索结果集合

并行汇聚场景：
  分支 A 返回 {results: [doc1, doc2]}
  分支 B 返回 {results: [doc3, doc4]}
  Reducer = append
  结果：{results: [doc1, doc2, doc3, doc4]}
```

**③ 合并（Merge）—— 字典合并**

```
旧值：{user_info: {name: "Alice"}}
新值：{user_info: {age: 30}}
结果：{user_info: {name: "Alice", age: 30}}   ← 深度合并

适用：逐步累积的复合对象（用户画像、配置项）
```

#### 3.4.4 自定义 Reducer：投票 / 取最优

复杂场景需要自定义：

```python
def vote_reducer(old_results, new_results):
    """三个翻译 Agent 的并行结果，按相似度投票选最优"""
    all_translations = old_results + new_results
    # 用某种相似度算法选最多人赞同的版本
    return [select_best_by_voting(all_translations)]

graph.add_state_field("translations", reducer=vote_reducer)
```

#### 3.4.5 类比 Redux

熟悉前端的话，state + reducer 是同一个东西：

```
Redux：
  (oldState, action) → newState

LangGraph / GraphAgent：
  (oldState, nodeOutput) → newState
            ↑
        通过 reducer 函数

不是巧合 —— 图编排本质上就是"分布式状态机"，
和前端的 Redux 是同一种思想的不同应用。
```

### 3.5 持久化：Checkpoint

#### 3.5.1 为什么需要 Checkpoint？

```
一个图有 10 个节点，执行到第 7 个节点时：
  - 服务器宕机 → 重启后能从第 7 个继续，不用从头跑
  - 需要人工审批 → 暂停，审批通过后恢复
  - 出错要回滚 → 回退到第 5 个节点的 state 重试
  - 调试 → 想看每一步的 state 变化
```

**Checkpoint = 在每个节点执行后保存一个"全局快照"**：

```json
{
  "checkpoint_id": "ckpt_abc123",
  "thread_id": "session_xyz",
  "current_node": "node_7_review",
  "state": {
    "user_query": "...",
    "intent": "...",
    "search_results": [...],
    "messages": [...]
  },
  "timestamp": "2026-06-26T14:30:00Z"
}
```

#### 3.5.2 Checkpoint 在 Agent 场景的特殊性

和你做的传统工作流相比，Agent 的 Checkpoint 要复杂得多：

```
传统工作流的状态：
  审批人、申请单号、当前步骤、几个业务字段
  → 几 KB 的结构化数据

Agent 的状态：
  + 完整的对话消息历史（messages）
  + 工具调用记录（含中间结果）
  + 检索到的文档片段
  + LLM 的思考链（chain of thought）
  → 可能几十 KB ~ 几 MB 的半结构化数据
```

**工程要点**：

- 存储后端选择：内存（开发）/ Redis（中量）/ Postgres / MongoDB（生产）
- 序列化：state 里可能有不能直接序列化的对象（如自定义类），要规范化为 JSON 友好结构
- 命名隔离：Checkpoint 一般按 `thread_id`（一次对话 / 一次任务）分组，避免互相覆盖

#### 3.5.3 Checkpoint 的两种使用方式

```
方式 A：保存所有节点的 Checkpoint（全量历史）
  → 调试友好、可任意时间旅行
  → 存储成本高

方式 B：只保存"最新"和"关键节点"的 Checkpoint
  → 节省存储
  → 适合生产环境
```

### 3.6 Interrupt / Resume：暂停与恢复

#### 3.6.1 最经典的场景：Human-in-the-Loop（人机协作）

```
[数据收集Agent] → [分析Agent] → [决策Agent] → 🛑 INTERRUPT
                                                    │
                                              checkpoint 落库
                                                    │
                                              通知人工审批
                                                    │
                                              （等待几分钟到几天）
                                                    │
                                              审批通过 ✅
                                                    │
                                                 RESUME
                                                    │
                                              从 checkpoint 恢复
                                                    │
                                                    ▼
                                             [执行Agent] → 输出
```

#### 3.6.2 工作机制（和你工作流平台的审批节点一样）

```
1. 图执行到标记了 interrupt_before 或 interrupt_after 的节点
2. 框架暂停执行
3. 把当前 state 写入 Checkpoint
4. 通知外部系统（webhook / 消息 / 改状态码等用户来轮询）
5. 外部决策完成后，调用 resume(checkpoint_id, decision_input)
6. 框架加载 Checkpoint，把 decision_input 注入 state
7. 从中断点继续执行后续节点
```

#### 3.6.3 Interrupt 的两种触发方式

```
① 静态声明（设计时定义）
   graph.add_node("approval", ..., interrupt_before=True)
   → 每次执行到这个节点都暂停

② 动态触发（运行时由节点自己决定）
   def some_node(state):
       if state["risk_score"] > 0.8:
           raise InterruptError("风险过高，需要人工介入")
   → 只有满足条件才暂停（动态人审）
```

#### 3.6.4 其他 Interrupt 场景

不只是人工审批：

```
1. 等待外部异步任务
   触发了一个长耗时数据处理 → 暂停 → 处理完回调 resume

2. 等待用户补充信息
   信息不够 → 暂停问用户 → 用户回复后 resume

3. 多 Agent 等待协作方
   等待另一个 Agent 的输出 → 暂停 → 对方完成后 resume

4. 限流 / 节流
   触发限流 → 暂停 X 秒 → 自动 resume
```

## 四、Agent-based 详解：TeamAgent

### 4.1 TeamAgent 的核心结构

```
                  ┌──────────────────┐
                  │   队长 Agent     │ ← LLM 驱动的"调度大脑"
                  │  (Coordinator /  │
                  │   Supervisor)    │
                  └────────┬─────────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ 成员 A   │ │ 成员 B   │ │ 成员 C   │
       │ (专长 X) │ │ (专长 Y) │ │ (专长 Z) │
       └──────────┘ └──────────┘ └──────────┘
```

**关键角色**：

| 角色 | 职责 |
|------|------|
| 队长 / Supervisor | 拆解任务、决定调谁、判断完没完、综合结果 |
| 成员 Agent | 各自有明确专长，被队长调度时执行 |

### 4.2 队长 Agent 的工作循环

队长本质上是一个**带工具的普通 Agent**，但它的"工具"是"调用其他 Agent"。

```
循环逻辑（伪代码）：

1. 队长收到用户任务
2. 队长 LLM 推理：
   "这个任务怎么拆？应该调谁？什么顺序？"
3. 队长输出 Action：调用成员 A
4. 成员 A 执行，返回结果
5. 队长 LLM 推理：
   "成员 A 的结果够吗？还需要调谁？"
   → 不够 → 继续调成员 B / 重新调成员 A 改进
   → 够了 → 综合结果 → 返回最终答案
```

**和单 Agent 的 ReAct 循环对比**：

```
单 Agent ReAct：
  Thought → Action（调工具）→ Observation → Thought → ...

队长 Agent：
  Thought → Action（调成员 Agent）→ Observation（成员的输出）→ Thought → ...

本质完全一样，只是工具变成了 Agent。
"调用一个成员 Agent" = "调用一个特殊的、复杂的工具"
```

### 4.3 队长怎么"知道"有哪些成员？

这是 TeamAgent 工程实现的关键。两种主流做法：

#### 4.3.1 方式 A：写在 System Prompt 里

```
队长的 System Prompt：

你是一个团队队长，可以调度以下成员 Agent：

1. researcher_agent
   - 专长：信息检索、网页搜索、文档查找
   - 何时调用：需要查找事实、最新信息时
   
2. coder_agent
   - 专长：写代码、调试、运行代码
   - 何时调用：需要编写或修改代码时

3. reviewer_agent
   - 专长：代码审查、报告审查
   - 何时调用：需要检查质量时

请根据用户任务，输出 JSON 决定下一步：
{
  "next_agent": "researcher_agent" | "coder_agent" | "reviewer_agent" | "DONE",
  "instruction": "给该成员的具体指令",
  "reason": "为什么调这个"
}
```

#### 4.3.2 方式 B：把每个成员封装成 Tool

```
每个成员 Agent 暴露一个 Tool Schema：

tool_researcher = {
  "name": "call_researcher",
  "description": "调用研究员 Agent 做信息检索",
  "parameters": {
    "task": "string - 给研究员的具体任务"
  }
}

队长用标准的 Function Calling 机制调用这些 Tool。
LLM 不需要知道"内部是个 Agent"，
它只看到一组工具，按需调用即可。
```

**对比**：

| 维度 | 方式 A（Prompt 描述） | 方式 B（Tool 封装） |
|------|----------------------|--------------------|
| 实现复杂度 | 低（拼字符串） | 中（标准 Tool 协议） |
| 与 Function Calling 兼容 | 弱 | 强 |
| 框架支持 | 自定义 | 主流 Agent 框架原生支持 |
| 调试 | 看不到结构化决策 | 工具调用日志清晰 |

**实战推荐**：方式 B（Tool 封装）。一是和现有 Function Calling 生态兼容，二是支持模型原生的工具调用能力，三是把"调成员"统一为"调工具"，概念更一致。

### 4.4 队长怎么判断"够了"？

这是 TeamAgent 的难点，因为没有显式的结束条件，全靠 LLM 自我判断。常见做法：

```
1. LLM 自判断（最简单也最不可靠）
   System Prompt 写："任务完成时输出 next_agent=DONE"
   问题：LLM 可能过早 DONE，也可能死循环

2. 最大轮次限制（兜底）
   max_iterations = 10
   超过就强制结束，由队长综合现有结果

3. 显式完成度信号
   要求每个成员返回 {result, confidence, is_complete}
   队长根据 confidence 判断

4. 外部裁判（最严谨）
   引入一个"质检 Agent"专门判断任务是否完成
   类似 Cycle 里的退出条件判断
```

### 4.5 TeamAgent 的两种典型变体

#### 4.5.1 Supervisor 模式（中心化）

```
        Supervisor
       /    |    \
      ▼     ▼     ▼
     A     B     C
     
特点：所有调度都过队长
适合：任务分工明确、需要严格控制
代表：本节默认讲的模式
```

#### 4.5.2 Swarm 模式（去中心化）

```
   A ←──────→ B
   ↑          ↑
   │          │
   └────────→ C
   
特点：成员之间可以互相调用
      谁完成自己的部分，主动把任务传给下一个
适合：协作流程灵活、成员之间有领域重叠
代表：Anthropic 的 Swarm 框架、CrewAI
```

**取舍**：

- Supervisor：可控性强、可调试，但所有决策都过队长，成本高、可能成为瓶颈
- Swarm：灵活、并发性强，但失控风险大、调试难

工程实战中 **90% 的场景 Supervisor 模式够用**。

### 4.6 TeamAgent 的代价

千万不要被"动态调度"的灵活性迷惑，TeamAgent 是有代价的：

```
代价 1：成本高
  每一轮决策都要调队长 LLM
  10 轮任务 = 至少 10 次队长调用 + N 次成员调用
  → token 成本比 Graph 模式高一个数量级

代价 2：延迟高
  每次决策都要等 LLM 返回（秒级）
  10 轮决策 = 10 秒以上的串行等待
  → 实时性差

代价 3：行为不可预测
  LLM 决策有随机性 → 同一个任务可能走不同路径
  → 生产事故难以复现
  → 单元测试无法覆盖所有路径

代价 4：失控风险
  LLM 决策可能死循环、可能漏调关键成员、可能反复调同一个成员
  → 必须有 max_iterations + 监控
```

## 五、Graph vs Team：什么时候用哪个？

```
你的任务流程：

  能用代码描述清楚每一步吗？
        │
   ┌────┴────┐
   │         │
  能         不能
   │         │
   ▼         ▼
 GraphAgent  TeamAgent
            （让 LLM 动态决策）
   │
   ▼
 流程中需要"动态判断"的地方
 用 LLM 节点 + 条件路由解决
```

**决策建议**：

```
✅ 用 GraphAgent 的场景：
  - 业务流程相对固定（客服分流、订单处理、内容审核）
  - 需要稳定输出（生产环境的关键链路）
  - 成本敏感（QPS 高的服务）
  - 需要可视化流程（团队协作开发）

✅ 用 TeamAgent 的场景：
  - 开放式问题（研究、写报告、多步推理）
  - 流程难提前预知（用户需求差异大）
  - 容忍较高延迟和成本（Toolkit 类工具、Copilot 类产品）
  - 探索性任务（Claude Code、ChatGPT Code Interpreter）

✅ 混合用：
  - 总体框架用 GraphAgent
  - 某个"需要灵活探索"的节点内部嵌一个 TeamAgent
  - 反之亦然
```

## 七、容易混淆的点

### 7.1 GraphAgent 里的节点能不能是 Agent？

**能，而且经常是**。

```
节点的内容可以是：
  - 一个工具函数（最简单）
  - 一个 LLM 调用
  - 一个完整的子 Agent（带自己的工具、Memory）
  - 一个 TeamAgent（嵌套）
  - 一个 GraphAgent（嵌套）

GraphAgent 提供的是"流程编排"，不限制节点是什么。
```

### 7.2 TeamAgent 和 Function Calling 有啥区别？

```
表面上看：
  Function Calling 是 LLM 调"工具"
  TeamAgent 是队长 LLM 调"成员 Agent"

本质上：
  "成员 Agent" 在工程实现上就是一个特殊的"工具"
  TeamAgent 是 Function Calling 的一种特化用法

区别在于：
  - 普通 Tool：通常是无状态的纯函数（查天气、算数学）
  - Agent-as-Tool：有状态、有自己的 LLM 调用、可能内部还嵌着工具链

所以可以理解为：TeamAgent = Function Calling + 把 Agent 包装成 Tool
```

### 7.3 状态 Reducer 和"消息历史"是一回事吗？

```
不是一个东西，但消息历史是 state 里的一个字段。

state = {
  "messages": [...],     ← 消息历史是 state 里的一个字段，Reducer 通常是 Append
  "intent": "...",        ← 当前意图是 state 里的另一个字段，Reducer 通常是 Replace
  "search_results": [...],← 不同字段配不同的 Reducer 策略
}

Reducer 是"state 字段的合并策略"，
消息历史只是其中一个字段（虽然是最重要的字段之一）。
```

### 7.4 Checkpoint 和 Memory 是同一个东西吗？

```
不是。两者目的完全不同：

Checkpoint（执行快照）：
  目的：技术性的"程序状态持久化"，为了能断点续跑
  内容：当前执行到哪个节点、state 是什么
  生命周期：任务完成就可以删
  类比：游戏存档

Memory（长期记忆）：
  目的：业务性的"用户画像沉淀"，为了跨会话记住用户
  内容：用户偏好、历史决策、画像信息
  生命周期：跟着用户走（天~永久）
  类比：客户档案

两者完全可以并存。一个 TeamAgent 跑任务时：
  - 每个节点后写 Checkpoint（万一宕机能恢复）
  - 任务完成后从对话中沉淀 Memory（下次还能用）
```

### 7.5 多个 Agent 共享 Memory / Session 吗？

```
看设计：

共享方案 A：全团队共享一个 Session
  优点：成员之间能看到彼此的对话历史
  缺点：消息历史可能爆炸（多个成员都在写）

共享方案 B：每个成员有自己的 Session
  优点：边界清晰、消息精简
  缺点：成员之间不知道彼此做了啥，要靠队长转述

实战常见混合：
  Session 隔离（每个成员独立）
  + 共享 state 字段传必要信息
  + 队长汇总记录到自己的 Session
```

### 7.6 Cycle 会不会死循环？怎么防？

```
会，而且很常见。三层防护：

第 1 层：硬性最大迭代次数
  graph.set_max_iterations(10)
  超过就强制结束

第 2 层：基于状态的退出条件
  if state["quality_score"] > 0.9: return END
  if state["error_count"] > 3: return END

第 3 层：智能停滞检测
  if state["last_output"] == state["last_last_output"]:
      # 连续两轮输出一样，说明卡住了
      return END
```

## 九、知识沉淀小结

| 类别 | 关键点 |
|------|--------|
| **两大范式** | Graph-based（流程在代码里） + Agent-based（流程在 LLM 脑子里） |
| **图的三种拓扑** | Chain（串行）/ Parallel（并行）/ Cycle（循环），可以混合使用 |
| **节点与边** | 节点 = 可执行单元（Agent / 工具 / 函数）；边 = 连接关系（普通边 + 条件边） |
| **条件路由** | 业务逻辑写在节点里，流程逻辑写在图里；路由本质是一个判断函数 |
| **State + Reducer** | 全局共享 state，Reducer 定义"多个节点更新同一字段时的合并策略" |
| **Reducer 三类** | Replace（覆盖）/ Append（追加）/ Merge（合并）+ 自定义 |
| **Checkpoint** | 节点执行后的全局快照，支持断点续跑、回滚、调试 |
| **Interrupt / Resume** | 图执行中暂停 → 落 Checkpoint → 外部决策 → 从断点恢复（Human-in-the-Loop 的核心） |
| **TeamAgent** | 队长 Agent（LLM 决策）+ 成员 Agent（各有专长）；队长本质就是一个"以 Agent 为工具的 ReAct Agent" |
| **Graph vs Team** | Graph 流程固定、可预测、低成本；Team 流程灵活、高成本、不可预测；可嵌套 |
| **常见误区** | Chain/Parallel/Cycle 不是三种独立形式而是图拓扑；Checkpoint ≠ Memory；Agent-as-Tool 是 TeamAgent 的实现底座 |
