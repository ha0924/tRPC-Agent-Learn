# 3.3 MCP、A2A 与 AG-UI 协议 + Agent 服务化

> **学习目标**：理解 Agent 生态中三个关键协议各自的定位与解决的问题——MCP（接工具）、A2A（接其他 Agent）、AG-UI（接前端）；建立"三个方向、三个协议"的全景认知；理解 Agent 如何通过 FastAPI / Gateway Server / Go Server 暴露为线上服务。

---

## 一、为什么需要协议？

### 1.1 Agent 不是孤岛

到目前为止我们写的 Agent，更多是一个"自闭"的程序：在本地起一个进程，跑一段 LLM 推理，调用几个本地函数当工具。但真实业务中，Agent 必然要**和外部世界打交道**：

```
一个真实业务里的 Agent 必须能：

  向下 ──→ 调用各种工具（数据库、API、文件、内部服务）
  横向 ──→ 和其他 Agent 协作（公司内的不同 Agent，甚至跨公司）
  向上 ──→ 把过程和结果展示给前端（流式输出、工具调用进度、状态变化）
  对外 ──→ 把自己作为服务暴露给业务系统调用（HTTP API、RPC）
```

每个方向都涉及"两个系统之间怎么对话"——这就是协议要解决的事情。

### 1.2 没有协议会怎样

```
没有 MCP：
  你写的工具想给 LangChain 用要写一遍
              想给 tRPC-Agent 用要再写一遍
              想给 Claude Desktop 用还要再写一遍
  → 工具提供方苦不堪言，工具生态碎片化

没有 A2A：
  公司里 A 团队的 Agent 想调 B 团队的 Agent
  → 两边商量一套私有接口，换个团队又要重新商量
  → 跨公司的 Agent 互通几乎不可能

没有 AG-UI：
  每个 Agent 框架自定义事件格式
  → 前端要为每种 Agent 写一套适配器
  → 切框架就要重写前端
```

协议本质就是把"双方约定"标准化，让生态可以横向扩展。

### 1.3 三个协议的全景定位

```
                    ┌──────────────────────────┐
                    │        用户 / 前端         │
                    └──────────┬───────────────┘
                               │
                          AG-UI 协议
                     （Agent → 前端事件流）
                               │
                    ┌──────────▼───────────────┐
                    │        Agent A           │
                    └───┬──────────────┬───────┘
                        │              │
                   MCP 协议         A2A 协议
                 (Agent → 工具)    (Agent ↔ Agent)
                        │              │
                        ▼              ▼
                 ┌──────────┐   ┌──────────┐
                 │ MCP Tool │   │ Agent B  │
                 │ Server   │   │（远程）   │
                 └──────────┘   └──────────┘
```

**一句话记住**：
- **MCP** 向下接工具
- **A2A** 横向接 Agent
- **AG-UI** 向上接前端

三个方向、三个协议、各管各的，互不冲突。

## 二、MCP 协议：Agent 接工具的标准

### 2.1 解决什么问题

之前的 Function Calling 解决了"LLM 怎么表达要调工具"，但"工具怎么暴露自己"这一段，各框架自己定义，导致工具和框架紧耦合。

```
Function Calling = LLM 和 Agent 框架之间的协议
                   （"LLM 告诉框架我要调什么"的格式）

MCP              = Agent 框架和工具服务之间的协议
                   （"工具告诉框架我能干什么 + 框架告诉工具去干"的格式）

调用链：
  LLM ─Function Calling─▶ Agent 框架 ─MCP─▶ 工具服务
```

MCP 之于工具，就像 USB 之于外设：**工具只实现一次 MCP Server，所有支持 MCP 的 Agent 都能用**。

#### 直觉理解：没有公共规范时的痛点

举个例子帮助理解"为什么需要 MCP 这种公共规范"。假设你写了一个"代码搜索工具"，工具内部用 OpenAI 协议和 LLM 通信。现在别的项目想用你这个工具，怎么办？

在 MCP 出现之前，实际选项都不好用：

| 方式 | 问题 |
|------|------|
| 直接 import 你的 Python 模块 | 语言绑定、版本耦合，对方必须也用 Python |
| 你暴露一个 REST API | 对方能调，但 Agent 不知道你有哪些能力、参数是什么、返回格式怎样——**需要人肉对接** |
| 写成 OpenAI Function Calling 的 Schema | 只能给 OpenAI 的 Agent 用，换 Claude/Gemini 又得改 |

**本质问题**：每个工具作者都在"自造轮子"来描述"我能做什么、怎么调用我、我返回什么"——没有公共规范。

MCP 就是把这件事统一了：**不管你的工具内部用什么（OpenAI API、数据库、爬虫……），对外统一用一套协议来暴露能力**。任何 MCP Client（不管是 Claude、你自己的 Agent、还是别人的项目）都能用标准方式发现你的能力、调用你的工具、拿到标准格式的返回。

> 这个思路同样适用于理解 A2A——只是 A2A 解决的是"Agent 之间怎么协作"，比"工具怎么暴露能力"更复杂一个量级（详见 3.2 节的差异对比）。

### 2.2 核心架构：Client / Server 分离

```
┌───────────────────────┐        ┌───────────────────────┐
│      Agent 框架        │        │      MCP Server       │
│                        │        │   (独立进程/远程服务)   │
│  ┌──────────────┐     │  通信   │                       │
│  │  MCP Client  │◄────┼────────┼──▶ 暴露工具能力       │
│  └──────────────┘     │  JSON-RPC                       │
└───────────────────────┘        └───────────────────────┘
```

- **MCP Server**：工具提供方，实现工具逻辑，暴露标准接口
- **MCP Client**：嵌入在 Agent 框架里，负责连接 Server、发送请求、接收结果
- **通信协议**：JSON-RPC 2.0（统一的消息格式）
- **传输层**：本地用 stdio（管道），远程用 HTTP+SSE

### 2.3 MCP Server 暴露的三类能力

```
┌──────────────┬──────────────┬───────────────┐
│    Tools     │  Resources   │   Prompts     │
│   (工具)      │  (资源)       │  (提示词模板)  │
├──────────────┼──────────────┼───────────────┤
│ 有副作用      │ 只读数据      │ 预设指令       │
│ 例：发邮件    │ 例：读配置    │ 例：审查模板   │
│ 例：写数据库  │ 例：查日志    │               │
└──────────────┴──────────────┴───────────────┘

实际使用中 Tools 占 90%+，另两个是补充能力。
```

### 2.4 工作流程（简化版）

```
阶段 1：初始化
  Client → Server: initialize（协商协议版本）
  Server → Client: 返回支持的能力

阶段 2：能力发现
  Client → Server: tools/list（你有哪些工具？）
  Server → Client: 返回工具 Schema 列表
  → Agent 框架把这些 Schema 拼进 LLM 请求里

阶段 3：工具调用（可多次）
  LLM 决定调用某工具
  Client → Server: tools/call（工具名 + 参数）
  Server → Client: 返回执行结果
  → 回传 LLM 继续推理

阶段 4：关闭
  Client 关闭连接，Server 退出
```

> 关于 MCP 的本地 stdio 传输、管道原理、远程 HTTP+SSE 传输，详见上一篇笔记《从 Function Calling 到 MCP》。

### 2.5 MCP 的价值

```
✅ 工具复用：一次实现，到处运行
✅ 生态繁荣：社区可以贡献大量 MCP Server，所有 Agent 框架直接受益
✅ 解耦：工具提供方和 Agent 框架方各自独立演进
✅ 安全：MCP Server 是独立进程，可以做权限隔离、沙箱化
```

## 三、A2A 协议：Agent 之间的协作标准

### 3.1 解决什么问题

回到 3.2 学过的多 Agent 协作——之前我们讨论的都是**同一个进程内**多个 Agent 协作（一个 Team、一个 Graph）。但真实场景中：

```
跨进程、跨团队、跨公司的 Agent 协作需求：

  - 公司内：业务部门的"客服 Agent" 想调用 IT 部门的"工单 Agent"
  - 跨平台：你的 Agent 想调用别人发布的 Agent 服务
  - 异构框架：A 用 LangGraph，B 用 tRPC-Agent，怎么互通？
  - 长任务：对方 Agent 干活要 5 分钟，怎么持续接收进度？
```

这些都不是单一框架内的 TeamAgent 能解决的——**需要一个跨框架、跨网络的 Agent 通信协议**。这就是 A2A（Agent-to-Agent Protocol，Google 在 2024 年底推出的开放协议）。

### 3.2 A2A 和 MCP 的本质差异

很多人会把 A2A 当成"MCP 的 Agent 版本"，这是误解。两者解决的问题根本不同：

```
┌──────────┬────────────────────────┬────────────────────────┐
│          │       MCP              │       A2A              │
├──────────┼────────────────────────┼────────────────────────┤
│ 角色     │  Agent ↔ 工具          │  Agent ↔ Agent         │
│          │  主从关系               │  对等关系               │
│          │                        │                        │
│ 模型     │  请求 - 响应            │  任务协作              │
│          │  调用就完事              │  有生命周期、有状态     │
│          │                        │                        │
│ 时长     │  通常很快（秒级）        │  可能很长（分钟/小时）  │
│          │                        │                        │
│ 输出     │  返回数据               │  返回 Artifact（制品）  │
│          │                        │  比如：报告、代码、图   │
│          │                        │                        │
│ 能力描述  │  工具 Schema           │  Agent Card（更丰富）   │
│          │                        │  包含技能、认证、定价等 │
│          │                        │                        │
│ 进度推送  │  无（一问一答）          │  有（流式 status 更新）│
└──────────┴────────────────────────┴────────────────────────┘
```

简单类比：
- **MCP** = 老板使唤工具（"帮我查个数据" → 立刻给结果）
- **A2A** = 部门之间合作（"帮我做个调研报告" → 持续沟通进度 → 最后交付制品）

### 3.3 A2A 的核心概念

#### Agent Card（能力名片）

A2A 用 Agent Card 来描述一个 Agent"是谁、能做什么、怎么调"。本质是一个 JSON 文件，通常放在 `https://your-agent.com/.well-known/agent.json`：

```json
{
  "name": "Research Agent",
  "description": "擅长行业调研和市场分析",
  "url": "https://research-agent.example.com/a2a",
  "capabilities": {
    "streaming": true,
    "pushNotifications": true
  },
  "skills": [
    {
      "id": "industry-research",
      "name": "行业调研",
      "description": "给定行业关键词，输出调研报告",
      "examples": ["调研 2026 年 AI 编程工具市场"]
    }
  ],
  "authentication": {
    "schemes": ["bearer"]
  }
}
```

任何 Agent 想调用它，只需要读这个 Card 就知道：地址、能力、认证方式。**这就是 Agent 世界的"服务发现"**。

#### Task（任务对象）

A2A 里所有协作的核心单元是 Task。一个 Task 有完整生命周期：

```
Task 生命周期：

  submitted   ──▶  提交了，等待开始
      │
      ▼
  working     ──▶  正在执行（可以持续推送进度）
      │
      ├──▶ input-required  需要补充输入（人/Agent 介入）
      │
      ▼
  completed   ──▶  成功完成，产出 Artifact
      │
      或
      ▼
  failed      ──▶  失败
      或
  canceled    ──▶  被取消
```

每个 Task 由 **Message**（双方对话消息）和 **Artifact**（最终产出物）组成。

#### 流式状态推送

长任务必须能推送进度，A2A 通过 SSE 来做：

```
客户端 Agent                          服务端 Agent
     │                                     │
     │   POST /tasks/send                  │
     │   {"task_id":"t1","message":"..."}  │
     │────────────────────────────────────▶│
     │                                     │
     │   SSE 流持续推送 status 事件          │
     │◀── status: working, "正在搜索..."  ──│
     │◀── status: working, "正在分析..."  ──│
     │◀── status: working, "正在生成报告" ──│
     │◀── artifact: 报告 PDF ────────────│
     │◀── status: completed ──────────────│
     │                                     │
```

### 3.4 A2A 工作流程

```
完整交互流程：

  1. 服务发现
     Client 请求 Server 的 /.well-known/agent.json
     → 拿到 Agent Card，知道能力和地址

  2. 提交任务
     Client → Server: POST /tasks/send
       Body: { task_id, message: "帮我调研 AI 编程市场" }

  3. 接收进度（SSE 长连接）
     Server 持续推送 status 事件给 Client
     Client 可以转发给前端 / 自己处理 / 转给下一个 Agent

  4. 中间交互（如需要）
     如果 Server 需要更多输入：
       Server → Client: status=input-required, "请提供目标地区"
       Client → Server: POST /tasks/send（继续这个 task_id）

  5. 任务完成
     Server → Client: status=completed
     携带 Artifact（最终产出物，比如 PDF/Markdown/结构化数据）

  6. 后续操作
     Client 可以拿 Artifact 给下个 Agent 或返回给用户
```

### 3.5 A2A 通信机制深入：和普通 HTTP 有何不同

很多人第一次看 A2A 会有个直觉："Agent 之间协作时间可能很长，通信方式应该和普通 HTTP 不一样吧？" 这个直觉完全正确——A2A **底层用的还是 HTTP**，但在 HTTP 之上设计了一套"长任务协作"的机制。

#### 普通 HTTP vs A2A 的通信对比

```
普通 HTTP API 调用：
  Client ──POST /api/search──▶ Server
  Client ◀──200 JSON 结果────── Server
  整个过程：几百毫秒，一来一回就结束

A2A 协议的调用：
  Client ──POST /tasks/send──▶ Server    ← 还是 HTTP！
  Client ◀── SSE 流（持续打开的连接）── Server
            │  data: {"status":"working","msg":"正在搜集资料..."}
            │  data: {"status":"working","msg":"分析中..."}
            │  data: {"status":"working","msg":"正在生成报告..."}
            │  data: {"artifact":{...报告内容...}}
            │  data: {"status":"completed"}
            └── 连接关闭
  整个过程：可能几秒到几十分钟
```

**关键区别**：普通 HTTP 是"发请求 → 等结果 → 连接关了"，A2A 是"发请求 → 保持连接 → 对面一边干活一边往回推状态 → 干完才关"。

#### A2A 实际提供两种通信模式

```
┌─────────────────────────────────────────────────────────────┐
│                  A2A 的两种通信模式                           │
├──────────────────────┬──────────────────────────────────────┤
│  模式一：SSE 流式     │  模式二：Webhook 推送                 │
│  （短中任务，秒~分钟） │  （超长任务，小时~天）                  │
├──────────────────────┼──────────────────────────────────────┤
│  Client ──POST──▶    │  Client ──POST──▶                    │
│  Client ◀──SSE流──   │  Server 返回 202 Accepted（收到了）    │
│  连接一直保持          │  连接立即断开                         │
│  Server 持续推事件     │  ...Server 默默干活...                │
│  完成后连接关闭        │  干完后：                              │
│                      │  Server ──POST──▶ Client 的回调 URL   │
│                      │  "嘿，任务完成了，结果在这"              │
└──────────────────────┴──────────────────────────────────────┘
```

- **模式一（SSE 流式）**：适合几秒到几分钟的任务。连接保持打开，实时推进度。这也是最常用的模式。
- **模式二（Webhook 推送）**：适合真正超长的任务（比如"写一份 50 页的调研报告"，可能要几小时）。HTTP 长连接撑不住，所以 Server 先 ACK 就断开，干完了再主动回调 Client 的 URL。

Agent Card 里会声明自己支持哪种模式：

```json
{
  "capabilities": {
    "streaming": true,           // 支持 SSE 流式
    "pushNotifications": true    // 支持 Webhook 回调
  }
}
```

#### A2A 的 HTTP 端点

A2A 协议本质就是几个普通的 HTTP POST 端点：

```
┌─────────────────────────┬────────────────────────────────┐
│ 端点                     │ 作用                           │
├─────────────────────────┼────────────────────────────────┤
│ GET  /.well-known/      │ 服务发现：拿 Agent Card         │
│      agent.json         │ "你是谁、你能干什么"             │
├─────────────────────────┼────────────────────────────────┤
│ POST /tasks/send        │ 发送任务 / 继续对话              │
│                         │ 等最终结果（可能等很久）          │
├─────────────────────────┼────────────────────────────────┤
│ POST /tasks/sendSubscribe│ 发送任务 + 订阅 SSE 流          │
│                         │ 边干边推进度——**最常用**         │
├─────────────────────────┼────────────────────────────────┤
│ POST /tasks/get         │ 查询任务当前状态                 │
├─────────────────────────┼────────────────────────────────┤
│ POST /tasks/cancel      │ 取消一个正在执行的任务            │
└─────────────────────────┴────────────────────────────────┘
```

#### 一次完整通信的实际数据

```
========== 第一步：服务发现 ==========

GET https://research-agent.example.com/.well-known/agent.json

Response:
{
  "name": "Research Agent",
  "url": "https://research-agent.example.com/a2a",
  "capabilities": { "streaming": true },
  "skills": [{ "id": "research", "name": "行业调研" }]
}

========== 第二步：发任务（开 SSE 流）==========

POST https://research-agent.example.com/a2a/tasks/sendSubscribe
Content-Type: application/json

{
  "jsonrpc": "2.0",             ← A2A 底层用 JSON-RPC 2.0 格式
  "method": "tasks/sendSubscribe",
  "params": {
    "id": "task-001",
    "message": {
      "role": "user",
      "parts": [
        { "type": "text", "text": "调研 2026 年 AI 编程工具市场" }
      ]
    }
  }
}

========== 第三步：SSE 流推送（连接保持打开）==========

← HTTP 200 OK
← Content-Type: text/event-stream

← data: {
←   "result": {
←     "id": "task-001",
←     "status": { "state": "working", "message": "正在搜集行业数据..." }
←   }
← }

← data: {
←   "result": {
←     "id": "task-001",
←     "status": { "state": "working", "message": "已收集 47 个数据源，分析中..." }
←   }
← }

← data: {
←   "result": {
←     "id": "task-001",
←     "artifact": {                    ← 最终产出物
←       "parts": [{
←         "type": "text",
←         "text": "# AI 编程工具市场调研报告\n\n## 一、市场规模..."
←       }]
←     },
←     "status": { "state": "completed" }  ← 任务完成
←   }
← }

← 连接关闭
```

#### 为什么不用 WebSocket？

要长连接，为什么不用 WebSocket？

```
WebSocket：
  ✅ 双向实时通信
  ❌ 不能穿透大多数企业代理/防火墙
  ❌ 不能被标准 HTTP 负载均衡器管理
  ❌ 调试困难（不是纯文本 HTTP）

SSE（Server-Sent Events）：
  ✅ 纯 HTTP，所有基础设施天然支持
  ✅ 穿透代理/CDN 无压力
  ✅ 调试简单（curl 就能看）
  ✅ 单向推送足够（A2A 不需要 Client 边推边收）
  ❌ 只能 Server→Client 单向
```

A2A 的通信模式本质是 **Client 发指令，Server 持续汇报**——不需要双向实时，所以 SSE 完全够用。如果 Client 需要补充信息，那是**新发一个 POST 请求**到同一个 `task_id`，而不是通过同一条连接反向发消息。

#### 一句话总结

```
A2A 的通信底层还是 HTTP，
但在 HTTP 之上封装了：
  · 任务生命周期（submitted → working → completed）
  · SSE 流式推送（实时进度）
  · JSON-RPC 消息格式（统一编码）
  · Webhook 回调（超长任务兜底）

→ 把简单的请求-响应变成了可持续几分钟甚至几小时的协作会话
→ 这就是它和普通 HTTP 远程调用的本质区别
```

### 3.6 A2A 的价值与现状

```
✅ 跨框架协作：LangGraph 的 Agent 可以无缝调 tRPC-Agent 的 Agent
✅ 跨组织生态：未来可能出现"Agent 市场"，按调用付费
✅ 标准能力发现：不再需要私下约定接口
✅ 长任务友好：流式进度、人工介入、最终制品交付都标准化

⚠️ 现状（截至 2026）：
  - 协议本身已稳定，但生态还在早期
  - 大厂在跟进（Google 主推，部分厂商接入）
  - 大多数公司内部仍用私有协议或 HTTP 直连
  - 真正"跨公司 Agent 互通"还需要时间
```

## 四、AG-UI 协议：Agent 向前端输出的标准

### 4.1 解决什么问题

Agent 运行时产生的信息**远不止"最终回答"**：

```
Agent 一次执行过程中，前端可能想看到：

  - LLM 流式吐字（token by token）
  - 当前调用了哪个工具，参数是什么
  - 工具执行结果
  - 中间状态变化（比如 "正在搜索..." → "正在分析..."）
  - 思考过程（Reasoning）
  - 生命周期事件（开始/结束/错误）
  - 用户需要确认的交互节点（Interrupt）
```

如果每个 Agent 框架都自己定义事件格式，前端就要为每种框架写适配器，**框架一换前端就崩**。AG-UI（Agent-User Interface Protocol）就是把"Agent 推给前端的事件流"标准化。

### 4.2 AG-UI 出现的本质背景：Agent 正在变成"独立服务"

如果你写过 Web 应用，可能会有个直觉疑问：

> "Agent 不就是我后端的一部分吗？前端调我后端就行了，为什么还要专门定义一个协议？"

这个问题问得非常好——它正好揭示了 AG-UI 出现的真正背景：**Agent 正在从"嵌在后端里的一段代码"，变成"独立部署的服务"**。

#### 架构演进的三个阶段

```
========== 阶段一：传统 Web 应用 ==========

  浏览器 ──HTTP──▶  后端 ──SQL──▶  数据库
        ◀──JSON──        ◀──行──

  交互：请求-响应，瞬时完成
  返回：结构化 JSON
  → REST/GraphQL 完全够用

========== 阶段二：早期 LLM 应用（耦合架构）==========

  浏览器 ──HTTP/SSE──▶  后端（Agent 逻辑写在里面）──▶  LLM API

  特点：Agent 代码是后端的一部分
  前端：和"自家后端"对接，事件格式怎么定都行
  问题：换 Agent 框架 = 前端跟着重写

========== 阶段三：Agent 即服务（现在的趋势）==========

  浏览器 ──AG-UI──▶  Agent 服务 A（LangGraph）
        ──AG-UI──▶  Agent 服务 B（CrewAI）
        ──AG-UI──▶  Agent 服务 C（tRPC-Agent）

  特点：Agent 是独立部署的服务，可能是别人家的
  前端：只要支持 AG-UI，就能对接任何 Agent
  价值：可插拔、可替换、可组合
```

AG-UI 想做的事，类比一下就是 **Agent 和前端之间的"HTTP + JSON 规范"**——把"Agent 怎么对前端说话"这件事标准化。

#### 但为什么不直接用 REST / GraphQL / WebSocket？

如果 Agent 的输出像传统 API 那样"一次返回 JSON"就完了，确实不需要新协议。**之所以需要 AG-UI，是因为 Agent 的输出形态本质不同**：

| 维度 | 传统 API | Agent |
|------|---------|-------|
| 输出时机 | 一次性返回 | **流式**（token 逐个吐） |
| 输出内容 | 单一结构化数据 | **多模态混合**（文本+工具调用+图表+表单...） |
| 中间状态 | 无（成功/失败） | **多阶段**（思考中、调用工具中、生成中...） |
| 可中断性 | 不需要 | **需要**（用户随时打断） |
| 上下文 | 通常无状态 | **有状态**（session / memory） |
| 双向交互 | 一来一回 | **Agent 可能反问用户、请求授权（HITL）** |

所以前端要展示的，远远不是"一个回答"，而是：

- 实时打字效果（streaming text）
- "正在调用 xxx 工具..."的提示（tool call status）
- 工具结果卡片（tool result rendering）
- Agent 反问的弹窗（human-in-the-loop）
- 中间生成的图表/表格/代码块（generative UI）
- "停止生成"按钮（cancellation）

如果没有 AG-UI，每家 Agent 框架都会自定义一套"我吐什么事件、字段叫什么、什么时候发"——前端对接每一家都要从头写适配器。

#### "可插拔"的真正价值

1. **"独立服务"是大趋势**：Agent 会像微服务一样独立部署，可能跨团队、跨公司提供，前端不再"自家对自家"。
2. **"复杂的过程"具体指什么**：不只是"流"，更关键是"**多类型事件混合的流**"——文本、工具调用、状态变更、交互请求会交错出现。
3. **"可插拔"的真正价值**：换 Agent 不用改前端、换前端不用改 Agent、甚至可以做一个"Agent 调试器/Playground"通用工具来对接任何 Agent。

#### 和 MCP / A2A 的对称性

到这里你会发现，三个协议背后的设计哲学完全对称：

```
MCP    ：标准化"Agent ↔ 工具"的对话    → 工具可插拔
A2A    ：标准化"Agent ↔ Agent"的对话   → Agent 之间可插拔
AG-UI  ：标准化"Agent ↔ 前端"的对话    → 前端可插拔

→ 本质都是同一个思路：把某个方向上的"双方约定"统一化，
  让生态可以横向扩展，避免 N×M 的适配地狱。
```

### 4.3 AG-UI 和普通 SSE 的关系

```
SSE     = 传输方式（服务端单向推送）
AG-UI   = 业务层协议（推什么事件、字段叫什么、什么类型）

类比：
  SSE   像"信封"（怎么寄）
  AG-UI 像"信纸格式"（信里写什么）

AG-UI 通常基于 SSE 传输，但事件 schema 是统一标准的。
```

### 4.4 核心事件类型

AG-UI 定义了一套标准事件，每个事件有 `type` 字段标识种类。简化后大致分五类：

```
1. 生命周期事件
   - RUN_STARTED      Agent 开始执行
   - RUN_FINISHED     Agent 执行结束
   - RUN_ERROR        Agent 执行出错

2. 文本流事件
   - TEXT_MESSAGE_START    一段消息开始
   - TEXT_MESSAGE_CONTENT  增量文本片段（token）
   - TEXT_MESSAGE_END      消息结束

3. 工具调用事件
   - TOOL_CALL_START       工具调用开始（工具名+参数）
   - TOOL_CALL_ARGS        参数流式输出
   - TOOL_CALL_END         调用结束
   - TOOL_CALL_RESULT      返回结果

4. 状态事件
   - STATE_SNAPSHOT        当前完整状态快照
   - STATE_DELTA           状态增量变化

5. 交互事件
   - INTERRUPT             等待用户确认
   - RESUME                用户已确认，恢复执行
```

### 4.5 一个完整事件流示例

用户问"帮我搜下今天的天气并总结"，前端通过 AG-UI 看到的事件序列：

```
→ event: RUN_STARTED
  data: { runId: "r1", threadId: "t1" }

→ event: TEXT_MESSAGE_START
  data: { messageId: "m1", role: "assistant" }

→ event: TEXT_MESSAGE_CONTENT
  data: { messageId: "m1", delta: "好的，" }
→ event: TEXT_MESSAGE_CONTENT
  data: { messageId: "m1", delta: "我来" }
→ event: TEXT_MESSAGE_CONTENT
  data: { messageId: "m1", delta: "查询..." }

→ event: TEXT_MESSAGE_END
  data: { messageId: "m1" }

→ event: TOOL_CALL_START
  data: { toolCallId: "tc1", toolName: "weather_search" }
→ event: TOOL_CALL_ARGS
  data: { toolCallId: "tc1", args: { city: "深圳" } }
→ event: TOOL_CALL_RESULT
  data: { toolCallId: "tc1", result: "深圳今天 28℃，多云" }
→ event: TOOL_CALL_END
  data: { toolCallId: "tc1" }

→ event: TEXT_MESSAGE_START
  data: { messageId: "m2", role: "assistant" }
→ event: TEXT_MESSAGE_CONTENT
  data: { messageId: "m2", delta: "深圳今天 28℃，多云，适合外出。" }
→ event: TEXT_MESSAGE_END
  data: { messageId: "m2" }

→ event: RUN_FINISHED
  data: { runId: "r1" }
```

前端拿到这个流后可以做到：

- 实时打字效果（TEXT_MESSAGE_CONTENT 逐字渲染）
- 工具调用气泡展示（TOOL_CALL_* 系列事件）
- 状态指示器（"AI 正在使用工具..."）
- 错误处理（RUN_ERROR）

### 4.6 AG-UI 的价值

```
✅ 前端通用：写一套渲染逻辑，可以对接所有支持 AG-UI 的 Agent
✅ 体验统一：流式输出、工具可视化、状态展示都标准化
✅ 框架解耦：Agent 框架可以换，前端不动
✅ 工具链复用：可以有标准的 AG-UI 客户端 SDK、调试工具
```

合起来：让 Agent 从孤岛变成生态。

## 六、Agent 服务化：暴露为线上服务

理解了协议之后，还有最后一公里：**怎么把 Agent 部署成可被业务系统/前端调用的线上服务？**

### 6.1 服务化的核心需求

```
一个 Agent 想上线服务，至少要满足：

  1. 网络可达：HTTP / RPC 接口对外暴露
  2. 并发支持：能同时服务多个用户
  3. 会话管理：区分不同用户的对话（Session/Thread）
  4. 流式输出：支持 SSE 推送（配合 AG-UI）
  5. 鉴权：身份验证 + 权限控制
  6. 可观测：日志、指标、追踪
  7. 弹性：限流、超时、熔断
```

不同的部署形态有不同的取舍。

### 6.2 先澄清一个常见误解：Gateway Server 不是"另一种技术"

很多资料（包括我自己最初的笔记）会把这三者并列对比：

```
  FastAPI Server  vs  Gateway Server  vs  Go Server
```

**这种对比是错位的，会误导未来的自己。** 真正要分清的是两个独立维度：

```
═══════════════════════════════════════════════════
"暴露为线上服务" 的本质永远只有一种：

  用某种后端框架（FastAPI / Gin / tRPC-Go / ...）
  写一个 HTTP/SSE Server，输出符合 AG-UI 规范的事件流
═══════════════════════════════════════════════════

变化的是两个正交的维度：

  维度 A —— 用什么实现（技术选型）
    Python：FastAPI / Starlette / Flask
    Go：    net/http / Gin / tRPC-Go
    Node：  Express / Fastify
    → 决定性能上限和开发效率

  维度 B —— 拆成几个服务、各承担什么角色（架构选型）
    单体：1 个服务做全部
    业务网关 + 后端：拆出"接入服务"和"Agent 服务"
    → 决定可维护性和可扩展性
```

**Gateway Server 属于维度 B**，它就是一个**业务网关**——本质上是"用 FastAPI 或 Go 写的、专门承担接入侧通用处理的后端服务"，和 FastAPI 不是同一层概念。

> 容易和它混的是另外两种"网关"：
> - **流量网关**（Nginx / APISIX）：不懂业务，只做反代/限流/TLS 终结
> - **API 网关产品**（AWS API Gateway / 腾讯云 API 网关）：云厂商托管的 SaaS 化网关
>
> 笔记里讨论的 Gateway Server 是**业务网关 / BFF 层**，理解 Agent 协议，做的是 Agent 维度的编排。

### 6.3 业务网关（Gateway Server）的角色定位

既然 Gateway 就是个后端服务，为什么要单独把它拎出来？因为它承担的**职责**是单体服务里没有的：

```
┌─────────────────────────────────────────────────────┐
│  业务网关（Gateway Server）的四大独特职责             │
├─────────────────────────────────────────────────────┤
│ 1. 协议出口统一                                       │
│    后端 Agent 内部可能用 gRPC / tRPC / 自定义 JSON   │
│    Gateway 把这些"翻译"成对外统一的 AG-UI 事件流      │
│                                                      │
│ 2. 会话粘性路由                                       │
│    同一 session 的请求要打到同一个后端实例            │
│    （因为 Agent 状态在那个实例的内存里）              │
│    单体不用考虑、拆服务后必须解决                     │
│                                                      │
│ 3. 跨 Agent 的统一鉴权与流量治理                      │
│    JWT 校验、限流、熔断、超时在 Gateway 一处完成      │
│    下游 Agent 信任入口，不重复实现                    │
│                                                      │
│ 4. 流式连接的特殊处理                                  │
│    SSE 长连接需要禁用 buffer、设置超时、优雅断连       │
│    Gateway 比 Nginx 更懂"什么时候该断、什么时候该续"   │
└─────────────────────────────────────────────────────┘
```

**一句话**：Gateway Server 不是新技术，是新的**角色分工**——这个角色用什么语言/框架实现都行。

### 6.4 演进路径：从单体到网关 + 多后端

下面这三个阶段，**不是三选一的选型**，而是**同一个系统随复杂度上升的形态**。

#### 阶段一：单体 FastAPI 服务

```
  Client ──HTTP/SSE──▶ FastAPI（接入 + Agent 逻辑 + 工具调用）──▶ LLM

  特点：1 个 Python 进程搞定一切，几十行代码起步
  适合：原型验证、内部小工具、个人项目
```

最简代码骨架：

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()
agent = build_my_agent()

@app.post("/chat")
async def chat(req: ChatRequest):
    async def event_stream():
        async for event in agent.run_stream(req.message, session_id=req.session_id):
            # 输出符合 AG-UI 规范的事件
            yield f"event: {event.type}\ndata: {event.json()}\n\n"
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

#### 阶段二：拆出业务网关，后端可异构

```
                   ┌───────────────────────┐
  Client ──────▶  │  Gateway Server       │  ← 也是 FastAPI/Go 写的
                   │  （业务网关）          │     一个普通后端服务
                   │  - 鉴权 / 限流         │
                   │  - 会话路由            │
                   │  - 协议出口 = AG-UI    │
                   └───┬───────┬───────┬───┘
                       ▼       ▼       ▼
                   Agent A  Agent B  Agent C
                   (Python) (Go)    (Python)

  关键认知：
    Gateway 和 Agent A 服务都是用后端框架写的，
    区别只是职责分工——Gateway 只做接入和路由，
    Agent A 只做跑 LLM 和调工具。
    这就是经典的"单体拆微服务"，Agent 领域也一样。

  适合：多 Agent 共存、需要统一鉴权/限流/灰度
```

#### 阶段三：高并发场景，热路径用 Go 重写

```
  Client ──▶  Gateway（Go） ──▶  Agent 实例（Go，tRPC-Agent）
                              ──▶  Agent 实例（Python，复杂逻辑保留）

  Go 的并发模型适合海量 SSE 长连接，单机能扛万级以上连接。
  架构没变，只是把维度 A 的实现换成性能更高的语言。

  适合：高并发、低延迟、强稳定性的生产环境
```

tRPC-Agent 项目本身就是为生产级 Go 部署设计的方向——把 Agent 框架做到 Go，配合 tRPC 这个 RPC 框架，原生支持高并发。

### 6.5 两个维度的正交组合

把维度 A（用什么实现）和维度 B（架构怎么拆）解耦之后，真实场景里可以自由组合：

| 组合 | 架构（维度 B） | 实现（维度 A） | 典型场景 |
|------|----------------|----------------|----------|
| 组合 1 | 单体 | FastAPI | 原型 / Demo / 内部小工具 |
| 组合 2 | 网关 + 后端 | 都用 FastAPI | 中型业务，多 Agent 共存 |
| 组合 3 | 网关 + 后端 | Go 网关 + Python 后端 | 入口要扛量、Agent 逻辑复杂 |
| 组合 4 | 网关 + 后端 | 都用 Go（tRPC-Agent） | 大规模高并发线上服务 |

### 6.6 渐进式演进路径

```
Step 1: FastAPI 单实例跑起来                  → 验证业务
Step 2: 加 Nginx 做流量层（TLS / 反代 / 限速） → 简单上线
Step 3: 流量/Agent 数量上来 → 拆出业务网关     → 接入层独立演进
Step 4: 入口性能瓶颈        → 把网关迁到 Go    → 维度 A 升级
Step 5: 多 Agent 互调       → 引入 A2A 协议    → 走向服务网格
```

**核心心法**：
- **Nginx 是流量层**，负责"管道"层面的转发
- **Gateway Server 是业务层**，负责"理解 Agent 协议"的编排
- 两者不冲突，生产环境通常 **Nginx 在前、Gateway 在后**

## 七、容易混淆的点

### 7.1 MCP vs Function Calling

```
Function Calling：LLM 输出"我要调工具"的格式约定（LLM 侧）
MCP：             工具如何对外暴露能力的约定（工具侧）

两者不替代，是协作关系：
  LLM ─Function Calling─▶ Agent 框架 ─MCP─▶ 工具
```

### 7.2 A2A vs MCP

```
都是"远程调用"，但场景完全不同：

  调用工具 → 用 MCP
    特征：你是老板、对方是干活的、立等可取

  和另一个 Agent 协作 → 用 A2A
    特征：双方对等、可能要持续沟通、产出物是制品
```

### 7.3 AG-UI vs 普通 SSE

```
SSE 是传输方式（怎么寄）
AG-UI 是事件规范（信里写什么）

一个不用 AG-UI 的 SSE：
  data: "嗨"
  data: "世界"
  → 前端要自己解析，自己定义类型

一个标准 AG-UI 的 SSE：
  event: TEXT_MESSAGE_CONTENT
  data: { messageId: "m1", delta: "嗨" }
  → 任何 AG-UI 客户端都能正确处理
```

### 7.4 A2A vs TeamAgent（3.2 学的）

```
TeamAgent：同进程内、单一框架内的多 Agent 协作
            队长在代码里直接 .call(成员) 即可
            适合：内部分工

A2A：       跨进程、跨框架、跨网络的 Agent 协作
            通过 HTTP+SSE 调用远程 Agent
            适合：外部协作、跨团队、跨公司

两者可以叠加：
  TeamAgent 内部某个"成员 Agent"实际是远程 A2A 调用
```

### 7.5 Gateway 和 Nginx 的区别

```
Nginx：通用反向代理，做的是"流量转发"层面的事
       不理解 AG-UI 事件、不理解 Agent 业务

Gateway Server：业务层网关，理解 Agent 业务
                能做协议转换（内部协议 → AG-UI）
                能做 Agent 路由（按 session_id 找实例）
                能做 Agent 维度的限流和监控

简单场景 Nginx 就够，复杂多 Agent 平台需要专门的 Gateway。
```

## 八、知识沉淀小结

```
三个协议的本质：
  MCP   = 工具的"USB 接口"        → 工具生态化
  A2A   = Agent 之间的"通信外交礼仪" → 协作标准化
  AG-UI = Agent 给前端的"事件统一格式" → 体验一致化

服务化的本质：
  把 Agent 从"本地脚本"变成"线上服务"
  = 网络可达 + 并发支持 + 流式输出 + 治理能力

三种部署形态：
  FastAPI    → 快速验证
  Gateway    → 多 Agent 统一接入
  Go Server  → 高并发生产
```

核心认知拼图：

```
  ┌─────────────────────────────────────────────┐
  │                  前端                        │
  └────────────────┬────────────────────────────┘
                   │ AG-UI（事件流标准）
  ┌────────────────▼────────────────────────────┐
  │       服务化层（FastAPI/Gateway/Go）          │
  │       - 鉴权、限流、流式、会话                  │
  └────┬──────────────────────┬─────────────────┘
       │                      │
       │ A2A                  │ MCP
       │（Agent 互通）          │（工具接入）
       ▼                      ▼
   远程 Agent              工具服务
```
