# 3.1 自测题：Memory 与 Knowledge / RAG

> 配套学习文档：[3.1-Memory与Knowledge-RAG.md](./3.1-Memory与Knowledge-RAG.md)
>
> 本题集用于检验对 Session / Memory / Knowledge 三者边界、RAG 流水线、向量检索原理、评估指标等核心知识点的掌握深度。
>
> 共 8 题，覆盖三个层次：**概念辨析、原理推理、工程判断**。
> 答题建议：每题用几句话讲清楚核心思路即可，不用长篇大论。重点是说出"为什么"，而不是复述定义。

---

## Q1（概念辨析）—— Session / Memory / Knowledge 的边界

用户说了一句："我喜欢用 vim 写代码"。

- 这句话会被存到 Session、Memory、还是 Knowledge 里？
- 为什么？
- 如果三个都不存，什么条件下它才会被存？

**考察点**：能否区分"对话原文"和"提炼后的用户特征"，能否区分"个人记忆"和"通用知识"。

## Q2（流程理解）—— RAG 离线与在线两阶段

- RAG 的**离线准备阶段**和**在线服务阶段**，分别有哪几步？
- 如果某一步出问题，会表现为什么症状？

举例填空：
| 出问题的环节 | 用户会看到的现象 |
|------------|----------------|
| 切分 chunk_size 太大 | ? |
| 切分 chunk_size 太小 | ? |
| 向量化模型选错 | ? |
| Top-K 设得太小 | ? |
| 没做 Rerank | ? |

**考察点**：是否真正理解每一步的作用，能否反向从"症状"推回"哪一步出问题"——这是排查 RAG 故障的核心能力。

## Q3（原理推理）—— 为什么 query 和 doc 必须用同一个 Embedding 模型？

文档里写了"query 和 doc 必须用同一个 Embedding 模型"。

- 请解释**为什么**。
- 如果有人反驳："两个模型维度一样不就行了吗？比如都是 1536 维，向量都能算余弦相似度啊"，你怎么回应？

**考察点**：是否理解"向量空间"这个抽象概念，能否解释"维度一样 ≠ 语义对齐"。这是从"记住结论"到"理解原理"的分水岭。

## Q4（工程判断）—— 错误码场景下的检索失效

用户问："错误码 E10024 是什么意思？"

你的 RAG 系统只配了**向量检索（dense retrieval）**，没有配 BM25 / 关键词检索。

- 这种情况下最可能出现什么问题？
- 你会怎么改？

**考察点**：是否理解"向量擅长语义、不擅长精确符号匹配"这个核心局限，是否知道混合检索的应用场景。

## Q5（指标理解）—— 召回率与精确率的计算

知识库里关于"超时配置"的相关文档**共有 10 篇**。

你的检索 Top-K=5，返回了 5 篇，其中 4 篇确实相关，1 篇无关。

- 请计算：召回率（Recall）= ?  精确率（Precision）= ?
- 如果把 K 从 5 调到 20，召回率和精确率分别会怎么变？为什么？
- 进一步思考：能不能通过"无脑增大 K"来提升系统效果？为什么？

**考察点**：能否准确计算指标，能否理解 K 与 Recall / Precision 的权衡关系，能否看穿"无脑大 K"的陷阱。

---

## Q6（Memory 深度）—— 记忆冲突如何处理？

用户上个月说："我喜欢 Python。"
今天说："我现在改用 Go 了。"

- Memory 系统应该怎么处理这种矛盾？（写入策略 / 时间戳 / 覆盖 / 软删除？）
- 如果不处理会怎样？（举一个用户视角的具体糟糕体验）
- 进一步：如果用户说"今天用 Go，但 Python 我还是会用"，又该怎么处理？

**考察点**：是否意识到 Memory 不是"只写不删"的日志，而是需要冲突消解机制；是否理解"覆盖 vs 共存"的判断。

---

## Q7（Rerank 理解）—— 为什么需要两阶段检索？

为什么 RAG 通常采用 **Bi-Encoder 粗排 + Cross-Encoder 精排** 的两阶段架构，而不是直接用 Cross-Encoder 做全量检索？

- Cross-Encoder 的精度更高，为什么不全程用它？
- Bi-Encoder 既然精度差，为什么还要它？

**考察点**：是否理解两类编码器的计算复杂度差异（O(N) vs O(1+K)），是否理解"召回追求广度、精排追求精度"的分工。

---

## Q8（端到端判断）—— RAG 系统排障思路

你构建了一个 RAG 系统，用户反馈："回答不准确。"

- 你怎么定位问题？请给出你的**排查步骤**（至少 3 步，最好覆盖检索层、生成层、数据层）。
- 每一步分别在验证什么假设？

**考察点**：是否具备端到端的系统思维，是否能把"用户感知的问题"逐层拆解到"具体的技术环节"。这是从"会搭 RAG"到"会运维 RAG"的关键能力。
