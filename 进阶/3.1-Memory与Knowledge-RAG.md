# 3.1 Memory 与 Knowledge / RAG

> **学习目标**：掌握 RAG 的完整流程与术语体系，理解文档加载、切分、向量化、检索、召回、Top-K、Rerank、提示词拼接的每一步；区分 Session（短期记忆）、Memory（长期记忆）、Knowledge（外部知识）三者的定位；理解 RAG 工程落地的核心难点与对应方案。

---

## 一、为什么需要 RAG 和 Memory？

### 1.1 LLM 的两个根本局限

继 Session 解决了"LLM 无状态"问题之后，我们还要面对 LLM 的另外两个根本局限：

```
局限 1：训练数据有截止日期
  GPT-4 的知识停留在某个时间点
  → 它不知道你公司内部的最新文档
  → 它不知道昨天上线的新接口
  → 它不知道你这个用户的偏好

局限 2：上下文窗口有限
  即使是 128K 窗口的模型，也塞不下你公司全部文档
  → 不能把全部知识都塞进 Prompt
  → 必须"按需检索"，只塞当下相关的部分
```

**RAG（Retrieval-Augmented Generation，检索增强生成）就是解决这两个局限的工程方案。**

### 1.2 RAG 的核心思想

```
不要让 LLM "记住所有知识"，
而是让 LLM "需要什么知识时，能精准查到"。

类比：
  传统做法：让员工把公司全部文档背下来 → 不现实
  RAG 做法：员工手边放一本能快速翻阅的参考手册 → 用到时翻一下
```

## 二、Session / Memory / Knowledge 的关系

这三个概念都涉及"喂给 LLM 的上下文"，但定位完全不同。先把它们的边界划清楚：

```
                Session（短期记忆）        Memory（长期记忆）          Knowledge（外部知识）

数据从哪来？    本次对话过程中累积          多次对话沉淀提取            人工提前导入的文档
                "用户刚说的话"             "用户的偏好/画像"           "tRPC 官方文档.pdf"

存什么？        当前对话的消息历史          用户级别的个性化信息         通用的领域知识
                + state + events           偏好、习惯、历史决策         API 文档、产品手册

谁写入？        每轮自动追加               Agent 自动沉淀 / 用户主动    管理员提前灌入
                                          要求"记住这个"

检索方式        按 session_id 直接加载     按 user_id 过滤 + 语义匹配  纯语义匹配（按 query）

生命周期        本次对话（分钟~小时）       跟着用户走（天~永久）        跟着知识库版本

类比            你现在桌上的便签            你记得老客户的习惯           你手边的参考手册
```

## 三、RAG 的完整流程

RAG 分为**两个阶段**：离线准备（一次性）+ 在线服务（每次提问）。

### 3.1 阶段 A：离线准备

```
原始文件（PDF / Word / Markdown / 网页 / 数据库记录...）
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 1：文档加载（Document Loading）        │
│                                           │
│ 把各种格式的文件统一变成纯文本             │
│                                           │
│ 输入：tRPC超时配置.pdf（二进制）           │
│ 输出："## 超时配置\n\ntRPC 支持..."（文本）│
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 2：文本切分（Chunking）                │
│                                           │
│ 长文档切成小段                             │
│ "## 超时配置\n..." → [chunk1, chunk2, ...] │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 3：向量化（Embedding）                 │
│                                           │
│ 每个 chunk 调 Embedding 模型 → 一组数字     │
│ chunk1 → [0.12, -0.35, 0.88, ...]         │
│ chunk2 → [0.45, 0.22, -0.67, ...]         │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 4：存入向量数据库（Indexing）          │
│                                           │
│ 向量 + 原文 一起存                         │
│ ┌─────────────────────────────────────┐  │
│ │ id=1  vector=[0.12,-0.35,...]        │  │
│ │       text="tRPC 支持三种超时配置..."  │  │
│ │ id=2  vector=[0.45,0.22,...]         │  │
│ │       text="单接口超时：在代码中..."   │  │
│ └─────────────────────────────────────┘  │
└───────────────────────────────────────────┘
        │
        ▼
      准备完毕，等待被检索
```

### 3.2 阶段 B：在线服务

```
用户提问："tRPC 怎么设置超时？"
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 5：查询向量化                          │
│                                           │
│ 用同一个 Embedding 模型把问题变成向量       │
│ "tRPC 怎么设置超时？" → [0.15, -0.30, ...] │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 6：检索（Retrieval）/ 召回（Recall）   │
│                                           │
│ 在向量数据库里找最相似的向量                │
│ → id=1（相似度 0.92）                      │
│ → id=2（相似度 0.87）                      │
│ → id=7（相似度 0.85）                      │
│ → ...                                     │
│                                           │
│ 注意：返回的是"找到的条目"，不是向量本身    │
│       同时把存着的原文取出来               │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 7：排序（Ranking）+ Top-K 截断          │
│                                           │
│ 按相关性打分排序，取前 K 条                  │
│ K=3 → 留下 id=1, id=2, id=7                │
│                                           │
│ （可选）Rerank：用更精细的模型再打一遍分    │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 8：上下文拼接（Context Assembly）      │
│                                           │
│ 把检索到的原文塞进 Prompt                  │
│                                           │
│ System: "你是 tRPC 技术助手。              │
│         根据以下参考文档回答问题。         │
│                                           │
│ 参考文档：                                 │
│ [1] tRPC 支持三种超时配置方式...           │
│ [2] 单接口超时：在代码中设置...            │
│ [3] 链路超时：通过 context 传递..."        │
│                                           │
│ User: "tRPC 怎么设置超时？"                │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 9：LLM 生成（Generation）              │
│                                           │
│ LLM 基于参考文档 + 用户问题生成回答        │
│ （不靠自己训练时记住的知识瞎编）           │
└───────────────────────────────────────────┘
        │
        ▼
      返回给用户
```

## 四、术语词典

来自信息检索（IR）领域的专业术语，先逐个讲透。

### 4.1 Embedding / 向量化

把一段文本变成一个固定维度的数字向量（比如 1536 维），使得**语义相近的文本，向量也相近**。

```
"超时配置"       → [0.12, -0.35, 0.88, ..., 0.05]
"timeout 设置"   → [0.13, -0.34, 0.87, ..., 0.06]  ← 和上面很接近
"鲜花订购"       → [-0.42, 0.55, 0.11, ..., 0.92]  ← 完全不同
```

向量是**单向**的：

```
文本 → 向量  ✅（Embedding 模型完成）
向量 → 文本  ❌（做不到，向量是有损压缩的数学表示）
```

所以向量数据库会**同时存向量和原文**，检索时用向量找条目，最终返回原文。

### 4.2 Chunk / 切分

把长文档切成小段。每个段是一个 Chunk。

**为什么必须切分**：

1. Embedding 模型有输入长度上限（通常 512 ~ 8192 tokens）
2. 太长的 Chunk 语义混杂，向量化后"什么都像什么都不像"
3. 检索时塞进 Prompt 的内容也要控制长度

### 4.3 Index / 索引

切分 + 向量化后，存入向量数据库形成的**可检索数据结构**就是 Index。

类似数据库的 B+ 树索引，向量数据库用的是专门的**近似最近邻（ANN）索引**：HNSW、IVF、PQ 等算法。让"在百万级向量里找最相似的几个"能在毫秒级完成。

### 4.4 Query / 查询

用户提出的问题，作为检索的输入。

### 4.5 Retrieval / 检索

广义概念，指"从知识库里把相关内容找出来"这个动作。包括召回 + 排序 + Top-K 截断。

### 4.6 Recall / 召回

**"召回"= "把可能相关的都搜出来"**，不是什么高深概念。

之所以用"召回"而不直接说"搜索"，是因为 IR 领域要区分两个阶段：

```
阶段 1：召回（Recall）—— 粗筛
        尽量把相关的都捞回来，宁多勿漏
        从 100 万条 → 几十条候选

阶段 2：排序（Ranking）—— 精排
        对召回结果打分排序，挑出最好的
        从 几十条候选 → Top-K 条
```

**为什么分两步**：全量数据精细打分太慢。先快速粗筛到小集合，再精细排序。

### 4.7 召回率 / 精确率

衡量召回效果的两个核心指标：

```
假设知识库里关于"超时配置"真正相关的文档共 10 篇
你的检索找回了 20 篇，其中 8 篇确实相关

召回率（Recall）= 8 / 10 = 80%
                  找回的相关数 / 全部相关数
                  关注"有没有漏"

精确率（Precision）= 8 / 20 = 40%
                    找回的相关数 / 总共找回数
                    关注"有没有混入噪音"
```

这两个指标通常此消彼长：

- 召回多（K 大）→ 召回率高但精确率低（噪音多）
- 召回少（K 小）→ 精确率高但召回率低（容易漏）

### 4.8 Top-K

排序后只取前 K 条结果。

```
经验值：
  简单问答：K = 3 ~ 5
  复杂分析：K = 5 ~ 10
  代码生成：K = 3（代码上下文要精确，噪音影响大）
```

### 4.9 Ranking / 排序、Rerank / 重排序

```
层次 1：向量余弦相似度（粗排）
  score = cosine(query_vector, doc_vector)
  范围 0~1，越接近 1 越相关
  特点：快（毫秒级），但相对粗糙

层次 2：Rerank 模型（精排）
  用专门的 Cross-Encoder 模型
  把 query 和每个候选文档一起送进模型
  输出更精确的相关性分数
  特点：慢（百毫秒级），但准

层次 3：LLM Judge
  让 LLM 判断："这段文档是否回答了用户的问题？"
  最准，但最贵
```

**Bi-Encoder vs Cross-Encoder**：

```
Bi-Encoder（向量检索用的模型）：
  query  → Encoder → vector_q
  doc    → Encoder → vector_d
  比较：cosine(vector_q, vector_d)
  特点：query 和 doc 分别编码，可预先把 doc 向量化好 → 快

Cross-Encoder（Rerank 用的模型）：
  [query + doc] → Encoder → score
  特点：query 和 doc 拼在一起送进模型，能理解两者关系 → 准但慢
        必须 query 来了才能算，没法预计算
```

典型 Pipeline：

```
100 万文档 ──向量粗排──▶ 50 条 ──Rerank精排──▶ 5 条 ──拼Prompt──▶ LLM
            < 50ms              < 200ms                    800ms~3s
```

### 4.10 Context Assembly / 上下文拼接

把检索到的 Top-K 原文，按一定模板塞进 Prompt。常见模板：

```
System: 你是 XX 助手。请基于以下参考资料回答用户问题。
        如果参考资料里没有相关信息，请直接说"我不知道"，不要编造。

参考资料：
[1] {chunk_1_text}
[2] {chunk_2_text}
[3] {chunk_3_text}

User: {user_query}
```

**注意点**：

- 加上"引用编号"`[1] [2]`，方便后续生成带引用的回答
- 明确指示"没有就说不知道"，降低幻觉
- 控制总长度，不要超过模型上下文窗口

### 4.11 Generation / 生成

LLM 基于拼好的 Prompt 输出回答。这一步和普通的 LLM 调用没区别，只是 Prompt 里多了参考资料。

### 4.12 Memory 相关：写入 / 沉淀 / 召回

```
写入（Write）
  把信息存进 Memory Store
  最直接：用户说"记住我是 Go 开发者" → 直接写

沉淀（Distill / Extract）
  从对话过程中"自动提取"值得长期记住的信息
  通常用 LLM 做提取（在会话结束后跑一遍）

  会话过程：
    用户：我是 Go 开发者，主要用 tRPC 框架
    Agent：好的

  会话后处理（沉淀）：
    Memory Service 调 LLM 分析这次对话
    → 提取："该用户是 Go 开发者，使用 tRPC 框架"
    → 向量化 → 存入 Memory Store（关联 user_id）

召回（Recall）
  新会话开始时，根据当前 query 检索这个用户的 Memory
  把相关的画像信息拼进 Prompt
```

## 五、工程落地的五大难点

概念之外，RAG 真正难的是工程落地。每一个难点都对应着工程经验。

### 难点 1：文档加载没那么简单

不同格式的坑各不相同：

```
PDF 的坑：
  - 扫描版 PDF 是图片，文字要 OCR
  - 表格用坐标画线条，不是结构化数据
  - 页眉页脚、页码混在正文里
  - 多栏排版，阅读顺序难判断

Word 的坑：
  - 嵌入图片里有文字（要不要 OCR）
  - 样式信息（加粗、标题层级）要不要保留

网页的坑：
  - 导航栏、广告、侧边栏都是噪音
  - 正文提取（Boilerplate Removal）

代码文档：
  - 代码块、注释、自然语言混合
  - 切分时不能切断代码逻辑
```

**经验**：早期可以只支持 Markdown（最干净），生产环境再针对性处理 PDF/Word。

### 难点 2：切分不准，语义被截断

```
糟糕的切分（按固定 200 字符）：
┌─────────────────────┐
│ Chunk 1:            │
│ tRPC 支持三种超时配  │ ← 句子被截断了！
└─────────────────────┘
┌─────────────────────┐
│ Chunk 2:            │
│ 置方式：             │ ← 上文没了，读不懂
│ 1. 全局超时...      │
└─────────────────────┘
```

**常见切分策略**：

| 策略 | 做法 | 适用 |
|------|------|------|
| 固定大小 | 每 N 字符切一刀 | 最简单，但最不准 |
| 按段落 | 以空行 / 换行为边界 | 一般文档 |
| 按标题层级 | 以 `##` `###` 为边界 | Markdown / 技术文档 |
| 递归切分 | 大标题切 → 太长再按段落 → 还太长按句子 | 通用性最好 |
| 语义切分 | 用模型判断"话题转换点" | 最准，最贵 |

**Overlap 重叠是最实用的兜底手段**：

```
Chunk 1: [字符 0 ~ 500]
Chunk 2: [字符 400 ~ 900]    ← 和 Chunk 1 重叠 100 字
Chunk 3: [字符 800 ~ 1300]   ← 和 Chunk 2 重叠 100 字

即使 400 处切断了一句话，Chunk 2 也包含完整上下文
```

**进阶技巧**：

- **保留标题前缀**：每个 Chunk 前面带上它所属的标题路径
  `[文档 > 第三章 > 超时配置] 单接口超时：在代码中设置...`
- **元数据过滤**：每个 Chunk 带上 `{"source": "trpc_doc.pdf", "section": "超时配置"}`，检索时可按元数据过滤

### 难点 3：纯向量检索做不了精确匹配

```
用户问："错误码 E10024 是什么意思？"

向量检索：把问题向量化 → 语义相似度搜索
  → 找到"错误码概述""E10023 说明""常见错误处理"
  → 就是找不到 E10024！因为 E10024 和 E10023 的向量非常接近！

关键词检索：直接搜 "E10024"
  → 精确命中 → 完美
```

**解决方案：混合检索（Hybrid Search）**

```
用户的查询
    │
    ├──────────────────┐
    ▼                  ▼
 向量检索            关键词检索
 （语义相似）         （BM25 / Elasticsearch）
    │                  │
    ▼                  ▼
 候选集 A            候选集 B
    │                  │
    └────────┬─────────┘
             ▼
   合并 + 去重 + 重排序（RRF）
             │
             ▼
         最终 Top-K
```

**RRF（Reciprocal Rank Fusion，倒数排名融合）**：把两个检索结果按"排名倒数之和"合并打分：

```
score(doc) = Σ  1 / (k + rank_i(doc))

例如某 doc 在向量检索排第 2、在关键词检索排第 5：
score = 1/(60+2) + 1/(60+5) = 0.0161 + 0.0154 = 0.0315

排名越靠前贡献越大，两种检索都靠前的会被排到最高
```

| 检索方式 | 擅长 | 不擅长 |
|---------|------|--------|
| 向量检索 | "怎么配置超时" → 找到"超时设置方法" | "E10024" → 找不到精确编号 |
| 关键词检索 | "E10024" → 精确命中 | "怎么让请求不等太久" → 找不到"超时" |
| **混合检索** | 两者都能处理 | 复杂度高 |

### 难点 4：Top-K 怎么选、怎么打分

**K 的选择是 Trade-off**：

```
K 太小（K=1）：可能漏掉关键信息（召回率低）
K 太大（K=20）：塞了太多噪音进 Prompt，LLM 被干扰（精确率低）

且 K 大 → Prompt 长 → token 成本上升 + 延迟增加
```

**打分方法**（前面已展开）：

```
向量余弦相似度（粗排）→ Rerank（精排）→ LLM Judge（终极裁判）
快但粗                  慢但准              准但贵
```

### 难点 5：怎么判断检索效果

**这是 RAG 工程最容易忽略的一环**。没有评估，等于盲修。

需要构造**评估数据集**：

```json
[
  {
    "question": "tRPC 怎么设置超时？",
    "expected_docs": ["doc_timeout_config.md"],
    "expected_answer_contains": ["trpc_go.yaml", "WithTimeout"]
  },
  ...
]
```

**核心指标**：

| 指标 | 定义 | 关注什么 |
|------|------|---------|
| Recall@K | 正确文档是否出现在 Top-K 中 | 有没有漏 |
| Precision@K | Top-K 中相关文档的比例 | 噪音多不多 |
| MRR | 正确文档第一次出现的位置的倒数 | 是不是排得够靠前 |
| nDCG | 考虑排名位置的加权得分 | 综合排序质量 |
| 端到端正确率 | LLM 最终回答是否正确（用 LLM Judge） | 用户感受到的质量 |

**评估的两个层次**：

```
层次 1：检索评估
  query → 检索结果 → 对比 expected_docs
  问题：检索准不准？

层次 2：端到端评估
  query → 检索 → LLM 生成 → 对比 expected_answer
  问题：最终用户拿到的回答对不对？

只评层次 1 是不够的——检索准但 LLM 没用上也白搭
只评层次 2 不够细——错了不知道是检索的锅还是生成的锅
```

## 六、Memory：长期记忆的工程实现

### 6.1 Memory 的两条写入路径

```
路径 1：显式写入（用户主动）
  用户："以后记住我用 Go 和 tRPC"
  → Agent 调用 memory.add 工具
  → 写入 Memory Store

路径 2：隐式沉淀（系统自动）
  会话过程：
    用户：我是 Go 开发者，主要用 tRPC
    Agent：好的，了解了

  会话结束后（异步任务）：
    Memory Service 把整段对话喂给 LLM
    Prompt："请从以下对话中提取值得长期记住的用户信息，
             以 JSON 数组返回，每条不超过 50 字"
    LLM 输出：[
      {"type": "preference", "content": "该用户是 Go 开发者，使用 tRPC 框架"}
    ]
    → 向量化 → 写入 Memory Store（关联 user_id）
```

### 6.2 Memory 的召回流程

```
新会话开始时：
  用户："帮我写一个超时中间件"
              │
              ▼
     Runner 触发 Memory.recall(
       user_id="user_001",
       query="超时中间件",
       top_k=3
     )
              │
              ▼
     在 Memory Store 里检索：
       - 按 user_id 过滤（只查这个用户的记忆）
       - 在过滤后的子集做向量相似度检索
              │
              ▼
     召回："该用户是 Go 开发者，使用 tRPC 框架"
              │
              ▼
     拼入 System Prompt：
       "已知用户信息：
        - 该用户是 Go 开发者，使用 tRPC 框架
        请基于此回答用户问题。"
              │
              ▼
     LLM 直接用 Go + tRPC 写中间件代码
     不需要再问"你用什么语言？"
```

### 6.3 Memory vs Knowledge 在实现上的关键差异

```
                Knowledge                  Memory

存储隔离        全局共享                   按 user_id 隔离
                所有用户检索同一个库         每个用户只能查自己的记忆

写入时机        提前批量灌入                运行时持续追加

数据规模        大（百万级 Chunk）          小（每用户几十~几百条）

更新频率        低（文档版本更新时）         高（每次对话都可能产生新记忆）

过期机制        通常无                     可能需要（旧记忆失效/合并）
```

## 七、容易混淆的点

### 7.1 向量数据库返回的是什么？

```
❌ 误解：检索 → 返回向量 → 再把向量"解码"成文本
✅ 正解：向量数据库同时存了向量和原文。
        用向量做相似度查找，找到对应条目后直接返回原文。

向量是检索的"索引"，不是最终内容。
```

### 7.2 RAG 和 Function Calling 是同一个东西吗？

```
不是。两者目的不同：

Function Calling：让 LLM "执行动作"
  - LLM 输出工具调用指令 → 外部执行 → 结果回传
  - 强调"交互"

RAG：让 LLM "获取知识"
  - 检索文档 → 拼进 Prompt → LLM 基于上下文回答
  - 强调"参考资料"

但工程上两者可以结合：
  把"检索知识库"封装成一个 Tool，让 Agent 自主决定何时调用。
  这种模式叫 Agentic RAG（智能体式 RAG）。
```

### 7.3 上下文窗口够大了，还需要 RAG 吗？

```
即使有 1M token 窗口（Gemini 1.5），RAG 仍然必要：

理由 1：成本
  1M token 的输入，单次调用成本几美元。每个请求都塞全部知识不现实。

理由 2：延迟
  长 Prompt → 推理时间显著增加（注意力机制是 O(n²)）。

理由 3：质量
  Lost in the Middle 现象：长 Prompt 中间部分的信息容易被模型忽略。
  少量精准的上下文 > 海量但稀释的上下文。

理由 4：动态性
  知识在变（文档更新、用户偏好变化），不可能每次都重新塞。

结论：上下文窗口变大只是放宽了 Top-K 的上限，不能取代 RAG 架构。
```

### 7.4 Embedding 模型怎么选？

```
关键维度：
  1. 维度（dimension）：常见 768 / 1024 / 1536 / 3072
     维度越高表达能力越强，但存储和检索成本也越高

  2. 多语言能力：是否支持中文 / 中英混合
     纯英文模型用在中文上效果差

  3. 领域适配：通用 vs 领域专用（代码 / 医学 / 法律）

  4. 部署形态：API（OpenAI、Cohere）vs 本地（BGE、M3E、Qwen-Embedding）

  5. 同一个项目里 query 和 doc 必须用同一个模型
     否则向量不在同一个空间，相似度没意义
```

## 九、知识沉淀小结

| 类别 | 关键点 |
|------|--------|
| **本质** | RAG = 检索增强生成，给 LLM 配一本"实时可查的参考手册" |
| **流程** | 离线（加载 → 切分 → 向量化 → 存储） + 在线（向量化 → 检索 → 排序 → 拼接 → 生成） |
| **加载** | 把各种格式文件统一变成纯文本，是第一步，不涉及向量化 |
| **切分** | Overlap + 递归切分 + 保留标题前缀，是工程常用组合 |
| **召回** | "召回" = "搜出来"，分粗排和精排两个阶段 |
| **检索方式** | 纯向量搜不到精确编号；混合检索（向量 + BM25 + RRF）是工程标配 |
| **Rerank** | Cross-Encoder 精排，比 Bi-Encoder 准但慢，对小集合做 |
| **评估** | 必须建评估集，看 Recall@K、Precision@K、MRR + 端到端正确率 |
| **Memory** | 按 user_id 隔离的长期记忆，靠 LLM 沉淀提取，按 query 召回 |
| **三者关系** | Session 短期 / Memory 长期个性化 / Knowledge 通用外部知识，共同拼进 Prompt |
| **常见误区** | 向量数据库存的不只是向量（也存原文）；上下文窗口大不能取代 RAG |
