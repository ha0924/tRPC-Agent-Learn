## 一、为什么需要 RAG 和 Memory？

### 1.1 LLM 的两个根本局限

继 Session 解决了"LLM 无状态"问题之后，我们还要面对 LLM 的另外两个根本局限：

```
局限 1：训练数据有截止日期
  GPT-4 的知识停留在某个时间点
  → 它不知道你公司内部的最新文档
  → 它不知道昨天上线的新接口
  → 它不知道你这个用户的偏好

局限 2：上下文窗口有限
  即使是 128K 窗口的模型，也塞不下你公司全部文档
  → 不能把全部知识都塞进 Prompt
  → 必须"按需检索"，只塞当下相关的部分
```

**RAG（Retrieval-Augmented Generation，检索增强生成）就是解决这两个局限的工程方案。**

### 1.2 RAG 的核心思想

```
不要让 LLM "记住所有知识"，
而是让 LLM "需要什么知识时，能精准查到"。

类比：
  传统做法：让员工把公司全部文档背下来 → 不现实
  RAG 做法：员工手边放一本能快速翻阅的参考手册 → 用到时翻一下
```

## 二、Session / Memory / Knowledge 的关系

这三个概念都涉及"喂给 LLM 的上下文"，但定位完全不同。先把它们的边界划清楚：

```
                Session（短期记忆）        Memory（长期记忆）          Knowledge（外部知识）

数据从哪来？    本次对话过程中累积          多次对话沉淀提取            人工提前导入的文档
                "用户刚说的话"             "用户的偏好/画像"           "tRPC 官方文档.pdf"

存什么？        当前对话的消息历史          用户级别的个性化信息         通用的领域知识
                + state + events           偏好、习惯、历史决策         API 文档、产品手册

谁写入？        每轮自动追加               Agent 自动沉淀 / 用户主动    管理员提前灌入
                                          要求"记住这个"

检索方式        按 session_id 直接加载     按 user_id 过滤 + 语义匹配  纯语义匹配（按 query）

生命周期        本次对话（分钟~小时）       跟着用户走（天~永久）        跟着知识库版本

类比            你现在桌上的便签            你记得老客户的习惯           你手边的参考手册
```

## 三、RAG 的完整流程

RAG 分为**两个阶段**：离线准备（一次性）+ 在线服务（每次提问）。

### 3.1 阶段 A：离线准备

```
原始文件（PDF / Word / Markdown / 网页 / 数据库记录...）
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 1：文档加载（Document Loading）        │
│                                           │
│ 把各种格式的文件统一变成纯文本             │
│                                           │
│ 输入：tRPC超时配置.pdf（二进制）           │
│ 输出："## 超时配置\n\ntRPC 支持..."（文本）│
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 2：文本切分（Chunking）                │
│                                           │
│ 长文档切成小段                             │
│ "## 超时配置\n..." → [chunk1, chunk2, ...] │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 3：向量化（Embedding）                 │
│                                           │
│ 每个 chunk 调 Embedding 模型 → 一组数字     │
│ chunk1 → [0.12, -0.35, 0.88, ...]         │
│ chunk2 → [0.45, 0.22, -0.67, ...]         │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 4：存入向量数据库（Indexing）          │
│                                           │
│ 向量 + 原文 一起存                         │
│ ┌─────────────────────────────────────┐  │
│ │ id=1  vector=[0.12,-0.35,...]        │  │
│ │       text="tRPC 支持三种超时配置..."  │  │
│ │ id=2  vector=[0.45,0.22,...]         │  │
│ │       text="单接口超时：在代码中..."   │  │
│ └─────────────────────────────────────┘  │
└───────────────────────────────────────────┘
        │
        ▼
      准备完毕，等待被检索
```

### 3.2 阶段 B：在线服务

```
用户提问："tRPC 怎么设置超时？"
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 5：查询向量化                          │
│                                           │
│ 用同一个 Embedding 模型把问题变成向量       │
│ "tRPC 怎么设置超时？" → [0.15, -0.30, ...] │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 6：检索（Retrieval）/ 召回（Recall）   │
│                                           │
│ 在向量数据库里找最相似的向量                │
│ → id=1（相似度 0.92）                      │
│ → id=2（相似度 0.87）                      │
│ → id=7（相似度 0.85）                      │
│ → ...                                     │
│                                           │
│ 注意：返回的是"找到的条目"，不是向量本身    │
│       同时把存着的原文取出来               │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 7：排序（Ranking）+ Top-K 截断          │
│                                           │
│ 按相关性打分排序，取前 K 条                  │
│ K=3 → 留下 id=1, id=2, id=7                │
│                                           │
│ （可选）Rerank：用更精细的模型再打一遍分    │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 8：上下文拼接（Context Assembly）      │
│                                           │
│ 把检索到的原文塞进 Prompt                  │
│                                           │
│ System: "你是 tRPC 技术助手。              │
│         根据以下参考文档回答问题。         │
│                                           │
│ 参考文档：                                 │
│ [1] tRPC 支持三种超时配置方式...           │
│ [2] 单接口超时：在代码中设置...            │
│ [3] 链路超时：通过 context 传递..."        │
│                                           │
│ User: "tRPC 怎么设置超时？"                │
└───────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────┐
│ 步骤 9：LLM 生成（Generation）              │
│                                           │
│ LLM 基于参考文档 + 用户问题生成回答        │
│ （不靠自己训练时记住的知识瞎编）           │
└───────────────────────────────────────────┘
        │
        ▼
      返回给用户
```

## 四、术语词典

来自信息检索（IR）领域的专业术语，先逐个讲透。

### 4.1 Embedding / 向量化

把一段文本变成一个固定维度的数字向量（比如 1536 维），使得**语义相近的文本，向量也相近**。

```
"超时配置"       → [0.12, -0.35, 0.88, ..., 0.05]
"timeout 设置"   → [0.13, -0.34, 0.87, ..., 0.06]  ← 和上面很接近
"鲜花订购"       → [-0.42, 0.55, 0.11, ..., 0.92]  ← 完全不同
```

向量是**单向**的：

```
文本 → 向量  ✅（Embedding 模型完成）
向量 → 文本  ❌（做不到，向量是有损压缩的数学表示）
```

所以向量数据库会**同时存向量和原文**，检索时用向量找条目，最终返回原文。

### 4.2 Chunk / 切分

把长文档切成小段。每个段是一个 Chunk。

**为什么必须切分**：

1. Embedding 模型有输入长度上限（通常 512 ~ 8192 tokens）
2. 太长的 Chunk 语义混杂，向量化后"什么都像什么都不像"
3. 检索时塞进 Prompt 的内容也要控制长度

### 4.3 Index / 索引

切分 + 向量化后，存入向量数据库形成的**可检索数据结构**就是 Index。

类似数据库的 B+ 树索引，向量数据库用的是专门的**近似最近邻（ANN）索引**：HNSW、IVF、PQ 等算法。让"在百万级向量里找最相似的几个"能在毫秒级完成。

### 4.4 Query / 查询

用户提出的问题，作为检索的输入。

### 4.5 Retrieval / 检索

广义概念，指"从知识库里把相关内容找出来"这个动作。包括召回 + 排序 + Top-K 截断。

### 4.6 Recall / 召回

**"召回"= "把可能相关的都搜出来"**，不是什么高深概念。

之所以用"召回"而不直接说"搜索"，是因为 IR 领域要区分两个阶段：

```
阶段 1：召回（Recall）—— 粗筛
        尽量把相关的都捞回来，宁多勿漏
        从 100 万条 → 几十条候选

阶段 2：排序（Ranking）—— 精排
        对召回结果打分排序，挑出最好的
        从 几十条候选 → Top-K 条
```

**为什么分两步**：全量数据精细打分太慢。先快速粗筛到小集合，再精细排序。

### 4.7 召回率 / 精确率

衡量召回效果的两个核心指标：

```
假设知识库里关于"超时配置"真正相关的文档共 10 篇
你的检索找回了 20 篇，其中 8 篇确实相关

召回率（Recall）= 8 / 10 = 80%
                  找回的相关数 / 全部相关数
                  关注"有没有漏"

精确率（Precision）= 8 / 20 = 40%
                    找回的相关数 / 总共找回数
                    关注"有没有混入噪音"
```

这两个指标通常此消彼长：

- 召回多（K 大）→ 召回率高但精确率低（噪音多）
- 召回少（K 小）→ 精确率高但召回率低（容易漏）

#### 4.7.1 召回率到底代表什么？

**一句话理解：召回率衡量的是"该找到的，你漏了多少"。**

- 召回率 100% = 一篇都没漏
- 召回率 50% = 有一半该找到的被漏掉了

需要纠正一个常见的理解偏差：

```
❌ 召回率 ≠ 文档处理的准确度（不是衡量"解析 PDF 对不对"）
✅ 召回率 = 检索系统的"找全"能力（衡量"该找到的有没有被找到"）
```

它是对**检索环节**的评估指标，不是对文档加载/切分环节的评估。当然，文档加载和切分做得差**会间接导致**召回率低（见下文原因分析）。

#### 4.7.2 为什么会出现召回率高/低？

召回率的高低，本质上取决于 **RAG 流水线每一步的质量**。任何一步出问题，都可能导致"该找到的文档找不到"。按流水线顺序逐个分析：

**原因 1：切分（Chunking）质量差 —— 语义被切碎了**

```
原文："tRPC 支持三种超时配置方式：全局超时、单接口超时、链路超时"

糟糕的切分（按 200 字符硬切）：
  Chunk A: "tRPC 支持三种超时配"         ← 句子断了，语义不完整
  Chunk B: "置方式：全局超时、单接口..."   ← 脱离上文，变成孤立片段

用户问："tRPC 怎么设置超时？"
  → Chunk A 和 Chunk B 的向量都跟"超时配置"的语义有偏差
  → 向量检索时相似度得分不够高 → 排到很后面 → 被 Top-K 截掉
  → 召回失败 → 召回率降低
```

解法：递归切分 + Overlap + 保留标题前缀（见 4.5 节）。

**原因 2：Embedding 模型能力不够 —— 语义没被正确表示**

```
用户问："怎么让请求不等太久？"
知识库里有："tRPC 超时配置详解"

好的 Embedding 模型：
  "不等太久" 和 "超时配置" → 向量距离很近 → 找到了 ✅

差的 Embedding 模型（或中英文不匹配）：
  "不等太久" 和 "超时配置" → 向量距离远 → 找不到 ❌
  → 召回率降低
```

核心问题：Embedding 模型是否真的理解了文本的**语义**。如果模型只做字面匹配，"不等太久"和"超时"就是毫不相关的两个短语。

**原因 3：只用了向量检索，没做混合检索**

```
用户问："错误码 E10024 是什么意思？"

向量检索：
  把 "E10024" 向量化 → 和 "E10023"、"E10025" 的向量几乎一样
  → 错误地召回了 E10023 和 E10025 的文档
  → 真正的 E10024 文档可能反而排到后面
  → 召回率低

加上关键词检索（BM25）：
  精确匹配 "E10024" → 直接命中
  → 召回率高
```

**原因 4：Top-K 设太小 —— 被截断了**

```
知识库里有 10 篇关于超时的文档
向量检索确实都找到了，排在前 10

但你设了 K=3（只取前 3 条）
  → 只返回了 3 篇
  → 另外 7 篇被丢弃
  → 召回率 = 3/10 = 30%

如果设 K=10
  → 召回率 = 10/10 = 100%
```

K 越大召回率越高——但代价是精确率下降（混入更多噪音），而且 Prompt 变长、成本变高。

**原因 5：文档加载就丢了信息**

```
一份 PDF 里有表格：

| 参数名    | 默认值 | 说明        |
|----------|--------|------------|
| timeout  | 1000ms | 超时时间    |

糟糕的 PDF 解析器提取后变成：
  "参数名 默认值 说明 timeout 1000ms 超时时间"
  → 表格结构丢失，变成一堆散碎词
  → 向量化后语义模糊
  → 用户问 "timeout 的默认值" 时找不到

好的解析器：
  "参数名: timeout, 默认值: 1000ms, 说明: 超时时间"
  → 结构保留 → 语义清晰 → 能被检索到
```

虽然这是"加载"环节的问题，但它**间接**影响了召回率。

**原因 6：Query 本身太模糊或太短**

```
用户问："怎么配？"
  → 这 3 个字向量化后，语义非常宽泛
  → 跟知识库里几乎所有"配置"文档都有点像，但又不特别像任何一篇
  → 真正相关的文档可能排不到前面
  → 召回率低

用户问："tRPC Go 项目怎么配置单接口超时？"
  → 语义精确 → 向量直指目标 → 召回率高
```

解法：Query Rewriting（查询改写）—— 让 LLM 先把用户的模糊问题改写成更精确的检索查询。

#### 4.7.3 一张因果图总结

```
召回率高低的因果链路：

文档加载质量差 ───────┐
                      │
切分策略不合理 ────────┤
                      │
Embedding 模型能力弱 ──┤──→ 该找到的文档 → 没被检索到 → 召回率低
                      │
只有向量检索无关键词 ──┤
                      │
Top-K 设得太小 ────────┤
                      │
Query 太模糊 ──────────┘
```

反过来，每一项做好了，召回率就上去了。

#### 4.7.4 一个直觉类比

把 RAG 想象成**考试前翻参考书**：

| 类比 | 对应 RAG |
|------|---------|
| 参考书有 10 页和你这道题相关 | 知识库里有 10 篇相关文档 |
| 你翻了翻，找到了 8 页 | 检索召回了 8 篇 |
| 还有 2 页你没翻到（被夹在其他页里了） | 2 篇被漏掉了 |
| 你的"找全率" = 8/10 = 80% | 召回率 = 80% |
| 为什么没找到那 2 页？ | 可能是目录索引不好（切分差）、可能是你翻得太快只看了前 3 页就停了（K 太小）、可能那 2 页的标题写得跟你想找的不像（Embedding 不准） |

**召回率本质上就是衡量"你漏了多少该找到的"。** 之所以会有高低之分，是因为 RAG 的每一步都有可能引入"漏找"的风险。

### 4.8 Top-K

排序后只取前 K 条结果。

```
经验值：
  简单问答：K = 3 ~ 5
  复杂分析：K = 5 ~ 10
  代码生成：K = 3（代码上下文要精确，噪音影响大）
```

### 4.9 Ranking / 排序、Rerank / 重排序

#### 4.9.0 三种相似度度量：余弦 / 欧氏 / 点积

在讲粗排之前，先把"相似度"这件事讲透。**不是只有"余弦相似度"一种选择**，向量数据库里常见三种度量方式：

**① 余弦相似度（Cosine Similarity）**

```
cosine(A, B) = (A · B) / (|A| × |B|)

几何意义：两个向量"方向"的夹角，忽略长度
取值范围：-1 ~ 1（归一化向量上一般是 0 ~ 1）
```

直觉：把每个向量看成"从原点射出的箭头"，**余弦只关心箭头指向的方向**，不关心箭头多长。

```
向量 A = [1, 0]   长度=1
向量 B = [10, 0]  长度=10   方向相同 → cosine = 1（最相似）
向量 C = [0, 1]   长度=1    方向垂直 → cosine = 0（不相似）
```

**② 欧氏距离（Euclidean / L2 Distance）**

```
L2(A, B) = √Σ(Aᵢ - Bᵢ)²

几何意义：两点之间的"直线距离"
取值范围：0 ~ +∞（越小越相似）
```

直觉：把每个向量看成"空间里的一个点"，**欧氏距离是两个点之间走直线的距离**。既关心方向也关心长度。

```
向量 A = [1, 0]
向量 B = [10, 0]   方向相同但远 → L2 = 9（不相似）
向量 C = [1, 1]    方向略偏但近 → L2 = 1（更相似）
```

**③ 点积（Dot Product / Inner Product）**

```
dot(A, B) = Σ Aᵢ × Bᵢ

几何意义：方向 + 长度的混合得分
取值范围：-∞ ~ +∞（越大越相似）
```

直觉：**既奖励"方向接近"，也奖励"长度大"**。常被推荐系统用来融合"相关性 + 热度"。

```
向量 A = [1, 0]
向量 B = [10, 0]   方向相同 + B 很长 → dot = 10（高分）
向量 C = [1, 0]    方向相同 + 长度小 → dot = 1（低分）
```

#### 4.9.1 三者的关系与选择

**关键数学事实**：当所有向量都做了 **L2 归一化**（即长度=1）后：

```
cosine(A, B) = dot(A, B) = 1 - L2(A, B)² / 2

三者排序结果完全等价！
```

这就是为什么大多数 Embedding 模型（OpenAI、BGE、M3E）的输出**都是归一化向量**——这样选哪种度量都一样，工程上简单。

**选择决策表**：

| 场景 | 推荐度量 | 理由 |
|------|---------|------|
| 文本语义检索（通用 RAG） | **余弦** | 只关心"语义方向"，不关心向量长度差异；Embedding 模型一般都按余弦优化 |
| 图像/人脸向量检索 | **欧氏距离** | 图像向量的长度本身可能含有信息（如置信度），需要保留 |
| 推荐系统（混合相关性+热度） | **点积** | 热门 item 的向量长度通常更大，点积自然倾向推荐"相关且热门" |
| 你不确定用哪个 | **余弦** | 默认选择，90% 的文本场景都用它 |

**给 RAG 用户的实用结论**：

```
1. 文本 RAG 一律用 cosine，不要纠结
2. 检查你的 Embedding 模型输出是不是归一化的：
   - OpenAI / BGE / M3E → 是归一化的，cosine=dot，选哪个都行
   - 自训的模型 → 不一定，建议显式 cosine
3. 不同度量混着用是灾难（同一索引里余弦和欧氏不可比）
```

#### 4.9.2 检索打分的三个层次

```
层次 1：向量余弦相似度（粗排）
  score = cosine(query_vector, doc_vector)
  范围 0~1，越接近 1 越相关
  特点：快（毫秒级），但相对粗糙

层次 2：Rerank 模型（精排）
  用专门的 Cross-Encoder 模型
  把 query 和每个候选文档一起送进模型
  输出更精确的相关性分数
  特点：慢（百毫秒级），但准

层次 3：LLM Judge
  让 LLM 判断："这段文档是否回答了用户的问题？"
  最准，但最贵
```

**Bi-Encoder vs Cross-Encoder**：

```
Bi-Encoder（向量检索用的模型）：
  query  → Encoder → vector_q
  doc    → Encoder → vector_d
  比较：cosine(vector_q, vector_d)
  特点：query 和 doc 分别编码，可预先把 doc 向量化好 → 快

Cross-Encoder（Rerank 用的模型）：
  [query + doc] → Encoder → score
  特点：query 和 doc 拼在一起送进模型，能理解两者关系 → 准但慢
        必须 query 来了才能算，没法预计算
```

典型 Pipeline：

```
100 万文档 ──向量粗排──▶ 50 条 ──Rerank精排──▶ 5 条 ──拼Prompt──▶ LLM
            < 50ms              < 200ms                    800ms~3s
```

### 4.10 Context Assembly / 上下文拼接

把检索到的 Top-K 原文，按一定模板塞进 Prompt。常见模板：

```
System: 你是 XX 助手。请基于以下参考资料回答用户问题。
        如果参考资料里没有相关信息，请直接说"我不知道"，不要编造。

参考资料：
[1] {chunk_1_text}
[2] {chunk_2_text}
[3] {chunk_3_text}

User: {user_query}
```

**注意点**：

- 加上"引用编号"`[1] [2]`，方便后续生成带引用的回答
- 明确指示"没有就说不知道"，降低幻觉
- 控制总长度，不要超过模型上下文窗口

### 4.11 Generation / 生成

LLM 基于拼好的 Prompt 输出回答。这一步和普通的 LLM 调用没区别，只是 Prompt 里多了参考资料。

### 4.12 Memory 相关：写入 / 沉淀 / 召回

```
写入（Write）
  把信息存进 Memory Store
  最直接：用户说"记住我是 Go 开发者" → 直接写

沉淀（Distill / Extract）
  从对话过程中"自动提取"值得长期记住的信息
  通常用 LLM 做提取（在会话结束后跑一遍）

  会话过程：
    用户：我是 Go 开发者，主要用 tRPC 框架
    Agent：好的

  会话后处理（沉淀）：
    Memory Service 调 LLM 分析这次对话
    → 提取："该用户是 Go 开发者，使用 tRPC 框架"
    → 向量化 → 存入 Memory Store（关联 user_id）

召回（Recall）
  新会话开始时，根据当前 query 检索这个用户的 Memory
  把相关的画像信息拼进 Prompt
```

## 五、工程落地的五大难点

概念之外，RAG 真正难的是工程落地。每一个难点都对应着工程经验。

### 难点 1：文档加载没那么简单

不同格式的坑各不相同：

```
PDF 的坑：
  - 扫描版 PDF 是图片，文字要 OCR
  - 表格用坐标画线条，不是结构化数据
  - 页眉页脚、页码混在正文里
  - 多栏排版，阅读顺序难判断

Word 的坑：
  - 嵌入图片里有文字（要不要 OCR）
  - 样式信息（加粗、标题层级）要不要保留

网页的坑：
  - 导航栏、广告、侧边栏都是噪音
  - 正文提取（Boilerplate Removal）

代码文档：
  - 代码块、注释、自然语言混合
  - 切分时不能切断代码逻辑
```

**经验**：早期可以只支持 Markdown（最干净），生产环境再针对性处理 PDF/Word。

### 难点 2：切分不准，语义被截断

```
糟糕的切分（按固定 200 字符）：
┌─────────────────────┐
│ Chunk 1:            │
│ tRPC 支持三种超时配  │ ← 句子被截断了！
└─────────────────────┘
┌─────────────────────┐
│ Chunk 2:            │
│ 置方式：             │ ← 上文没了，读不懂
│ 1. 全局超时...      │
└─────────────────────┘
```

**常见切分策略**：

| 策略 | 做法 | 适用 |
|------|------|------|
| 固定大小 | 每 N 字符切一刀 | 最简单，但最不准 |
| 按段落 | 以空行 / 换行为边界 | 一般文档 |
| 按标题层级 | 以 `##` `###` 为边界 | Markdown / 技术文档 |
| 递归切分 | 大标题切 → 太长再按段落 → 还太长按句子 | 通用性最好 |
| 语义切分 | 用模型判断"话题转换点" | 最准，最贵 |

**Overlap 重叠是最实用的兜底手段**：

```
Chunk 1: [字符 0 ~ 500]
Chunk 2: [字符 400 ~ 900]    ← 和 Chunk 1 重叠 100 字
Chunk 3: [字符 800 ~ 1300]   ← 和 Chunk 2 重叠 100 字

即使 400 处切断了一句话，Chunk 2 也包含完整上下文
```

**进阶技巧**：

- **保留标题前缀**：每个 Chunk 前面带上它所属的标题路径
  `[文档 > 第三章 > 超时配置] 单接口超时：在代码中设置...`
- **元数据过滤**：每个 Chunk 带上 `{"source": "trpc_doc.pdf", "section": "超时配置"}`，检索时可按元数据过滤

### 难点 3：纯向量检索做不了精确匹配

```
用户问："错误码 E10024 是什么意思？"

向量检索：把问题向量化 → 语义相似度搜索
  → 找到"错误码概述""E10023 说明""常见错误处理"
  → 就是找不到 E10024！因为 E10024 和 E10023 的向量非常接近！

关键词检索：直接搜 "E10024"
  → 精确命中 → 完美
```

**解决方案：混合检索（Hybrid Search）**

```
用户的查询
    │
    ├──────────────────┐
    ▼                  ▼
 向量检索            关键词检索
 （语义相似）         （BM25 / Elasticsearch）
    │                  │
    ▼                  ▼
 候选集 A            候选集 B
    │                  │
    └────────┬─────────┘
             ▼
   合并 + 去重 + 重排序（RRF）
             │
             ▼
         最终 Top-K
```

**RRF（Reciprocal Rank Fusion，倒数排名融合）**：把两个检索结果按"排名倒数之和"合并打分：

```
score(doc) = Σ  1 / (k + rank_i(doc))

例如某 doc 在向量检索排第 2、在关键词检索排第 5：
score = 1/(60+2) + 1/(60+5) = 0.0161 + 0.0154 = 0.0315

排名越靠前贡献越大，两种检索都靠前的会被排到最高
```

| 检索方式 | 擅长 | 不擅长 |
|---------|------|--------|
| 向量检索 | "怎么配置超时" → 找到"超时设置方法" | "E10024" → 找不到精确编号 |
| 关键词检索 | "E10024" → 精确命中 | "怎么让请求不等太久" → 找不到"超时" |
| **混合检索** | 两者都能处理 | 复杂度高 |

### 难点 4：Top-K 怎么选、怎么打分

**K 的选择是 Trade-off**：

```
K 太小（K=1）：可能漏掉关键信息（召回率低）
K 太大（K=20）：塞了太多噪音进 Prompt，LLM 被干扰（精确率低）

且 K 大 → Prompt 长 → token 成本上升 + 延迟增加
```

**打分方法**（前面已展开）：

```
向量余弦相似度（粗排）→ Rerank（精排）→ LLM Judge（终极裁判）
快但粗                  慢但准              准但贵
```

### 难点 5：怎么判断检索效果

**这是 RAG 工程最容易忽略的一环**。没有评估，等于盲修。

需要构造**评估数据集**：

```json
[
  {
    "question": "tRPC 怎么设置超时？",
    "expected_docs": ["doc_timeout_config.md"],
    "expected_answer_contains": ["trpc_go.yaml", "WithTimeout"]
  },
  ...
]
```

**核心指标**：

| 指标 | 定义 | 关注什么 |
|------|------|---------|
| Recall@K | 正确文档是否出现在 Top-K 中 | 有没有漏 |
| Precision@K | Top-K 中相关文档的比例 | 噪音多不多 |
| MRR | 正确文档第一次出现的位置的倒数 | 是不是排得够靠前 |
| nDCG | 考虑排名位置的加权得分 | 综合排序质量 |
| 端到端正确率 | LLM 最终回答是否正确（用 LLM Judge） | 用户感受到的质量 |

#### 5.1 MRR 用例子讲透

**MRR（Mean Reciprocal Rank，平均倒数排名）**：衡量"正确答案第一次出现的位置"。**只看第一个命中**，命中越靠前分数越高。

**单个 query 的计算**：

```
RR（Reciprocal Rank）= 1 / (第一个正确答案的排名)

检索返回 Top-5：[doc_A, doc_B, doc_C ✅, doc_D, doc_E]
第一个正确答案 doc_C 排在第 3 位
RR = 1/3 ≈ 0.333

如果排在第 1 位 → RR = 1.0（满分）
如果 Top-5 里完全没找到 → RR = 0
```

**多个 query 求平均**：

```
评估集有 3 个 query：
  Q1: 正确答案排第 1 → RR = 1.0
  Q2: 正确答案排第 2 → RR = 0.5
  Q3: 正确答案排第 4 → RR = 0.25

MRR = (1.0 + 0.5 + 0.25) / 3 ≈ 0.583
```

**MRR 适合什么场景**：

- **只关心"找到第一个对的"** 的场景：FAQ 问答、客服查文档、代码片段查找
- 不适合"需要多篇文档综合回答"的场景（因为它忽略第 2、3 个命中）

**直觉记忆**：MRR 像"你翻参考书找答案，翻到第几页找到的就给倒数分"。

#### 5.2 nDCG 用例子讲透

**nDCG（Normalized Discounted Cumulative Gain，归一化折损累计增益）**：衡量"整个排序质量"——既看命中数量，也看每个命中的位置和相关度等级。

它比 MRR 复杂，但能回答："**我这一整页结果排得好不好？**"

**三个核心要素**：

```
1. 相关度等级（gain）：每个文档对 query 的相关程度
   - 0 = 完全无关
   - 1 = 弱相关
   - 2 = 强相关
   - 3 = 完美匹配
   （不像 Recall 只有"相关/不相关"二元判断）

2. 位置折扣（discount）：排得越靠后，价值越低
   公式：gain / log₂(position + 1)
   第 1 位：÷ log₂(2) = 1     （满折扣价值）
   第 2 位：÷ log₂(3) ≈ 1.58
   第 3 位：÷ log₂(4) = 2
   第 4 位：÷ log₂(5) ≈ 2.32

3. 归一化（normalize）：除以"理想排序"的 DCG
   把得分归到 0 ~ 1 之间，便于横向比较
```

**完整计算示例**：

假设知识库里关于"超时配置"有 3 篇文档，相关度分别是 `3, 2, 1`。

**情况 A：完美排序**（最理想）

```
Top-5 排序     相关度    位置折扣后的 gain
位置 1: doc_X    3       3 / log₂(2) = 3 / 1.00 = 3.000
位置 2: doc_Y    2       2 / log₂(3) = 2 / 1.58 = 1.262
位置 3: doc_Z    1       1 / log₂(4) = 1 / 2.00 = 0.500
位置 4: 噪音     0       0
位置 5: 噪音     0       0

DCG_ideal = 3.000 + 1.262 + 0.500 = 4.762
```

**情况 B：你的系统实际排序**

```
Top-5 排序     相关度    位置折扣后的 gain
位置 1: 噪音     0       0
位置 2: doc_Z    1       1 / log₂(3) = 0.631
位置 3: doc_X    3       3 / log₂(4) = 1.500
位置 4: 噪音     0       0
位置 5: doc_Y    2       2 / log₂(6) = 0.774

DCG_actual = 0 + 0.631 + 1.500 + 0 + 0.774 = 2.905

nDCG = DCG_actual / DCG_ideal = 2.905 / 4.762 ≈ 0.61
```

**怎么解读 nDCG = 0.61**：

```
0.61 表示你的排序只发挥了"理想排序"61% 的价值。
nDCG = 1.0：完美（相关度高的全排前面）
nDCG > 0.8：很好
nDCG > 0.6：一般
nDCG < 0.5：排序有大问题
```

#### 5.3 三个指标怎么选用？

| 场景 | 用什么 | 为什么 |
|------|--------|--------|
| FAQ / 单文档命中场景 | **MRR** | 只关心第一个对的排在哪 |
| 多文档综合回答（典型 RAG） | **nDCG** | 多个相关文档都重要，且位置和相关度等级都要考虑 |
| 快速冒烟测试 | **Recall@K** | 简单二元判断，能跑出"有没有漏"就够了 |
| 噪音敏感场景（医学/法律） | **Precision@K + nDCG** | 既要避免噪音，也要保证排序质量 |
| 端到端用户体验 | **LLM Judge 评分** | 最终用户拿到的回答好不好，前面指标都是中间过程 |

**实战建议**：

```
RAG 评估的最小可用组合：
  Recall@5  → 看检索有没有漏（找全率）
  MRR       → 看排序有没有把对的放前面
  端到端正确率 → 看 LLM 最终回答对不对

够了，nDCG 等更复杂的指标，到优化阶段再引入。
```

#### 5.4 评估的两个层次

```
层次 1：检索评估
  query → 检索结果 → 对比 expected_docs
  问题：检索准不准？

层次 2：端到端评估
  query → 检索 → LLM 生成 → 对比 expected_answer
  问题：最终用户拿到的回答对不对？

只评层次 1 是不够的——检索准但 LLM 没用上也白搭
只评层次 2 不够细——错了不知道是检索的锅还是生成的锅
```

## 六、Memory：长期记忆的工程实现

### 6.1 Memory 的两条写入路径

```
路径 1：显式写入（用户主动）
  用户："以后记住我用 Go 和 tRPC"
  → Agent 调用 memory.add 工具
  → 写入 Memory Store

路径 2：隐式沉淀（系统自动）
  会话过程：
    用户：我是 Go 开发者，主要用 tRPC
    Agent：好的，了解了

  会话结束后（异步任务）：
    Memory Service 把整段对话喂给 LLM
    Prompt："请从以下对话中提取值得长期记住的用户信息，
             以 JSON 数组返回，每条不超过 50 字"
    LLM 输出：[
      {"type": "preference", "content": "该用户是 Go 开发者，使用 tRPC 框架"}
    ]
    → 向量化 → 写入 Memory Store（关联 user_id）
```

### 6.1.5 Memory 的冲突处理（旧记忆 vs 新记忆）

这是 Memory 系统**最容易被忽略、但最影响用户体验**的环节。

**问题场景**：

```
上个月，用户说："我喜欢 Python"
  → Memory 写入：{user.lang_preference: "Python"}

今天，用户说："我现在改用 Go 了"
  → 这条新信息和旧 Memory 矛盾，怎么办？
```

如果不处理冲突，会出现什么糟糕体验：

```
两个月后用户问："给我写个 HTTP 服务"
  → Memory 召回了两条："喜欢 Python" + "改用 Go"
  → 全塞进 Prompt → LLM 困惑 → 可能写出 Python 或 Go，甚至两个都写
  → 用户："你怎么不知道我现在用 Go？"
  → 信任崩塌
```

#### 三种冲突类型和对应策略

**类型 1：彻底覆盖型（互斥事实）**

```
旧：用户在腾讯工作
新：用户跳槽去了字节
关系：互斥（一个人同时只在一家公司）

策略：覆盖 / 软删除
  - 把旧记忆标记 deprecated_at=now()
  - 写入新记忆
  - 召回时过滤 deprecated_at != null 的条目
```

**类型 2：演化型（偏好/口味变化）**

```
旧：用户喜欢 Python
新：用户改用 Go 了
关系：演化（用户技术栈在变）

策略：时间戳版本化
  - 旧记忆保留但加上 valid_until 时间戳
  - 新记忆写入时带 valid_from
  - 召回时按时间窗口取最新（"当前有效"的版本）
  - 历史可追溯（"用户在 2024 年用 Python，2025 改用 Go"）
```

**类型 3：共存型（多偏好叠加）**

```
旧：用户喜欢 Python
新：用户今天开始也用 Go，但 Python 还会用
关系：共存（用户技能栈扩展，不是替换）

策略：合并 / 累积
  - 旧记忆不动
  - 新记忆作为补充写入
  - 召回时两条都返回："用户熟悉 Python 和 Go"
```

#### 工程实现：怎么判断属于哪种类型？

**最实用的做法：在沉淀阶段用 LLM 做"冲突检测"**

```
新记忆要写入前：
  Step 1: 向量召回该用户已有的相似记忆（topK=5）
  Step 2: 把新记忆 + 已有记忆送给 LLM，问：
    "新信息和已有信息是否矛盾？
     - 矛盾：说明哪条作废
     - 演化：说明是替换还是补充
     - 不冲突：直接写入"
  Step 3: 按 LLM 判断执行对应的写入策略
```

**字段设计建议**：

```json
{
  "memory_id": "mem_001",
  "user_id": "user_xxx",
  "content": "用户使用 Go 语言",
  "type": "preference.language",   // 用 type 分类便于互斥判断
  "created_at": "2025-06-26",
  "valid_from": "2025-06-26",      // 起始有效时间
  "valid_until": null,             // null 表示当前有效
  "supersedes": "mem_old_042",     // 显式标记替代了哪条旧记忆
  "confidence": 0.95
}
```

#### 冲突处理的两个原则

```
原则 1：永远不要直接 DELETE
  → 用 soft delete（valid_until / deprecated_at）
  → 历史可追溯，出 bug 能回滚

原则 2：召回时显式过滤"当前有效"的版本
  WHERE user_id = ? AND (valid_until IS NULL OR valid_until > NOW())
  → 不要让"作废的记忆"污染 Prompt
```

#### 极端情况：用户说"忘掉这件事"

```
用户："忘记我之前说的工作单位"
  → 不是 DELETE，而是写入一条"否定记忆"：
    {type: "forget", target: "preference.company", forget_at: now()}
  → 召回时遇到 type=forget 的条目，把对应 target 的记忆全过滤掉
  → 既执行了"忘记"，又保留了"用户主动要求忘记"这个事实
```

### 6.2 Memory 的召回流程

```
新会话开始时：
  用户："帮我写一个超时中间件"
              │
              ▼
     Runner 触发 Memory.recall(
       user_id="user_001",
       query="超时中间件",
       top_k=3
     )
              │
              ▼
     在 Memory Store 里检索：
       - 按 user_id 过滤（只查这个用户的记忆）
       - 在过滤后的子集做向量相似度检索
              │
              ▼
     召回："该用户是 Go 开发者，使用 tRPC 框架"
              │
              ▼
     拼入 System Prompt：
       "已知用户信息：
        - 该用户是 Go 开发者，使用 tRPC 框架
        请基于此回答用户问题。"
              │
              ▼
     LLM 直接用 Go + tRPC 写中间件代码
     不需要再问"你用什么语言？"
```

### 6.3 Memory vs Knowledge 在实现上的关键差异

```
                Knowledge                  Memory

存储隔离        全局共享                   按 user_id 隔离
                所有用户检索同一个库         每个用户只能查自己的记忆

写入时机        提前批量灌入                运行时持续追加

数据规模        大（百万级 Chunk）          小（每用户几十~几百条）

更新频率        低（文档版本更新时）         高（每次对话都可能产生新记忆）

过期机制        通常无                     可能需要（旧记忆失效/合并）
```

## 七、容易混淆的点

### 7.1 向量数据库返回的是什么？

```
❌ 误解：检索 → 返回向量 → 再把向量"解码"成文本
✅ 正解：向量数据库同时存了向量和原文。
        用向量做相似度查找，找到对应条目后直接返回原文。

向量是检索的"索引"，不是最终内容。
```

### 7.2 RAG 和 Function Calling 是同一个东西吗？

```
不是。两者目的不同：

Function Calling：让 LLM "执行动作"
  - LLM 输出工具调用指令 → 外部执行 → 结果回传
  - 强调"交互"

RAG：让 LLM "获取知识"
  - 检索文档 → 拼进 Prompt → LLM 基于上下文回答
  - 强调"参考资料"

但工程上两者可以结合：
  把"检索知识库"封装成一个 Tool，让 Agent 自主决定何时调用。
  这种模式叫 Agentic RAG（智能体式 RAG）。
```

#### 7.2.1 传统 RAG vs Agentic RAG 架构对比

**传统 RAG（也叫 Naive RAG / Vanilla RAG）**：流程是**固定的、单次的、被动的**。

```
用户提问
   │
   ▼
[固定流程，无判断]
   │
   ▼
向量化 query → 检索 Top-K → 拼 Prompt → LLM 生成 → 返回
   │
   ▼
回答用户

特点：
  - 每个 query 都走一次检索，不管要不要
  - 检索一次就结束，不会"再检索"
  - LLM 是"被喂"上下文，没有决策权
```

**Agentic RAG（智能体式 RAG）**：流程是**动态的、可循环的、主动的**。

```
用户提问
   │
   ▼
LLM 分析问题 ──→ 这个问题需要查资料吗？
   │                   │
   │              ┌────┴────┐
   │             需要      不需要
   │              │          │
   │              ▼          ▼
   │       调 retrieve     直接回答
   │       工具            （闲聊/常识）
   │              │
   │              ▼
   │       检索结果回来
   │              │
   │              ▼
   │       结果够回答了吗？
   │              │
   │         ┌────┴────┐
   │        够        不够
   │         │         │
   │         ▼         ▼
   │      生成答案   改写 query 再检索
   │                  / 换关键词
   │                  / 拆子问题分别查
   │                  （循环）
   │
   ▼
回答用户

特点：
  - LLM 是"指挥官"，决定要不要检索、检索几次、查什么
  - 可以多轮检索（迭代式 retrieval）
  - 可以拆解复杂问题成子问题，分别检索后综合
```

#### 7.2.2 用一个例子看差异

**问题**："对比一下 tRPC-Go 和 gRPC-Go 在拦截器实现上的差异"

**传统 RAG 的做法**：

```
1. 把整句问题向量化
2. 检索一次 → Top-5
3. 可能召回：
   - "tRPC-Go 拦截器原理"（相关）
   - "tRPC 框架概述"（弱相关）
   - "gRPC 入门"（弱相关）
   - 其他噪音
4. 拼 Prompt → LLM 凑合回答

问题：tRPC-Go 和 gRPC-Go 各自的核心文档可能没被同时召回到 Top-K
     因为查询是混合的，向量被"稀释"了
```

**Agentic RAG 的做法**：

```
1. LLM 分析：这是一个对比问题，要分别查两个对象
2. 拆分子问题：
   - 子查询 A: "tRPC-Go 拦截器实现"
   - 子查询 B: "gRPC-Go 拦截器实现"
3. 分别检索 A → 拿到 tRPC 的核心文档
              B → 拿到 gRPC 的核心文档
4. LLM 看到结果后判断："两边的资料都够了"
5. 综合两份资料生成对比回答

优势：精准度大幅提升，因为每次检索的 query 都是聚焦的
```

#### 7.2.3 Agentic RAG 的常见模式

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| **Self-Querying** | LLM 把自然语言 query 改写成"过滤条件 + 检索词" | 带元数据的检索（"2024 年之后的 Go 相关文档"）|
| **Multi-Query** | LLM 把一个 query 改写成 N 个相似 query 分别检索后合并 | 提升召回率，对抗用户表达模糊 |
| **Sub-Question** | LLM 把复杂问题拆成子问题，分别检索后综合 | 对比类、多跳推理类问题 |
| **Iterative Retrieval** | 检索一次不够，LLM 判断后再检索（多轮） | 复杂研究类问题 |
| **Self-Reflection** | LLM 检查检索结果质量，不满意就重新检索 | 高精度需求场景 |
| **ReAct-style RAG** | Reasoning（推理）和 Retrieval（检索）交替进行 | 通用 Agentic 框架 |

#### 7.2.4 工程取舍

```
传统 RAG：
  ✅ 实现简单（一条流水线）
  ✅ 延迟低（一次检索就完事）
  ✅ 成本低（一次 LLM 调用）
  ❌ 复杂问题处理差

Agentic RAG：
  ✅ 处理复杂问题强
  ✅ 召回质量高（多次/聚焦检索）
  ❌ 延迟高（多轮 LLM + 多次检索）
  ❌ 成本高（一个 query 可能触发 3~10 次 LLM 调用）
  ❌ 链路复杂，难调试

实战选择：
  简单 FAQ / 客服 / 文档查询 → 传统 RAG 足够
  研究分析 / 多跳推理 / 跨文档对比 → Agentic RAG
  生产系统的实用模式：默认传统 RAG，对识别出的复杂 query 升级到 Agentic
```

### 7.3 上下文窗口够大了，还需要 RAG 吗？

```
即使有 1M token 窗口（Gemini 1.5），RAG 仍然必要：

理由 1：成本
  1M token 的输入，单次调用成本几美元。每个请求都塞全部知识不现实。

理由 2：延迟
  长 Prompt → 推理时间显著增加（注意力机制是 O(n²)）。

理由 3：质量
  Lost in the Middle 现象：长 Prompt 中间部分的信息容易被模型忽略。
  少量精准的上下文 > 海量但稀释的上下文。

理由 4：动态性
  知识在变（文档更新、用户偏好变化），不可能每次都重新塞。

结论：上下文窗口变大只是放宽了 Top-K 的上限，不能取代 RAG 架构。
```

### 7.4 Embedding 模型怎么选？

```
关键维度：
  1. 维度（dimension）：常见 768 / 1024 / 1536 / 3072
     维度越高表达能力越强，但存储和检索成本也越高

  2. 多语言能力：是否支持中文 / 中英混合
     纯英文模型用在中文上效果差

  3. 领域适配：通用 vs 领域专用（代码 / 医学 / 法律）

  4. 部署形态：API（OpenAI、Cohere）vs 本地（BGE、M3E、Qwen-Embedding）

  5. 同一个项目里 query 和 doc 必须用同一个模型
     否则向量不在同一个空间，相似度没意义
```

#### 7.4.1 为什么 query 和 doc 必须用同一个 Embedding 模型？

这条规则**几乎是 RAG 系统的"第一性原理"**，理解它就理解了向量检索的本质。下面层层推导。

**第一步：先建立"向量空间"的直觉**

Embedding 模型本质上是一个**映射函数**：

```
f: 文本 → ℝᵈ（d 维实数向量）
```

但**关键不在维度，而在"如何把文本映射到这个 d 维空间"**。每个 Embedding 模型在训练时，都学到了一套**独有的、内部一致的坐标系**：

```
模型 A 训练后的坐标系：
  "苹果"  → [0.8, 0.2, ..., 0.1]   ← 在 A 模型的"水果区"
  "香蕉"  → [0.7, 0.3, ..., 0.2]   ← 距离"苹果"近
  "手机"  → [-0.5, 0.6, ..., 0.9]  ← 在 A 模型的"电子产品区"

模型 B 训练后的坐标系（即使维度一样）：
  "苹果"  → [0.1, 0.9, ..., 0.4]   ← 在 B 模型自己的"水果区"
  "香蕉"  → [0.2, 0.8, ..., 0.5]   ← 距离"苹果"近
  "手机"  → [0.7, 0.2, ..., 0.1]   ← 在 B 模型自己的"电子产品区"
```

**关键观察**：两个模型都"对"——在各自的空间里，"苹果"和"香蕉"都接近、和"手机"都远。**但这两个空间之间没有任何对应关系**。

**第二步：跨模型计算相似度会发生什么？**

假设你用模型 A 向量化了知识库的 doc："iPhone 是苹果公司的产品"，
用模型 B 向量化了用户的 query："苹果手机怎么用？"

```
A 模型对 "iPhone 是苹果公司的产品" → [0.6, -0.2, 0.8, ...]（A 的电子产品+水果交叉区）
B 模型对 "苹果手机怎么用？"        → [0.3, 0.7, -0.1, ...]（B 的电子产品+水果交叉区）

cosine 相似度 = ???
```

虽然你能强行算出一个数，但**这个数没有任何语义意义**：

- A 空间的第一维可能编码"是否是水果"
- B 空间的第一维可能编码"是否是名词"
- 两个空间的"第一维"含义完全不同
- 算余弦就像问"东方 0.6 和 北方 0.3 谁更接近"——单位都不一样

**第三步：反驳"维度一样不就能算了吗？"**

这是最常见的反直觉点。直接反驳：

```
反驳 1：维度只是"空间的容器大小"，不是"空间的内容"

  比喻：北京和纽约都是地球上的"二维坐标(经度,纬度)"
  但 "北京的经纬度" 和 "纽约的经纬度" 数学上可以做加减
  得出的结果毫无意义 —— 因为这两个坐标描述的是不同地点的位置
  
  同理：模型 A 输出的 1536 维 和 模型 B 输出的 1536 维
  只是"都是 1536 个数字"，但每个维度代表的语义完全不同

反驳 2：Embedding 训练的目标决定了空间的"语义结构"

  每个模型用不同的训练数据、不同的损失函数、不同的负样本采样策略训练
  → 最终学到的"什么和什么应该在一起"的判断完全不同
  → 即使输入完全相同的文本，输出的向量也是不同的"坐标系下的坐标"

反驳 3：实证检验也证明了这一点

  你可以自己跑一下：
    text = "tRPC 是腾讯开源的 RPC 框架"
    vec_openai = openai.embed(text)    # 1536 维
    vec_bge    = bge.embed(text)       # 1024 维  ← 即便维度一样，也不可比
    
    cosine(vec_openai, vec_openai) = 1.0  ✅ 同空间，自己和自己相似
    cosine(vec_openai, vec_bge) = ???     ❌ 维度都不同，算不了
    
    强行把 bge 投影到 1536 维后算 cosine：
    → 得到的相似度近乎随机（≈ 0 周围抖动）
```

**第四步：什么时候这条规则会被"打破"？**

只有一种例外：**专门训练的跨模型对齐技术**

```
1. 蒸馏（Distillation）：
   把小模型 B 的输出对齐到大模型 A 的空间
   → 学生模型 B 才能在 A 的空间里检索

2. 对齐学习（Alignment）：
   用大量平行语料强制两个模型的输出空间对齐

但这两种都需要专门的训练，不是"默认能做的"。
对工程师而言：现成模型之间，永远不要混用。
```

**第五步：实战中的常见踩坑**

```
踩坑 1：升级 Embedding 模型时没重建索引
  你之前用 text-embedding-ada-002 建了 100 万条向量
  今天切到 text-embedding-3-small（虽然都是 OpenAI 出品）
  → 即使都是 OpenAI 自家模型，它们的空间也不同
  → 必须重新对全部 doc 做一次向量化，重建索引
  → 否则检索质量会断崖式下降

踩坑 2：本地开发用 BGE，线上用 OpenAI
  → 开发环境检索效果好，线上一塌糊涂
  → 因为同一个 query 在两个空间里，找到的是完全不同的"邻居"

踩坑 3：缓存了旧模型的 query 向量
  → 用户每次同一个 query 命中缓存的旧向量，doc 却是新模型的向量
  → 慢性中毒，难排查
```

**核心结论一句话**：

```
Embedding 模型不是输出"通用语义向量"，
而是输出"自己这套坐标系下的语义坐标"。
跨模型混用就像用两把不同刻度的尺子量身高——数字算得出来，意思全错。
```

## 九、知识沉淀小结

| 类别 | 关键点 |
|------|--------|
| **本质** | RAG = 检索增强生成，给 LLM 配一本"实时可查的参考手册" |
| **流程** | 离线（加载 → 切分 → 向量化 → 存储） + 在线（向量化 → 检索 → 排序 → 拼接 → 生成） |
| **加载** | 把各种格式文件统一变成纯文本，是第一步，不涉及向量化 |
| **切分** | Overlap + 递归切分 + 保留标题前缀，是工程常用组合 |
| **召回** | "召回" = "搜出来"，分粗排和精排两个阶段 |
| **检索方式** | 纯向量搜不到精确编号；混合检索（向量 + BM25 + RRF）是工程标配 |
| **Rerank** | Cross-Encoder 精排，比 Bi-Encoder 准但慢，对小集合做 |
| **评估** | 必须建评估集，看 Recall@K、Precision@K、MRR + 端到端正确率 |
| **Memory** | 按 user_id 隔离的长期记忆，靠 LLM 沉淀提取，按 query 召回 |
| **三者关系** | Session 短期 / Memory 长期个性化 / Knowledge 通用外部知识，共同拼进 Prompt |
| **常见误区** | 向量数据库存的不只是向量（也存原文）；上下文窗口大不能取代 RAG |
