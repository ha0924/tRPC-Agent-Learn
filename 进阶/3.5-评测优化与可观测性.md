# 3.5 评测、优化与可观测性

> **学习目标**：理解 Agent 评测范式与传统系统监控范式的本质差异；掌握 Eval Set、Metric、LLM Judge、Rubric Evaluator 这套"把主观判断工程化为可重复打分"的方法论；建立 Prompt Iteration / AgentOptimizer 等数据驱动的优化闭环；将既有的 OpenTelemetry 经验迁移到 Agent 场景，理解 Langfuse 等 Agent 专用可观测平台的差异化价值；最终建立"Agent 跑得好不好、稳不稳、贵不贵"的判断体系。

---

## 〇、为什么 3.5 是分水岭

3.1~3.4 解决的是 **"怎么把 Agent 做出来"**（构造视角）：记忆、协作、协议、Skills、CodeExecutor。

3.5 解决的是 **"怎么判断这个 Agent 到底行不行"**（评估视角）。

这两者之间有一条非常关键的分界线：

```
3.1~3.4：构造视角
   │
   ▼  ── 分水岭 ──
   │
3.5：  评估视角
       └─→ 这是从"个人玩具"到"生产系统"的唯一桥梁
```

**为什么 3.5 之前的所有内容都"不可证伪"**：

- 你改一版 prompt，Agent 看起来好像变好了——**但没有数据支撑**
- 你换一个 backbone 模型，似乎更聪明了——**但回归集呢？**
- 你新增一个 Tool，AI 似乎学会用了——**有几个 case 验证？**

→ 没有评测体系，所有的"优化"都是**主观感受**。
→ 没有可观测性，所有的"线上问题"都是**事后诸葛亮**。

### 2.1 最关键的认知差距：两种范式

**传统系统监控范式**（你熟悉的）：

```
日志/Trace → 异常 → 告警 → 修复
```

**Agent 评测范式**（新增能力）：

```
Eval Set → 跑出结果 → LLM/规则 打分 → 找弱项 → 改 prompt/工具 → 回归测试
   ↑                                                          │
   └──────────────── 持续迭代 ←────────────────────────────────┘
```

**核心差异**：

- 传统系统：**故障是二值的**（500 错误 = 坏）
- Agent 系统：**好坏是连续的、主观的**（"这个回答还行但不够准确"）

→ 所以 Agent 评测的核心命题是：**怎么把"主观的好坏"变成"可重复测量的数字"**。

## 五、Agent 评测的本质：用工作流测试做对比

为什么 Agent 评测不能直接套用传统测试方法？看一个直接对比：

```
传统工作流节点测试：
  输入：{ user_id: 123 }
  期望输出：{ name: "张三", balance: 1000 }
  断言：实际输出 == 期望输出 ？通过 : 失败

  → 二值、确定、可自动化

═══════════════════════════════════════════════════
Agent 任务测试：
  输入："帮我查下张三的账户余额，顺便提醒他充值"
  期望输出：？？？

  AI 可能给出：
    A. "张三的余额是 1000 元，建议及时充值"  ✅
    B. "张三账户余额：1000，请尽快充值"      ✅（措辞不同但同样对）
    C. "余额 1000，需要充值吗？"             ⚠️（少了提醒动作）
    D. "已查到余额，请告知张三充值"           ❌（理解错了，没说出数字）

  → 多值、模糊、需要"判断力"
═══════════════════════════════════════════════════
```

**这就是为什么 Agent 必须引入"LLM Judge"**：

- 字符串匹配做不了"语义对不对"
- 正则也搞不定"措辞不同但意思相近"
- 只有再用一个 LLM 当裁判，按 Rubric（评分标准）去打分

→ **Agent 评测 = 把"主观判断"工程化为"可重复打分"**
→ **Agent 优化 = 拿打分结果反向调 Prompt / Tool / Memory，再回归验证**

## 六、一个高频认知盲点：优化优先级金字塔

新手常见的直觉是：**"Agent 没调对工具 → 微调模型"**。

这是个典型误区——成本极高、周期长、还可能搞坏其他能力。

**生产环境中 99% 的情况，应该改的是这三样**：

```
① 工具的 description 写得不够好（AI 看不懂何时该用）
② 工具的 schema 不清晰（参数歧义）
③ 系统 prompt 没有引导 AI 在合适时机调工具

   → 这叫 "Prompt Iteration"，是 3.5 的核心技能
   → 不是 fine-tune，是"提示词工程的科学化"
```

**优化优先级金字塔**（生产里反复被验证）：

```
              ▲ 成本高、周期长、效果不确定
              │
         ┌────┴────┐
         │ 微调模型 │           ← 万不得已才用
         └─────────┘
       ┌─────┴─────┐
       │ 换更强模型 │           ← 钱包说了算
       └───────────┘
     ┌──────┴──────┐
     │ 调整 Workflow │          ← 拆分/编排
     └──────────────┘
   ┌──────┴──────┐
   │ 改工具 schema │           ← 工程改造
   └───────────────┘
 ┌──────┴──────┐
 │ Prompt 迭代  │              ← 80% 问题在这里解决
 └─────────────┘
              │
              ▼ 成本低、迭代快、立竿见影
```

→ 这意味着：**3.5 章节真正的"产出物"，是建立"用数据驱动 Prompt 迭代"的方法论**。这比"会用 Langfuse"重要 10 倍。

## 九、10 个核心概念逐一解析

> 本节是 3.5 章节的"概念词典"——对评测、优化、可观测性中最核心的 10 个概念做深度解释。建议配合第三节"4 象限图"对照阅读，每个概念都能挂到某个象限上。

### 9.1 Eval Set（评测集）

**一句话**：Eval Set 就是给 Agent 出的"考试题库"——一组预先准备好的输入 + 期望行为描述，用来可重复地衡量 Agent 表现。

**类比**：传统软件有单元测试的 test case，Eval Set 就是 Agent 的 test case。区别在于传统 test case 断言精确值（`assertEqual(result, 42)`），而 Eval Set 的"期望"往往是模糊的语义描述。

**一条 Eval Set 记录通常长这样**：

```json
{
  "id": "eval_001",
  "input": "帮我查下张三的账户余额，顺便提醒他充值",
  "expected_tool_calls": ["query_balance", "send_reminder"],
  "expected_output_contains": ["余额", "1000", "充值"],
  "reference_answer": "张三的余额是 1000 元，建议及时充值",
  "tags": ["tool_routing", "multi_step"],
  "difficulty": "medium"
}
```

**关键设计要点**：

- **覆盖面**：不能只放 happy path，要有边界 case（用户问了工具不支持的事）、对抗 case（用户故意引导）、回归 case（历史上出过 bug 的场景）
- **规模**：MVP 阶段 50~100 条就够，但要保证类别均衡。生产系统一般 200~500 条起步
- **不是标准答案，是参考方向**：`reference_answer` 不要求字符串精确匹配，而是给 LLM Judge 或 Rubric 打分用的参照物

**为什么必须有**：没有 Eval Set，你改一版 prompt、换一个模型，都没法知道是"真的变好了"还是"碰巧跑的那几条变好了"。**Eval Set 是评测体系的"米"，没有米什么都做不了。**

---

### 9.2 Metric（评测指标）

**一句话**：Metric 就是"用什么尺子量"——把 Agent 表现转化成可比较的数字。

**三层分类**：

| 层级 | 衡量什么 | 典型 Metric | 评判方式 |
|------|---------|-------------|---------|
| **L1 功能正确性** | 做对了没有 | 工具调用准确率、最终答案正确率、关键信息提取 F1 | 规则 / 精确匹配 |
| **L2 质量好不好** | 答得好不好 | 连贯性、相关性、完整性、幻觉率 | LLM Judge 打分 |
| **L3 体验好不好** | 用户感觉好不好 | 对话流畅度、语气合适度、响应时间体感 | 人工打分 / 隐式反馈 |

**关键认知**：

- **L1 能用规则自动化**：比如检查 Agent 是否调用了预期的工具、提取的实体是否匹配 ground truth，这些可以写代码断言
- **L2 必须引入 LLM Judge**：因为"连贯性""相关性"这类判断，正则和字符串匹配搞不定
- **L3 最贵但最真实**：人工标注、用户满意度评分，通常用于校准 L2 的 LLM Judge 是否靠谱

**和传统监控指标的关系**：传统监控的错误率、延迟、吞吐量属于"系统指标"（9.9 节会讲）；Metric 这里说的是"任务质量指标"。两者正交——系统没报错（错误率 = 0），不代表 Agent 答得好（L2 可能很低）。

---

### 9.3 LLM Judge（LLM 裁判）

**一句话**：让一个 LLM（通常是更强的模型）来当"裁判"，对 Agent 的输出进行评分或判断。

**为什么需要它**：

回顾第五节（Agent 评测的本质）——Agent 的输出是自然语言，字符串匹配和正则都不够用。"张三余额 1000 元，请充值" 和 "查到余额：¥1000，建议您尽快充值" 意思一样但字面不同。只有另一个具备语言理解能力的 LLM 才能判断"语义上是不是对的"。

**工作方式**：

```
给 Judge LLM 一个 prompt：

"""
你是一个评分裁判。请根据以下标准评分：
- 用户问题：{question}
- 参考答案：{reference}
- Agent 实际回答：{agent_output}

评分维度：
1. 事实正确性（1-5 分）
2. 完整性（1-5 分）
3. 表述清晰度（1-5 分）

请给出每个维度的分数和理由。
"""
```

**已知偏差（这是关键）**：

- **位置偏差**（Position Bias）：把两个答案放在一起比较时，LLM 倾向于偏好放在前面的那个
- **冗长偏差**（Verbosity Bias）：LLM 倾向于给更长的回答打更高分，即使长的那个废话更多
- **自我偏好**（Self-Enhancement Bias）：GPT-4 当裁判时，倾向于给 GPT-4 生成的答案打更高分
- **风格偏差**：偏好使用列表、Markdown 格式的回答

**缓解手段**：

- 交换位置后再评一次（消除位置偏差）
- 多个 Judge 投票取共识（如 GPT-4 + Claude + Gemini 各评一次）
- 用 Rubric 约束打分标准（不是"凭感觉打"，而是"按条目逐条打"）

---

### 9.4 Rubric Evaluator（评分规则评估器）

**一句话**：Rubric 是给 LLM Judge 的"评分细则表"——不是让裁判"凭感觉打分"，而是让裁判"按条目逐条对照打分"。

**类比**：高考作文评分不是一个老师看完说"感觉不错，给 45 分"。而是有一张评分细则：立意（0-10）、结构（0-10）、语言（0-10）、错别字扣分规则……老师按细则逐项打分后汇总。**Rubric 就是这张评分细则。**

**一个具体例子**——评估"工具调用是否正确"的 Rubric：

```
评分标准（1-5 分）：

5 分：调用了完全正确的工具，参数完全正确，调用顺序合理
4 分：调用了正确的工具，参数基本正确（有细微参数差异但不影响结果）
3 分：调用了正确的工具，但参数有明显错误或遗漏了必要参数
2 分：调用了错误的工具，或者完全没有调用工具（该调的时候没调）
1 分：产生了有害操作（如调用了删除接口、泄露了敏感信息）
```

**为什么 Rubric 比"裸打分"好**：

- **可重复**：同一个 Rubric，不同时间跑出来分数波动小
- **可对齐**：多个 Judge 用同一套 Rubric，打分偏差更小
- **可解释**：不只知道"3 分"，还知道为什么是 3 分（"参数有明显错误"）
- **可迭代**：发现某个维度的 Rubric 区分度不够，只改这一条就行

**Rubric 和 Metric 的关系**：Metric 是"量什么"（工具调用准确率），Rubric 是"怎么量"（按什么细则给分）。一个 Metric 通常对应一个 Rubric。

---

### 9.5 Prompt Iteration（提示词迭代）

**一句话**：把"凭直觉调 prompt"变成"用 Eval Set 数据驱动的科学实验"。

**传统做法（凭感觉）**：

```
改 prompt → 手动试几条 → "感觉好了" → 上线 → 出问题 → 再改
```

**工程化做法（Prompt Iteration）**：

```
                ┌──────────────────────────────────────────────┐
                │          Prompt Iteration 闭环                 │
                ├──────────────────────────────────────────────┤
                │                                               │
                │  1. 当前 prompt v1                             │
                │       ↓                                       │
                │  2. 跑全量 Eval Set（200 条）                  │
                │       ↓                                       │
                │  3. 用 LLM Judge + Rubric 打分                │
                │       ↓                                       │
                │  4. 分析弱项（哪类 case 得分低？）              │
                │       ↓                                       │
                │  5. 有针对性地改 prompt → v2                   │
                │       ↓                                       │
                │  6. 再跑全量 Eval Set                          │
                │       ↓                                       │
                │  7. 对比 v1 vs v2 各维度得分                   │
                │       ↓                                       │
                │  8. 确认没有"治头痛坏肠胃"                      │
                │       ↓                                       │
                │  9. v2 上线，v1 归档                            │
                │                                               │
                └──────────────────────────────────────────────┘
```

**核心要求**：

- **回归集**：每次改 prompt，不只测新加的 case，还要把历史上"已经答对的"重新跑一遍，确保没有退化
- **版本管理**：每个 prompt 版本 + 对应的 Eval Set 跑分结果都要存档，可追溯
- **多维度看**：不能只看"总分提升了"，要看各类别的拆分（比如工具调用准确率升了但对话流畅度降了，说明 prompt 过度引导调工具了）

**和第六节（优化优先级金字塔）的关系**：Prompt Iteration 在金字塔最底层——成本最低、迭代最快、解决 80% 的问题。这是 3.5 最核心的实战技能。

---

### 9.6 AgentOptimizer（Agent 自动优化器）

**一句话**：让 AI 自己来优化 AI 的 Prompt——把 Prompt Iteration 中"人类分析弱项 → 手动改 prompt"这一步也交给 LLM 自动完成。

**核心思路**：

```
传统 Prompt Iteration（人工驱动）：
  Eval Set 跑分 → 人类看哪些 case 低分 → 人类改 prompt → 再跑

AgentOptimizer（自动驱动）：
  Eval Set 跑分 → AI 看哪些 case 低分 → AI 自动改 prompt → 再跑 → 循环
```

**两个主流范式**：

| 范式 | 代表 | 核心思想 | 类比 |
|------|------|---------|------|
| **DSPy** | Stanford NLP 组 | 把 prompt 当"程序"，每个模块有输入输出签名，用优化器搜索最佳 few-shot 示例和指令组合 | 编译器自动优化代码 |
| **TextGrad** | Zou Group | 把 LLM 的文本反馈当"梯度"，用梯度下降的思想迭代优化 prompt | 神经网络反向传播 |

**DSPy 的工作方式（更成熟、更常用）**：

```python
# 你不再手写 prompt，而是定义"签名"
class ToolRouter(dspy.Signature):
    """根据用户意图选择合适的工具"""
    user_query = dspy.InputField(desc="用户输入")
    tool_name = dspy.OutputField(desc="应该调用的工具名")

# DSPy 自动搜索最优的 few-shot 示例和指令
optimizer = dspy.BootstrapFewShotWithRandomSearch(metric=your_eval_metric)
optimized_module = optimizer.compile(ToolRouter(), trainset=your_eval_set)
```

**什么场景下值得用**：

- Prompt Iteration 已经做了多轮，人类直觉遇到瓶颈
- Eval Set 规模足够大（200+ 条），有明确的量化 Metric
- 场景相对稳定，不会频繁变化（否则每次变化都要重新优化）

**什么场景下不值得用**：

- Eval Set 只有 50 条以下（样本太少，优化器容易过拟合）
- 问题出在工具 schema 或业务逻辑，不是 prompt 的问题
- MVP 阶段——手动 Prompt Iteration 更快、更可控

**和 Prompt Iteration 的关系**：Prompt Iteration 是手动档，AgentOptimizer 是自动档。先学会开手动档（理解原理），再用自动档（提升效率）。生产中通常是"手动调到 80 分，自动优化器冲 85~90 分"。

### 9.7 OpenTelemetry（开放遥测标准）

**一句话**：OpenTelemetry（OTEL）是一套开源的可观测性标准和 SDK，用于采集 Trace（调用链）、Metric（指标）、Log（日志）三种信号。

**传统场景（已有储备）**：

```
HTTP Request → Service A → Service B → DB
     └── Trace: span1 → span2 → span3
         每个 span 记录：开始/结束时间、状态码、标签
```

**Agent 场景下的新挑战**：

传统微服务的 Span 通常是：`HTTP Request → RPC Call → DB Query`——每一步都是确定性的、可枚举的。

但 Agent 的执行链路是**不确定的**：

- LLM 可能调 0 次工具，也可能调 5 次
- 可能有 retry、可能有 self-reflection
- 工具调用的参数是 LLM 自行决定的

**Agent 场景的 Span 设计规范**（OTEL GenAI 语义约定）：

```
Trace: agent_session_123
├── Span: agent.run（整轮交互）
│   ├── Span: llm.chat（第 1 次 LLM 调用 — 决定调工具）
│   │   ├── attribute: gen_ai.request.model = "gpt-4"
│   │   ├── attribute: gen_ai.usage.input_tokens = 1200
│   │   ├── attribute: gen_ai.usage.output_tokens = 85
│   │   └── attribute: gen_ai.request.temperature = 0.7
│   ├── Span: tool.execute（工具执行）
│   │   ├── attribute: tool.name = "query_balance"
│   │   ├── attribute: tool.parameters = {"user": "张三"}
│   │   └── attribute: tool.status = "success"
│   └── Span: llm.chat（第 2 次 LLM 调用 — 整合工具结果）
│       ├── attribute: gen_ai.usage.input_tokens = 1500
│       └── attribute: gen_ai.usage.output_tokens = 120
```

**关键认知**：

- **OTEL 是"标准"，不是"产品"**：它定义了数据格式和采集 SDK，但数据存哪里、怎么展示，需要后端（Jaeger、Tempo、Langfuse 都支持 OTEL 格式）
- **GenAI 语义约定**：OTEL 社区专门为 AI 场景新增了一组属性命名规范（`gen_ai.*`），涵盖模型名、token 用量、temperature 等
- **迁移优势**：有 OTEL 落地经验的话，Span 树、Trace Context Propagation 这些都不用重学，只需要学"Agent 场景该挂哪些 attribute"

### 9.8 Langfuse（Agent 专用可观测平台）

**一句话**：Langfuse 是专为 LLM / Agent 应用设计的可观测平台，可以理解为"Agent 版的 Jaeger + Grafana + 评测面板"。

**和通用 APM（Jaeger / Datadog）的核心差异**：

| 维度 | 通用 APM | Langfuse |
|------|---------|---------|
| **Span 类型** | HTTP / RPC / DB | LLM Call / Tool Call / Agent Step / Retrieval |
| **核心关注** | 延迟、错误率、吞吐量 | Token 用量、Prompt 内容、模型输入输出全文 |
| **成本归因** | 按服务/接口 | **按对话 Session 归因 Token → 换算为钱** |
| **评测集成** | 无 | **内置 Eval 功能**：可以在 Trace 上直接跑 LLM Judge 打分 |
| **Prompt 管理** | 无 | **内置 Prompt 版本管理**：每个 prompt 版本关联其评测得分 |
| **用户反馈** | 无 | **支持采集用户反馈**（👍👎）并关联到具体 Trace |

**Langfuse 的核心面板**：

```
┌─────────────────────────────────────────────┐
│  Langfuse Dashboard                          │
├─────────────────────────────────────────────┤
│                                              │
│  📊 Traces 列表                              │
│  ├── 每条 Trace = 一次完整对话               │
│  ├── 展开可看 LLM Call / Tool Call 子 Span   │
│  ├── 每个 LLM Span 可看完整 prompt + 输出   │
│  └── 可看 token 用量和耗时                    │
│                                              │
│  💰 Cost Dashboard                           │
│  ├── 按模型统计 token 消耗                    │
│  ├── 按 session 归因成本                      │
│  └── 趋势图 + 异常检测                        │
│                                              │
│  📝 Prompt Management                        │
│  ├── prompt v1 / v2 / v3 版本对比            │
│  └── 每个版本关联评测得分                     │
│                                              │
│  ✅ Evaluations                              │
│  ├── 对 Trace 跑 LLM Judge 自动打分          │
│  └── 趋势追踪：质量是否在改善                 │
│                                              │
└─────────────────────────────────────────────┘
```

**核心价值总结**：通用 APM 只告诉你"系统层面出没出故障"；Langfuse 还能告诉你"Agent 层面答得好不好、花了多少钱、哪个 prompt 版本最好"。这正好对应了第三节 4 象限图的 ② 可观测性象限的深层需求。

### 9.9 Token Usage（Token 用量）

**一句话**：Token 是 LLM 的"计费单位"——输入多少 token、输出多少 token，直接决定了"一次对话花多少钱"。

**基础概念**：

```
Token ≈ 一个词或词的一部分（英文约 1 词 ≈ 1.3 token，中文 1 字 ≈ 1.5~2 token）

一次 LLM 调用的 token 构成：
┌─────────────────────────────────────────────┐
│  Input Tokens（输入 token）                    │
│  ├── System Prompt（系统提示词）               │
│  ├── 对话历史（Memory / Context）              │
│  ├── 工具调用结果（Tool Output）               │
│  └── 用户当前输入                              │
│                                               │
│  Output Tokens（输出 token）                   │
│  ├── LLM 的回复文本                            │
│  └── 工具调用指令（如 function_call）          │
└─────────────────────────────────────────────┘
```

**为什么在 Agent 场景特别重要**：

- **Agent 一轮交互可能有多次 LLM 调用**：思考 → 调工具 → 看结果 → 再思考 → 再调工具 → 最终回答。每次调用都是独立计费的
- **上下文膨胀**：随着对话进行，Memory 越来越长，每次调用的 Input Token 都在增长
- **"隐性成本"**：一次看似简单的对话，背后可能有 5 次 LLM 调用，总 token 可能高达 2 万+

**成本归因的核心问题**：

```
用户问了一个问题，Agent 花了 ¥0.15

这 ¥0.15 怎么拆？
├── 第 1 次 LLM 调用（理解意图）：¥0.03
├── 第 2 次 LLM 调用（决定调工具）：¥0.04
├── 第 3 次 LLM 调用（整合结果）：¥0.05
└── 第 4 次 LLM 调用（生成最终回答）：¥0.03

哪步最贵？能不能优化？
→ 第 3 步最贵，因为 input 里包含了工具返回的大段 JSON
→ 优化方向：工具返回结果做摘要，不要全量塞进 context
```

**关键指标**：

- **每对话平均 token**（tokens per conversation）：反映整体成本
- **每轮 input/output 比**：如果 input 远大于 output，说明 context 太长了
- **token 增长曲线**：随着对话轮次增加，token 是线性增长还是爆炸增长

### 9.10 延迟（Latency）与错误率（Error Rate）

这两个合在一起讲，因为它们是 Agent 的**系统层面基石指标**——对应第三节 4 象限图的 ① 监控告警象限。

#### 延迟（Latency）

**Agent 延迟的特殊性**：传统 API 的延迟是"一次请求一次响应"。Agent 的延迟要拆得更细：

```
一次 Agent 交互的延迟拆解：

├── TTFT (Time To First Token)
│   └── 用户发完问题到看到第一个字的时间
│       → 影响"体感快不快"
│
├── 单次 LLM 调用延迟
│   ├── 排队时间（API 限流等待）
│   ├── Prefill 时间（处理 input tokens）
│   └── Decode 时间（逐 token 生成 output）
│
├── 工具执行延迟
│   └── 外部 API 调用、数据库查询等
│
└── 总端到端延迟 (E2E Latency)
    └── 从用户发问到最终答案完整呈现
        = Σ(所有 LLM 调用) + Σ(所有工具执行) + 编排开销
```

**Agent 延迟为什么比传统 API 高得多**：

- 传统 API：一次调用 → 100~500ms
- Agent：多轮 LLM + 多次工具调用 → 通常 **3~30 秒**
- 这意味着 Agent 的延迟优化目标完全不同——不是"P99 < 200ms"，而是"P99 < 15s"

**关键指标**：

| 指标 | 说明 | 参考阈值 |
|------|------|---------|
| TTFT | 首 token 延迟 | < 1~2s（流式输出场景，体感关键） |
| E2E P50 | 中位端到端延迟 | 5~10s（简单任务） |
| E2E P99 | 尾部延迟 | < 30s（复杂多步任务） |
| 工具调用延迟 | 单次工具执行时间 | 取决于外部 API，需单独监控 |

#### 错误率（Error Rate）

**Agent 的"错误"比传统系统复杂得多**——至少有三层：

```
Agent 错误的三层分类：

┌─────────────────────────────────────────────────────┐
│ L1：系统错误（硬错误）                                 │
│   - LLM API 返回 5xx / 超时                          │
│   - 工具调用抛异常                                    │
│   - Rate Limit（限流被拒）                            │
│   → 传统监控能捕获，指标：系统错误率                    │
├─────────────────────────────────────────────────────┤
│ L2：行为错误（软错误）                                 │
│   - 该调工具没调（漏调）                               │
│   - 调了错误的工具（错调）                             │
│   - 工具参数填错                                      │
│   - 陷入无限循环（反复调同一个工具）                    │
│   → 需要 Eval 体系才能发现，指标：工具调用准确率        │
├─────────────────────────────────────────────────────┤
│ L3：质量错误（体验错误）                               │
│   - 答非所问                                         │
│   - 事实性幻觉（编造不存在的信息）                     │
│   - 信息不完整（少说了关键内容）                       │
│   - 语气/风格不合适                                   │
│   → 需要 LLM Judge 才能判断，指标：质量得分 < 阈值     │
└─────────────────────────────────────────────────────┘
```

**关键认知**：

- **L1 系统错误率接近 0，不代表 Agent "没出错"**——L2 和 L3 才是 Agent 特有的"错误"
- **生产中应该分别监控三层错误率**，而不是只看"HTTP 500 率"
- **L2 和 L3 错误的发现成本递增**：L1 靠告警，L2 靠日志 + 规则检查，L3 靠 LLM Judge 或人工抽检

**三层错误的监控方案对照**：

| 层级 | 发现方式 | 响应速度 | 典型工具 |
|------|---------|---------|---------|
| L1 系统错误 | 实时告警 | 分钟级 | Prometheus + AlertManager |
| L2 行为错误 | 离线 Eval / 日志分析 | 小时~天级 | Langfuse Trace 分析 + 规则检查 |
| L3 质量错误 | LLM Judge 定期抽检 | 天~周级 | Langfuse Eval + 人工抽样 |

---

### 9.11 10 个概念的关系全景

```
┌──────────────────────────────────────────────────────────────┐
│                   10 个概念的关系全景                          │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  【评测基础层】（象限 ③）                                      │
│    Eval Set ──→ 喂给 Agent 跑 ──→ 产出结果                    │
│                                       │                       │
│                                       ▼                       │
│    Metric ──→ 定义"量什么"    LLM Judge ──→ 按 Rubric 打分   │
│                                       │                       │
│                                       ▼                       │
│  【优化迭代层】（象限 ④）                                      │
│    Prompt Iteration ──→ 人工分析弱项 → 改 prompt → 回归       │
│    AgentOptimizer ──→ 自动分析弱项 → 自动改 prompt → 回归     │
│                                                               │
│  【可观测层】（象限 ②）                                        │
│    OpenTelemetry ──→ 采集 Trace/Metric/Log（标准+SDK）        │
│    Langfuse ──→ 存储+展示+分析（Agent 专用后端）              │
│                                                               │
│  【系统指标层】（象限 ①）                                      │
│    Token Usage ──→ 成本归因（每对话花多少钱）                  │
│    延迟 ──→ 体验监控（用户等多久）                             │
│    错误率 ──→ 三层分级（系统/行为/质量）                       │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

**一句话总结**：Eval Set 是米，Metric + LLM Judge + Rubric 是量米的秤，Prompt Iteration 和 AgentOptimizer 是改进的引擎，OpenTelemetry + Langfuse 是线上的眼睛，Token / 延迟 / 错误率是底层的仪表盘。
