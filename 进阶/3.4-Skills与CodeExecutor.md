# 3.4 Skills 与 CodeExecutor

> **学习目标**：理解 Skills 作为"可复用工作手册"的本质和 `SKILL.md` 的描述协议；理解 CodeExecutor 作为"让 AI 现场写代码并执行"的桥梁；建立"五种执行环境"的安全谱系认知（本地 / Workspace Runtime / 容器 / Cube / E2B）；理解 Workspace Runtime 这一层"标准化工作环境 + 生命周期管理"的抽象价值；掌握生产级 CodeExecutor 服务的工程落地链路、镜像策略与数据隐私防御。

---

## 一、Skills：AI 的"可复用工作手册"

### 1.1 从一个直觉开始

如果你已经用过 Claude、ChatGPT 这类带 Agent 能力的产品，应该对"Skills"这个词不陌生。我们先用一个最朴素的直觉切入：

> **Skill 本质上就是一个可复用的工具包：里面可能是纯文本的规范，也可能是包含多个工具调用的运行包。**

这个直觉抓住了七成本质。但要把剩下三成补齐，得先把 Skill 和 Tool、MCP Server 这几个容易混的概念拆开。

### 1.2 Tool vs Skill vs MCP Server

这三个概念在日常表达里经常被混用，工程上必须分清：

```
┌─────────────────────────────────────────────────────┐
│  Tool（工具）   vs   Skill（技能）   vs   MCP Server │
├─────────────────────────────────────────────────────┤
│                                                       │
│  Tool                                                 │
│    - 单次原子能力                                     │
│    - 例：read_file、search_content、web_fetch        │
│    - 一次调用 = 一次执行 = 一个结果                    │
│    - 全部注册到 LLM，LLM 看 schema 决定调哪个         │
│                                                       │
│  Skill                                                │
│    - 一套工作流 + 知识包                              │
│    - 例：pdf skill = "处理 PDF 的完整 SOP"           │
│    - 加载后，AI 按 SOP 去组合多个 Tool                │
│    - 一个 Skill 可能产生十几次 tool 调用              │
│                                                       │
│  MCP Server                                           │
│    - 远程的、独立进程的工具服务                       │
│    - 通过 JSON-RPC 协议提供能力                       │
│    - 注册到 Agent 后作为 Tool 被调用                 │
│                                                       │
└─────────────────────────────────────────────────────┘
```

**类比记忆**：

```
Tool      = 瑞士军刀的每一片刀片
Skill     = 修自行车的手册（告诉你什么时候用扳手、什么时候用打气筒）
MCP Server = 外部的"工具服务公司"
```

**关键区别**：
- **Tool 是"原子能力"，Skill 是"知识 + 工作流"，MCP 是"外部服务"**
- Skill 和 Agent 同进程；MCP 跨进程通信
- 加载 Skill 等于让 AI 看一本手册；调用 MCP 等于打电话给外部服务

### 1.3 SKILL.md：技能描述的协议

`SKILL.md` 不是随便写一段说明文档，它有**协议化的结构**：

```markdown
---
name: pdf
description: Use this skill whenever the user wants to do anything with PDF files.
             This includes reading or extracting text/tables from PDFs, combining
             or merging multiple PDFs into one, splitting PDFs apart, ...
location: plugin
---

# 这里是给 AI 看的执行手册

## 1. 何时使用
（什么样的任务该调用这个 skill）

## 2. 工作流程
（按步骤描述 AI 应该怎么做事）

## 3. 调用 scripts/extract.py
（哪些场景下要执行配套脚本）

## 4. 失败兜底
（出错时怎么处理）
```

**frontmatter 三件套必看**：

| 字段 | 作用 |
|------|------|
| `name` | 技能唯一标识 |
| `description` | **AI 判断"要不要加载这个 skill"的唯一依据** |
| `location` | skill 存储位置（plugin/manager 等） |

⚠️ **description 是整个 skill 设计的关键**——写不好，AI 发现不了它的存在。

### 1.4 Progressive Disclosure：渐进式披露

这是 Skills 设计里最聪明的机制，专门用来**省 context**：

```
┌──────────────────────────────────────────────────┐
│  第一层：启动时注入"目录"（几十 token / skill）    │
│    系统提示词里只放 name + description + location │
│    AI 看到的是一份"技能目录"                       │
│                                                    │
│  第二层：AI 判断要用 → 显式调用 use_skill          │
│    SKILL.md 全文才加载进上下文                     │
│                                                    │
│  第三层：SKILL.md 里引用了 reference/xxx.md        │
│    或 scripts/yyy.py → AI 按需读取                 │
└──────────────────────────────────────────────────┘
```

**为什么这么设计？** 如果一上来就把所有 Skill 全部塞进 system prompt，10 个 Skill 就能吃掉几万 token。Progressive Disclosure 让 context 只承载"当前任务真正需要的"内容。

### 1.5 一个完整 Skill 的运行包形态

```
pdf-skill/
├── SKILL.md              ← AI 入口（必读）
├── scripts/              ← 可执行脚本
│   ├── extract_text.py
│   └── merge_pdfs.py
├── reference/            ← 大段参考资料（AI 按需读）
│   └── pdf_spec.md
└── assets/               ← 模板、示例文件
    └── template.pdf
```

**AI 调用一个 Skill 的完整流程**：

```
1. 接收任务："帮我合并这两个 PDF"
2. 扫描 available_skills 目录，发现 pdf 的 description 命中
3. 调用 use_skill("pdf") → 加载 SKILL.md 全文进上下文
4. 按 SKILL.md 指示：
   - read_file 看 reference/pdf_spec.md
   - 执行 scripts/merge_pdfs.py
5. 把结果反馈给用户
```

### 1.6 Skill 的两种典型形态

```
形态 A：纯描述型 Skill
  只有 SKILL.md
  本质就是"结构化的提示词模板"
  适用：流程标准化的任务（如"代码审查 SOP"）

形态 B：带资源型 Skill
  SKILL.md + scripts/ + assets/ + reference/
  AI 可以调用脚本、读取参考资料
  适用：需要执行具体动作的任务（如"PDF 处理"）
```

## 二、CodeExecutor：让 AI 直接写代码跑代码

### 2.1 为什么需要 CodeExecutor

Function Calling 是"AI 选已有工具"，CodeExecutor 是"AI 现场写代码"。两者本质区别：

```
Function Calling：
  AI → 选 calculate_sum(a, b) → 框架执行已注册函数
       （能干啥事先就定了，"菜单点餐"）

CodeExecutor：
  AI → 生成 "import pandas; df.groupby('city').mean()..."
     → 框架找个地方把这段代码跑了 → 返回结果
       （能干啥取决于 AI 当场想出啥，"开放式厨房"）
```

### 2.2 CodeExecutor 解决了 LLM 的三大固有缺陷

```
缺陷 1：算数不准
  LLM 算 374×521 经常错，但 Python 永远不会错
  → CodeExecutor 让 AI"用工具算"而不是"用脑算"

缺陷 2：上下文长度有限
  100MB 的 CSV 塞不进 context
  → CodeExecutor 让 AI 用 pandas 处理，只把摘要带回来

缺陷 3：无法直接产生副作用
  LLM 只能输出文本，不能"画图""存文件""发请求"
  → CodeExecutor 是 LLM 接触真实世界的"手"
```

这三点决定了 CodeExecutor 不是可有可无的功能，而是**让 LLM 从"会聊天"进化到"能干活"的关键基础设施**。

### 2.3 CodeExecutor 在 Agent 副作用边界中的位置

很多人会把 CodeExecutor 理解成"Agent 执行操作的边界"——这个直觉方向对，但概念合并了。准确说法是：

```
┌──────────────────────────────────────────────────────────┐
│                Agent 的"执行能力"全景                       │
├──────────────────────────────────────────────────────────┤
│                                                            │
│  ① 纯推理（无副作用）                                       │
│     LLM 生成文本 → 不动外部世界                             │
│                                                            │
│  ② Function Calling（预定义工具）                          │
│     调 calculate_sum(1,2) / get_weather("北京")            │
│                                                            │
│  ③ MCP / 外部服务调用                                       │
│     远程 RPC 调外部能力                                     │
│                                                            │
│  ④ 文件系统操作                                             │
│     read_file / write_file（路径白名单）                    │
│                                                            │
│  ⑤ 命令执行（Shell Executor）                              │
│     run_command("git status")                              │
│                                                            │
│  ⑥ CodeExecutor ← 本章主角                                 │
│     执行 AI 现场生成的任意代码                              │
│                                                            │
│  ⑦ 浏览器自动化（Browser Use）                              │
│     Playwright/Puppeteer 操作真实浏览器                    │
│                                                            │
└──────────────────────────────────────────────────────────┘

         ↑                                            ↑
    能力越强                                      安全风险越大
    自由度越高                                    实现越复杂
```

**核心规律**：从 ① 到 ⑦，**能力的开放性递增，相应地安全边界的复杂度也递增**。

- ② Function Calling：AI 只能在你定好的"菜单"里选，菜单边界 = 安全边界
- ⑥ CodeExecutor：AI 可以写任意代码，**没有菜单**，所以安全边界要靠**执行环境的隔离**来兜底

**一句话定义**：

> **CodeExecutor = Agent 把"开放式代码生成"落地为"受控副作用"的桥梁。**
> 它要解决的核心矛盾是：AI 的能力是开放的（能写任何代码），系统的安全是封闭的（不允许任何代码动我硬盘）。CodeExecutor 就是这两者之间的"调停者"。

### 2.4 CodeExecutor 的典型应用场景

CodeExecutor 远不止"代码辅助"，列一下真实生产里的主流场景：

```
场景 1：数据分析（最高频，可能占 60%+）
  用户："分析这个 CSV，画出销量趋势"
  AI 生成 pandas + matplotlib 代码 → 沙箱跑 → 返回图片
  代表产品：ChatGPT Advanced Data Analysis、Claude Analysis Tool

场景 2：数学 / 科学计算
  本质：用代码弥补 LLM 算数不准的硬伤

场景 3：文件批处理
  用户："把这 100 个 PDF 都转成 markdown"
  AI 生成循环代码 → 批量产出

场景 4：代码辅助
  - 单元测试运行：写完测试代码跑一下
  - 代码片段验证：这段 SQL 能不能跑通
  - 调试：在沙箱里复现 bug

场景 5：Agent 自检 / 自修正（Self-Correcting）
  AI 生成代码 → 执行 → 报错 → AI 看到报错 → 修改 → 再执行
  专业名词：Reflexion / ReAct loop

场景 6：动态工具创造（最前沿）
  AI 发现没有合适的 Tool → 现场写一个 → 注册成 Tool → 用
  代表：Voyager (Minecraft Agent) / Toolformer
```

记忆口诀：
- 数据分析是**最大的市场**
- 数学计算是**LLM 的拐杖**
- 代码辅助是**最自然的延伸**
- 自修正是**最聪明的玩法**
- 动态工具创造是**最未来的方向**

## 三、五种执行环境：安全谱系

CodeExecutor 的安全性，本质上取决于"代码在哪里跑"。我们用一条谱系把所有方案排开：

```
易用性 高 ─────────────────────────────────────▶ 低

本地执行 ──▶ Workspace Runtime ──▶ 容器执行 ──▶ Cube 沙箱 ──▶ E2B 云沙箱

安全性 低 ─────────────────────────────────────▶ 高
```

### 3.1 本地执行（Local Executor）

```python
# 伪代码
exec(llm_generated_code)   # ← 看到这行就该警觉了
```

**机制**：直接在跑 Agent 的那个 Python 进程里 `exec()` 或 `subprocess.run()`。

**安全边界**：**基本没有**。AI 可以：
- `os.system("rm -rf /")` 删你硬盘
- `requests.get("http://内网IP/admin")` 打内网
- 读你的 `~/.ssh/id_rsa`

**适用**：**只用于本地开发调试**，绝不能上生产。

### 3.2 Workspace Runtime（工作区运行时）

⚠️ **Workspace Runtime 是一个比"执行环境"更高一层的抽象**，我们在第四节单独讲它的结构。这里先看它作为"执行环境"时的安全边界。

```
Workspace Runtime 的边界：
  ✓ 允许：读写指定工作区目录内的文件
  ✓ 允许：执行 git、npm、python 等工具
  ✗ 禁止：访问工作区外的文件（路径白名单）
  ⚠️ 网络：通常不限制
  ⚠️ 进程：可以 fork 任意子进程
```

**本质**：**路径沙箱 + 用户确认机制**，不是真正的隔离。Claude Code、CodeBuddy 这类 IDE Agent 是典型代表——靠"用户在旁边盯着 + 危险命令需点确认"作为安全兜底。

### 3.3 容器执行（Docker / Container）

```
Agent 进程
   │
   │ docker run --rm python:3.11 -c "<生成的代码>"
   ▼
┌────────────────────────────┐
│  Docker 容器（一次性）       │
│  - 独立文件系统             │
│  - 独立进程空间             │
│  - 可限制 CPU / 内存        │
│  - 可断网                  │
└────────────────────────────┘
```

**安全边界**：
- ✅ 文件系统隔离（除非挂载宿主目录）
- ✅ 进程隔离
- ✅ 可控网络（`--network=none`）
- ⚠️ **不是真正的虚拟化**：共享宿主内核，存在容器逃逸漏洞（CVE）
- ⚠️ 启动慢（每次几百毫秒到秒级）

**坑点**：很多人挂载了 `-v /:/host` 或 `--privileged`，等于没沙箱。

### 3.4 Cube 沙箱（轻量级安全沙箱）

**Cube Sandbox** 是一个具体的产品，定位非常聪明——它是 **E2B 协议兼容的私有化沙箱平台**：

```
═══════════════════════════════════════════════════
Cube Sandbox = E2B 协议兼容的私有化沙箱平台

  ┌─────────────────────────────────────┐
  │ 你的 Agent（用 E2B SDK 写）          │
  └────────────┬────────────────────────┘
               │ E2B HTTP 协议
               ▼
       ┌───────┴────────┐
       │                 │
  E2B 公有云         Cube Sandbox（私有化）
  （海外 SaaS）        （部署在你机房）

  应用代码完全不变，只改一个 URL
═══════════════════════════════════════════════════
```

**它解决了什么问题**：E2B 虽好，但对中国 / 企业用户有三大痛点：

```
痛点 1：跨境合规  E2B 公有云在美国 → 数据出境问题
痛点 2：网络延迟  从国内 Agent 到 E2B → 几百毫秒往返
痛点 3：自建成本  想用 Firecracker 自建 → 要懂 microVM、调度、运维
```

**Cube 的技术架构**：

```
┌────────────────────────────────────────────────────┐
│  Cube Sandbox 架构                                  │
├────────────────────────────────────────────────────┤
│  控制面：接收 E2B 协议、调度、鉴权、计费             │
│  数据面：KVM microVM 池                             │
│         - 每个 session 一台独立微虚机                │
│         - 100ms 级冷启动                            │
│         - 单机百并发                                │
│  模板系统：code-interpreter-v1 等预装环境            │
└────────────────────────────────────────────────────┘
```

**一句话**：Cube ≈ **"E2B 的国产平替 + 私有化版本"**。

### 3.5 E2B 云沙箱

[E2B](https://e2b.dev/)（Execute to Build）是这个赛道的**事实标准**，专门为 AI Agent 设计的**云端代码沙箱即服务**：

```
你的 Agent ──HTTPS──▶  E2B 云端
                       │
                       ▼
                ┌──────────────────────┐
                │ 启动 Firecracker VM   │ ← ~200ms
                │ 完整 Ubuntu 22.04     │
                │ 预装 Python 数据栈    │
                │ 独立内核（不和邻居共享）│
                └──────────────────────┘
                       │
                  执行代码
                       │
                       ▼
                返回 stdout / stderr / 文件
                       │
                       ▼
                 销毁 VM（默认）
```

**核心特性**：

| 特性 | 说明 |
|------|------|
| 隔离技术 | Firecracker microVM，硬件级隔离 |
| 冷启动 | ~200ms |
| 生命周期 | 默认每次执行一台 VM，用完销毁 |
| 持久化 | 同 session 内文件保留，支持多步任务 |
| 预装环境 | Python / Node + 几十个常用库 |
| 计费 | 按 VM 在线时长 |
| 客户 | Perplexity、Cursor、Manus 等 |

**E2B SDK 使用示例**（5 行代码）：

```python
from e2b import Sandbox

sbx = Sandbox()                       # 启动云端沙箱
sbx.files.write("data.csv", csv)      # 上传文件
result = sbx.run_code("""             # 执行任意 Python
    import pandas as pd
    df = pd.read_csv('data.csv')
    print(df.describe())
""")
print(result.stdout)                  # 拿结果
```

### 3.6 五种环境对比总表

| 环境 | 隔离强度 | 启动延迟 | 网络默认 | 适用场景 | 危险等级 |
|------|---------|---------|---------|---------|---------|
| 本地执行 | 无 | 0ms | 通 | 本地调试 | 🔴 高 |
| Workspace Runtime | 弱（路径） | 0ms | 通 | IDE Agent（有用户监督）| 🟠 中高 |
| 容器执行 | 中 | 100ms~1s | 可控 | 自建生产环境 | 🟡 中 |
| Cube 沙箱 | 强 | ~100ms | 可控 | 国内企业 / 强合规场景 | 🟢 低 |
| E2B 云沙箱 | 强 | ~200ms | 可控 | SaaS 产品 / 海外团队 | 🟢 低 |

### 3.7 Cube vs E2B 的精确对比

| 维度 | E2B | Cube Sandbox |
|------|-----|--------------|
| 性质 | 公有云 SaaS | 私有化部署产品 |
| 部署位置 | 海外（美国为主）| 你自己的机房 |
| 协议 | E2B 协议（自定义）| **100% 兼容 E2B 协议** |
| 底层 | Firecracker microVM | KVM microVM |
| 冷启动 | ~200ms | ~100ms |
| 数据合规 | 需考虑出境 | 完全本地 |
| 计费 | 按时长付费 | 一次性部署 + 自维护 |
| 网络延迟 | 跨境 200ms+ | 内网 ~10ms |
| 适用场景 | 海外产品 / 个人开发者 | 国内企业 / 金融 / 政企 |

**精炼记忆**：
- **E2B = 沙箱界的 AWS Lambda**（开箱即用、按需付费）
- **Cube = E2B 的国产私有化版**（同协议、自部署、数据不出门）

## 四、Workspace Runtime：执行环境的标准化抽象

前面提到 Workspace Runtime 是"比执行环境高一层的抽象"，这一节展开讲。

### 4.1 一个直觉：Workspace 是什么

```
比喻：
  CodeExecutor = 程序员（会写代码、会跑代码）
  Workspace    = 程序员的办公桌（有抽屉放材料、有桌面工作、有产出筐）
  Runtime      = 这间办公室的水电气网（基础设施）

  Workspace Runtime = 为 Agent 准备的"标准化办公室 + 配套基础设施"
```

**工程定义**：

```
═══════════════════════════════════════════════════
Workspace Runtime =
  一个为 Agent 代码执行准备的、有结构的、有生命周期的、
  可控的"工作环境"

  包含三件事：
    ① 文件系统结构（固定目录约定）
    ② 执行后端（具体哪个 Executor）
    ③ 生命周期管理（创建、初始化、复用、清理）
═══════════════════════════════════════════════════
```

### 4.2 固定的目录约定

每次 Agent 执行代码时，它面前都有这样一个标准目录结构：

```
workspace/              ← 一次执行的"办公桌"
├── work/
│   ├── inputs/         ← 抽屉：用户上传的文件放这里
│   │   ├── data.csv
│   │   └── report.pdf
│   └── (任意中间文件)   ← 桌面：处理中的临时文件
│
├── out/                ← 产出筐：最终结果放这里
│   ├── chart.png
│   └── summary.md
│
└── runs/               ← 笔记本：日志、调试信息
    └── 2026-06-26.log
```

**为什么要做这种约定？**

```
没有约定的世界：
  AI 写代码 → 文件存哪？/tmp？/var？/home？
            → 用户上传的文件在哪？AI 怎么找？
            → 哪些是要返回给用户的？怎么挑出来？

有约定的世界：
  AI 看到 SKILL.md 里规定：
    "用户文件在 work/inputs/，最终产出放 out/"
  → AI 写代码就有"地图"，框架收集结果也有"地图"
  → 用户上传 → 框架自动放 inputs/
  → 执行完成 → 框架自动从 out/ 收集
```

**约定大于配置**——这是 Workspace Runtime 给开发者的第一个礼物。

### 4.3 执行后端：Workspace 上面"装着哪个 Executor"

Workspace 是"办公室"，但办公室里"做事的人"可以换：

```
            同一个 Workspace 抽象
                    │
                    ▼
        ┌─────────────────────────┐
        │ Workspace               │
        │ ├── work/inputs/        │
        │ ├── out/                │
        │ └── runs/               │
        └────────────┬────────────┘
                     │ 上层抽象一致
                     ▼
              ┌──────┴───────┐
              ▼              ▼
        local Executor    container Executor   ...
        (本地跑)          (Docker 跑)

应用代码不变，换个 Executor 就换了隔离强度
```

### 4.4 Workspace 的生命周期

Runtime 这个词的核心——它管理 Workspace 的**整个时间维度的流程**：

```
┌─────────────────────────────────────────────────────┐
│  Workspace 的生命周期                                │
├─────────────────────────────────────────────────────┤
│                                                      │
│  ① 创建（Create）                                    │
│     准备目录结构、分配底层资源                       │
│                                                      │
│  ② 初始化（Init Hook）                               │
│     拉取固定输入文件、跑初始化命令（如 pip install） │
│     只执行一次，结果被复用                           │
│                                                      │
│  ③ Stage 输入（StageInputs）                         │
│     用户上传的文件 → 自动放到 work/inputs/           │
│                                                      │
│  ④ 执行（RunProgram）                                │
│     AI 生成代码 → 在 workspace 里跑                  │
│     可能多次执行（多轮对话每轮一次）                 │
│                                                      │
│  ⑤ 收集产物（Collect）                               │
│     自动扫描 out/，返回给上层调用方                  │
│                                                      │
│  ⑥ 持久化策略                                        │
│     PerSession：同 session 内文件保留                │
│     PerTurn：每轮独立 workspace                      │
│                                                      │
│  ⑦ 销毁（Destroy）                                   │
│     容器停止、文件清理、资源回收                     │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 4.5 用一个真实场景串起来

**场景**："帮我把这个 CSV 做一份销售分析报告，要有图表"（多轮对话）

```
═══════════════════════════════════════════════════
第 1 轮：用户上传 sales.csv 问"帮我分析销售数据"
═══════════════════════════════════════════════════

① 创建 workspace（绑定 container backend）
   workspace-abc123/
     ├── work/inputs/
     ├── out/
     └── runs/

② 跑 init hook：装好 pandas/matplotlib

③ StageInputs：把 sales.csv 放到 work/inputs/sales.csv

④ Agent 生成代码并执行：
   df = pd.read_csv('work/inputs/sales.csv')
   df['month'].value_counts().plot()
   plt.savefig('out/chart1.png')

⑤ Collect：扫描 out/，返回 chart1.png
   反馈给 LLM："已生成 chart1.png，显示了月度趋势"

═══════════════════════════════════════════════════
第 2 轮：用户追问"基于这个图，再算下季度环比"
═══════════════════════════════════════════════════

⑥ 持久化策略 = PerSession
   → 复用 workspace-abc123（不新建）
   → sales.csv 还在！chart1.png 还在！

⑦ Agent 生成代码：
   df = pd.read_csv('work/inputs/sales.csv')  ← 文件还在
   df['quarter_pct'] = df['sales'].pct_change(3) * 100
   df.to_excel('out/quarterly.xlsx')

⑧ Collect 新产物：quarterly.xlsx

═══════════════════════════════════════════════════
第 3 轮：用户："谢谢，结束"
═══════════════════════════════════════════════════

⑨ Session 结束 → Workspace 销毁
   容器停止、磁盘清理、资源回收
```

### 4.6 Workspace Runtime 解决了哪些"不做就会痛"的问题

```
┌───────────────────────────────────────────────────────┐
│  没有 Workspace Runtime 时，每个 Agent 开发者会撞墙：  │
├───────────────────────────────────────────────────────┤
│                                                        │
│  痛 1：文件路径混乱                                    │
│    "AI 生成的代码该读 /tmp/xxx 还是 ./inputs/xxx？"   │
│                                                        │
│  痛 2：用户上传文件无处安放                            │
│    用户传了个 PDF，怎么告诉 AI 它在哪？                │
│                                                        │
│  痛 3：产物收集是个体力活                              │
│    AI 写了 5 个文件，哪个是给用户的？                  │
│                                                        │
│  痛 4：多轮对话的状态丢失                              │
│    第 2 轮要用第 1 轮的中间结果，文件丢了              │
│                                                        │
│  痛 5：环境初始化重复执行                              │
│    每轮对话都 pip install 一遍，慢死                   │
│                                                        │
│  痛 6：换执行后端要改业务代码                          │

│                                                        │
└───────────────────────────────────────────────────────┘

Workspace Runtime 把这 6 个坑一次性填平。
```

## 五、容器执行的工程落地

这一节是这一章里**最工程化、最有"实战交付感"**的部分。

### 5.1 完整工程链路（8 步时序图）

```
┌──────────┐                                              ┌──────────┐
│  Agent   │                                              │  容器     │
│ (LLM)    │                                              │ Runtime   │
└────┬─────┘                                              └────┬─────┘
     │                                                         │
     │ ① LLM 生成代码字符串                                     │
     │   "import pandas; df = pd.read_csv('/data/sales.csv')" │
     │                                                         │
     │ ② 框架接收代码，做静态校验                                │
     │   - AST 解析，黑名单检查（禁 import os.system）          │
     │   - 长度限制（防 prompt 注入塞超长代码）                  │
     │                                                         │
     │ ③ 准备工作目录                                          │
     │   /tmp/sandbox-xxx/                                    │
     │     ├── code.py        ← AI 生成的代码                  │
     │     └── input/         ← 用户上传的文件                  │
     │                                                         │
     │ ④ 启动容器（关键参数全在这）                              │
     │   ───────────────────────────────────▶                  │
     │   docker run                                            │
     │     --rm                          ← 用完即删             │
     │     --network=none                ← 默认断网             │
     │     --memory=512m                 ← 内存限制             │
     │     --cpus=1.0                    ← CPU 限制             │
     │     --pids-limit=64               ← 进程数限制           │
     │     --read-only                   ← 根文件系统只读       │
     │     --tmpfs /tmp:size=100m        ← /tmp 可写但限大小    │
     │     -v /tmp/sandbox-xxx:/work:ro  ← 代码挂载（只读）     │
     │     -v /tmp/sandbox-xxx/out:/out  ← 输出目录（可写）     │
     │     --user 1000:1000              ← 非 root 运行         │
     │     --cap-drop=ALL                ← 丢弃所有 capability  │
     │     --security-opt=no-new-privileges                    │
     │     python-sandbox:3.11           ← 预 build 的镜像      │
     │     timeout 30 python /work/code.py  ← 超时控制          │
     │                                                         │
     │ ⑤ 容器内执行                                            │
     │                                                         │
     │ ⑥ 容器退出，收集结果                                     │
     │   ◀───────────────────────────────────                  │
     │   - exit code                                          │
     │   - stdout / stderr 文本                                │
     │   - /tmp/sandbox-xxx/out/ 下的产物文件                   │
     │                                                         │
     │ ⑦ 结果格式化为 LLM 可读                                  │
     │   {                                                     │
     │     "stdout": "...",                                   │
     │     "stderr": "...",                                   │
     │     "files": ["chart.png"],                            │
     │     "exit_code": 0                                     │
     │   }                                                     │
     │                                                         │
     │ ⑧ 反馈给 LLM 继续推理 + 清理工作目录                     │
     │                                                         │
```

### 5.2 三个最容易踩坑的工程细节

**细节 1：镜像是预先 build 的"沙箱模板"**

```dockerfile
# python-sandbox:3.11 的 Dockerfile（示意）
FROM python:3.11-slim

# 预装数据分析全家桶
RUN pip install --no-cache-dir \
    pandas numpy matplotlib seaborn \
    scipy scikit-learn requests

# 创建非 root 用户
RUN useradd -m -u 1000 sandbox
USER sandbox
WORKDIR /work

CMD ["python"]
```

⚠️ **常见误解**：每次 AI 生成代码 → 现场 build 镜像 → 跑
✅ **实际做法**：镜像预先 build 好 → 每次只是启动新容器 → 跑代码 → 销毁

**细节 2：文件交换靠"挂载卷"，不是塞参数**

```
代码进容器：    宿主机文件 → 挂载卷 → 容器内 /work/code.py
数据进容器：    用户上传文件 → 挂载到 /work/input/
结果出容器：    容器写到 /work/out/ → 宿主机自动可见
日志出容器：    docker logs <容器ID> 抓 stdout/stderr
```

**细节 3：超时是"两层防线"**

```
第一层：容器内 timeout 30 python code.py
   └─ Python 进程自己超时退出

第二层：宿主机 docker run 后台 + Timer
   └─ 30 秒后强制 docker kill <容器ID>
   └─ 防止容器内的 timeout 被代码绕过
```

### 5.3 容器执行的"安全边界"到底来自哪里

容器的安全边界本质来自 **Linux 内核的 4 大隔离机制**，Docker 只是把它们包装好用：

```
┌────────────────────────────────────────────────────────┐
│  机制 1：Namespaces（命名空间）—— 视图隔离              │
│  让容器"看不到"宿主机的资源                              │
│    PID / Network / Mount / UTS / IPC / User namespace  │
│  本质：内核给每个容器维护一套"假的全局视图"               │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│  机制 2：Cgroups（控制组）—— 资源限制                   │
│  让容器"用不超"宿主机的资源                              │
│    cpu / memory / pids / blkio cgroup                  │
│  本质：内核在调度时给容器加配额                          │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│  机制 3：Capabilities（能力位）—— 权限削减              │
│  让容器"做不了"危险操作                                  │
│    --cap-drop=ALL 表示全部丢弃 → 即使容器内是 root，    │
│    也不能改网络配置、装载文件系统等                      │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│  机制 4：Seccomp / AppArmor / SELinux —— 系统调用过滤  │
│  让容器"调不通"危险的系统调用                            │
│  本质：内核在 syscall 入口拦截危险调用                   │
└────────────────────────────────────────────────────────┘
```

**Docker 的天生短板**：

```
⚠️ 容器和宿主共享同一个内核
   一旦内核漏洞被利用 → 容器逃逸 → 拿到宿主机权限

历史 CVE：
  - CVE-2019-5736：runc 漏洞，覆盖宿主机 runc 二进制
  - CVE-2022-0185：内核 fsconfig 漏洞，容器内获取宿主 root
  - CVE-2024-21626：runc 文件描述符泄漏
```

这就引出了下一节的"补内核漏洞"方案。

### 5.4 内核级沙箱：gVisor / Firecracker / Kata

Cube 这类沙箱的核心思路是：**不要和宿主共享内核**。但又不想用传统虚拟机（太重、启动慢）。所以有了三种巧妙的实现路径：

**路径 A：gVisor —— "用户态假内核"**

```
传统容器：
  容器进程 → syscall → 宿主内核（共享，危险）

gVisor：
  容器进程 → syscall → ┌─────────────────┐
                       │ Sentry（用户态） │  ← Go 写的"假内核"
                       │ 实现 200+ syscall│
                       └────┬────────────┘
                            │ 只用极少数安全的真 syscall
                            ▼
                       宿主内核
```

代价：Syscall 性能下降 20%~50%。

**路径 B：Firecracker —— "微型虚拟机"**

```
传统 VM（KVM + QEMU）：
  完整模拟 PC：BIOS、网卡、声卡、显卡... → 启动几十秒

Firecracker：
  砍掉所有不必要的虚拟硬件
  只保留 virtio-net + virtio-block + 串口 → 启动 125ms
  内存占用 < 5MB
```

**产业应用**：
- AWS Lambda 和 Fargate 底层是 Firecracker
- E2B 的底层也是 Firecracker
- Cube 的底层很可能也是 Firecracker（性能指标高度匹配）

**路径 C：Kata Containers —— "兼容容器接口的微 VM"**

- 对外：像普通 Docker 容器（兼容 OCI、Kubernetes）
- 对内：每个容器其实是一个 Firecracker/QEMU 微 VM

**三种方案对比**：

| 方案 | 隔离原理 | 启动 | Syscall 性能 | 兼容性 | 典型用户 |
|------|---------|------|-------------|--------|---------|
| Docker | 共享内核 + Namespaces | 50ms | 100% | 极好 | 常规生产 |
| gVisor | 用户态假内核 | 100ms | 50%~80% | 90% | Google Cloud Run |
| Firecracker | 微型 VM | 125ms | 95%+ | 完整 Linux | AWS Lambda、E2B、Cube |
| Kata | 微 VM + OCI 兼容 | 200ms | 90%+ | 极好 | 安全容器平台 |

## 六、镜像策略：让 CodeExecutor "时间更快"

容器执行的核心痛点之一是"镜像该怎么准备"。本节讲三个核心工程问题。

### 6.1 问题 1：镜像粒度的三种策略

**策略 A：单一巨型镜像（All-in-One）** —— 装 200 个常用库
- ✅ AI 写啥都能跑、调度简单
- ❌ 镜像 3~8 GB、启动慢、攻击面大、升级痛苦

**策略 B：场景化镜像（Scenario-based，最主流）** —— 按任务类型切分

```
python-base:3.11           ← 纯 Python（轻量、快速启动）
python-data:3.11           ← + pandas/numpy/matplotlib
python-ml:3.11             ← + scikit-learn/torch
python-web:3.11            ← + requests/bs4/lxml
python-pdf:3.11            ← + pdfplumber/PyPDF2
python-image:3.11          ← + Pillow/opencv
node-js:20                 ← Node 环境
go-runner:1.22             ← Go 环境
```

调度逻辑：

```python
def pick_image(code: str) -> str:
    if "import pandas" in code or "import numpy" in code:
        return "python-data:3.11"
    if "import torch" in code or "import sklearn" in code:
        return "python-ml:3.11"
    if "from PIL" in code or "import cv2" in code:
        return "python-image:3.11"
    return "python-base:3.11"
```

**这是 80% 生产场景的最佳实践**。

**策略 C：基础镜像 + 运行时安装（动态注入）** —— 缺包了在容器内 pip install
- ✅ 镜像极简、无穷扩展
- ❌ 首次执行慢（几秒到几十秒）、需开网络（安全风险）、版本不可预测

**三种策略对比**：

| 维度 | 巨型镜像 | 场景化镜像 | 动态安装 |
|------|---------|----------|---------|
| 镜像数量 | 1 | 5~10 | 1~2 |
| 单镜像大小 | 5GB+ | 200MB~1GB | 100MB |
| 启动延迟 | 慢 | 快 | 快 + pip 慢 |
| 维护成本 | 低 | 中 | 低 |
| 安全性 | 差 | 好 | 差 |
| **推荐度** | ⚠️ Demo 用 | ✅ **主流** | ⚠️ 开发用 |

### 6.2 问题 2：AI 代码缺包怎么办？四层兜底

```
┌─────────────────────────────────────────────────────┐
│  兜底层 1：执行前静态检测（毫秒级，最优）              │
│    AST 解析代码 → 提取 import 列表                   │
│    查表：所有 import 在哪个镜像里 → 选最匹配的镜像   │
├─────────────────────────────────────────────────────┤
│  兜底层 2：升级到"更全"的镜像（秒级）                  │
│    base 不够 → 切到 python-data → 切到 python-ml    │
├─────────────────────────────────────────────────────┤
│  兜底层 3：临时容器内 pip install（十秒级）           │
│    确实冷门库 → 起容器 → 临时 pip install → 跑       │
│    需要短暂放开网络（限制到 pypi.org）                │
├─────────────────────────────────────────────────────┤
│  兜底层 4：上报 → 异步沉淀到新镜像（分钟~小时级）       │
│    统计高频缺包 → 触发"镜像演进流水线"                │
│    自动 build 新版本 → 灰度发布替换                  │
│    系统越用越快                                      │
└─────────────────────────────────────────────────────┘
```

**架构思想**：**让"冷启动慢"变成"渐近自愈"**。

### 6.3 问题 3：镜像怎么"快速更新"

**技巧 1：利用 Docker 的"分层缓存"**

```dockerfile
# 把变化频率低的放上面，频率高的放下面
FROM python:3.11-slim         # 层 1：基础层（100MB，几乎不变）
RUN apt-get install ...       # 层 2：系统依赖（50MB，季度变化）
RUN pip install pandas numpy  # 层 3：核心依赖（300MB，月度变化）
RUN pip install matplotlib    # 层 4：可选依赖（200MB，周度变化）
COPY config /etc/...          # 层 5：配置（几 KB，频繁变化）
```

改最上层只需重 build 层 5，更新速度从分钟级降到秒级。

**技巧 2：基础镜像复用（共享 Base）**

```
       ┌──────────────────┐
       │ company-base:1.0 │  ← 装公司通用工具、监控 agent
       └────────┬─────────┘
                │
       ┌────────┴─────────┐
       ▼                   ▼
  python-base:3.11    node-base:20
       │                   │
       ▼                   ▼
  python-data         python-ml ...
```

**技巧 3：镜像更新的"灰度发布"**

```
1. 构建新版本：python-data:3.11.5
2. 推送到镜像仓库
3. 调度器改流量比例：
     3.11.4 → 95%，3.11.5 → 5%
4. 观察失败率、延迟
5. 逐步提到 100%，老版本 1~2 周后下线
```

**技巧 4：镜像预热 + 容器池（最关键）**

```
每台 Node 维护一个"温热容器池"：
  - python-data:3.11.5  × 10 个（已启动、等待中）
  - python-ml:3.11.5    × 5 个
  - python-base:3.11.5  × 20 个

请求来 → 直接拿一个温热容器塞代码 → 启动延迟 ≈ 10ms
用完销毁 → 后台再补一个新的进池子
```

**这就是 E2B 能做到 200ms 启动的核心秘密**——它后台一直在维护温热的 Firecracker microVM 池。

## 七、安全设计的七层防御

不管选哪种执行环境，**生产级 CodeExecutor 都要叠加防御**：

```
┌─────────────────────────────────────────┐
│ 第 1 层：代码静态扫描                    │
│   AST 解析，禁用 import os / subprocess │
├─────────────────────────────────────────┤
│ 第 2 层：执行环境隔离                    │
│   容器 / 沙箱（五种环境选一）             │
├─────────────────────────────────────────┤
│ 第 3 层：资源配额                        │
│   CPU 1 核 / 内存 512MB / 超时 30s      │
├─────────────────────────────────────────┤
│ 第 4 层：网络白名单                      │
│   默认断网，按需放通特定域名             │
├─────────────────────────────────────────┤
│ 第 5 层：文件系统只读 + 临时目录          │
│   除 /tmp 外只读，重启清空               │
├─────────────────────────────────────────┤
│ 第 6 层：用户/调用频率限流                │
│   防 AI 被 prompt 注入后无限调用         │
├─────────────────────────────────────────┤
│ 第 7 层：审计日志                        │
│   每次执行的代码 + 输出全量记录           │
└─────────────────────────────────────────┘
```

**特别提醒**（联系 OWASP 安全原则）：
- **SSRF 防护**：沙箱内默认禁止访问 `9.*/10.*/11.*/21.*/30.*` 等内网网段
- **Secrets 隔离**：宿主机的 env 千万别透传进沙箱
- **反序列化**：AI 写的 `pickle.load(网络数据)` 是经典 RCE 入口

## 八、数据隐私：被忽视的安全维度

大家都在防"代码逃逸搞坏宿主机"，却忘了防"数据被合规地送出门"。这是 AI Agent 安全里**最容易被忽视、但其实最致命的一面**。

### 8.1 两类完全不同的安全问题

```
┌────────────────────────────────────────────────────┐
│  维度 A：执行隔离（防"代码破坏环境"）                │
│    问题：恶意代码会不会逃出沙箱、影响宿主机？         │
│    解决：Firecracker / gVisor / 容器                │
│    本质：保护"宿主"不被"代码"伤害                    │
│                                                     │
│  维度 B：数据合规（防"数据被合规带走"）              │
│    问题：我的数据进了沙箱，还算不算"我的"？           │
│    解决：私有化部署 / 数据加密 / 审计                │
│    本质：保护"数据"不被"环境"看见                    │
└────────────────────────────────────────────────────┘
```

Cube/E2B 的 microVM 完美解决了维度 A，但**维度 B 是另一个故事**——这也是为什么 Cube 这种"协议兼容 + 私有化"产品有市场。

### 8.2 数据流路：一份文件到底走了多远

**场景**：用户上传"员工薪酬表.xlsx"，让 Agent 分析

```
┌──────────────┐
│  用户浏览器   │ ① 上传文件
└──────┬───────┘
       │ HTTPS
       ▼
┌──────────────┐
│ Agent 后端    │ ② 文件落到服务器
│ (你的机房)    │
└──────┬───────┘
       │ ③ 调 LLM 推理
       │    └─→ 把"分析需求"发给 LLM ⚠️ 可能带敏感字段名
       │        OpenAI / Claude       ← 数据出境点 1
       │
       │ ④ LLM 生成代码
       │ ⑤ 上传文件到沙箱
       ▼
┌─────────────────────────────────┐
│  E2B 公有云（美国）              │  ← 数据出境点 2
│  Firecracker VM                  │  ← 全表数据落地
│  - 文件就在这里                  │
│  - Python 进程读取它              │
└──────┬──────────────────────────┘
       │ ⑥ 执行结果回传
       ▼
┌──────────────┐
│ Agent 后端    │ ⑦ 把结果再发给 LLM 总结  ← 数据出境点 3
└──────┬───────┘
       │ ⑧ 总结返回用户
       ▼
┌──────────────┐
│  用户浏览器   │
└──────────────┘
```

一份薪酬表，整个链路里至少**经过 3 个出境点**：LLM 推理、沙箱执行、结果回 LLM 总结。

### 8.3 隐私风险的三层暴露面

**层面 1：原始数据落到第三方服务器（最显眼）**

```
你把 customer.csv 上传到 E2B
→ 这份 CSV 物理上躺在 AWS 的某台服务器
→ 即使 VM 销毁了，磁盘数据在被覆盖前还能恢复
→ 即使加密了，密钥在服务商手里
```

合规风险：GDPR、《个人信息保护法》、金融/医疗/政府行业的强合规约束。

**层面 2：元数据泄漏（最容易忽视）**

就算只把分析结果回传——**结果本身就是数据**：

```python
print(df.groupby('部门').agg({'薪酬': ['mean', 'max']}))
# 输出："技术部 35000 80000" ← 已泄漏"技术部最高薪 80K"

print(df.head())   # 第一条记录可能就含身份证号
print(df.shape)    # "公司有 4327 个员工"也是商业秘密
```

**只要 LLM 看到了执行结果，敏感信息就进了 LLM 厂商的服务器**。

**层面 3：模型训练数据风险（最深的坑）**

```
你以为：和 OpenAI 签了 "no training" 协议，数据不会被训练
真实：
  - 协议覆盖范围有边界（ChatGPT free 默认会训练）
  - 缓存、日志、风控扫描可能保留数据
  - 模型记忆攻击：研究表明能"吐回"训练数据
```

### 8.4 生产级"五层防御"

```
┌────────────────────────────────────────────────────────┐
│  Level 1：基础（个人 / 演示）                           │
│    E2B 公有云 + OpenAI/Claude                          │
│    用户被告知"数据会经过云服务"                          │
│    适合：Demo、玩具项目、公开数据分析                    │
├────────────────────────────────────────────────────────┤
│  Level 2：脱敏（一般 B 端）                             │
│    数据进沙箱前脱敏：身份证号 → 哈希、姓名 → 编号        │
│    保留分析能力，结果回来后反脱敏映射                    │
│    适合：通用 SaaS、跨境团队                            │
├────────────────────────────────────────────────────────┤
│  Level 3：私有沙箱（企业内）                            │
│    沙箱用 Cube 替代 E2B → 数据不出公司网络              │
│    LLM 仍可用公有云（只发"任务描述"，不发原始数据）      │
│    适合：大部分企业内部 Agent                           │
├────────────────────────────────────────────────────────┤
│  Level 4：全私有化（金融 / 医疗 / 政企）                 │
│    沙箱：Cube 或自建 Firecracker                        │
│    LLM：私有部署（Qwen / DeepSeek / 混元）              │
│    整条链路不出公司网络 + 全量审计                       │
│    适合：强合规行业                                     │
├────────────────────────────────────────────────────────┤
│  Level 5：极致隔离（涉密 / 国家级）                     │
│    Level 4 全部 + 物理网络隔离                          │
│    沙箱内执行结果再次脱敏后才给 LLM                      │
│    适合：涉密、国家级场景                               │
└────────────────────────────────────────────────────────┘
```

### 9.2 协议兼容的威力

tRPC-Agent 没有独立的 `cube backend`——**它通过 `codeexecutor/e2b` 包同时支持 E2B 和 Cube**，靠的是 Cube 的协议兼容：

```go
// 用 E2B 公有云
ce, _ := e2b.New(
    e2b.WithAPIURL("https://api.e2b.dev"),
    e2b.WithAPIKey("e2b_xxxx"),
)

// 用 Cube 私有化（只改两行）
ce, _ := e2b.New(
    e2b.WithAPIURL("https://cube-sandbox.your-domain.internal"),
    e2b.WithAPIKey("cube_xxxx"),
)
```

**应用代码零改动，后端可以自由切换**——这就是协议兼容的威力。

## 十、Skills 和 CodeExecutor 的组合范式

这两个东西不是孤立的，它们**经常组合使用**：

```
═══════════════════════════════════════════════════
典型组合：数据分析 Skill + CodeExecutor 沙箱
═══════════════════════════════════════════════════

数据分析 Skill (data-analysis/SKILL.md)：
  ────────────────────────────────────────
  ## 何时使用
  用户上传 CSV/Excel 并要求统计分析时

  ## 工作流程
  1. 把用户文件放到 work/inputs/
  2. 用 CodeExecutor 执行 pandas 代码
  3. 把图表保存到 out/
  4. 返回 out/ 下的产物

CodeExecutor (e2b backend)：
  ────────────────────────────────────────
  接收 SKILL.md 引导下生成的 pandas 代码
  在 Cube/E2B microVM 里跑
  返回 stdout + 文件产物
```

这就是 **ChatGPT Advanced Data Analysis / Claude Analysis Tool 的核心架构**——Skill 是"工作手册"，CodeExecutor 是"执行引擎"，Workspace 是"工作台"。

## 十一、本章小结

### 11.1 核心概念回顾

```
✅ Skill：可复用的"工作手册"
   - SKILL.md = 描述协议（name/description/location + 正文）
   - Progressive Disclosure = 渐进式披露，省 context
   - Skill vs Tool vs MCP：手册 vs 刀片 vs 外部服务

✅ CodeExecutor：让 AI 现场写代码并执行
   - 解决 LLM 三大缺陷：算数不准 / 上下文有限 / 无副作用
   - Agent 副作用边界中"开放性最强、风险最大"的一环
   - 应用场景：数据分析 / 数学计算 / 文件处理 / 自修正 / 动态工具

✅ 五种执行环境（安全谱系）
   本地执行 < Workspace Runtime < 容器 < Cube < E2B

✅ Workspace Runtime：执行环境的标准化抽象
   - 三件事：目录约定 + Executor 装载 + 生命周期管理
   - 解决文件路径、产物收集、多轮状态、环境复用等 6 个痛点

✅ 容器执行的工程化
   - 预 build 镜像 + 卷挂载交换 + 双层超时
   - 安全靠 Linux 四件套：Namespaces / Cgroups / Capabilities / Seccomp
   - 内核级沙箱：gVisor（用户态假内核）/ Firecracker（微 VM）/ Kata

✅ 镜像策略
   - 场景化镜像（主流）+ 四层缺包兜底 + 灰度发布 + 容器池预热
   - 系统越用越快的"自愈"演进流水线

✅ 七层防御 + 数据隐私五级防御
   - 不只防"代码逃出沙箱"，更要防"数据被合规带走"
   - Level 1～5 按合规等级递增

✅ tRPC-Agent-Go 的实现
   - 4 个 backend：local / container / jupyter / e2b
   - 通过 E2B 协议兼容同时支持 E2B 公有云和 Cube 私有化
```

### 11.2 一句话本章总结

> **Skills 是给 AI 的"工作手册"，CodeExecutor 是 AI 的"动手能力"，Workspace Runtime 是 AI 的"标准化工作台"。三者组合，让 LLM 从"会聊天"进化到"能干活"——而所有这些能力的边界，都靠"执行环境的隔离强度 + 数据流转的合规设计"来保证。**

### 11.3 进阶思考题

1. 如果让你设计一个企业内的 CodeExecutor 服务，5 种执行环境你会选哪个？为什么？
2. 你写过 SKILL.md 吗？如果让你写一个"数据库 SQL 执行" Skill，你的 description 会怎么写才能让 AI 准确发现它？
3. Workspace 的 PerSession 持久化策略，会带来哪些安全风险？怎么平衡复用效率和数据清理？
4. 假设 AI 生成的代码在沙箱里执行成功了，但返回的是错误结果（比如算错了），如何实现"自修正"闭环？
5. Cube 和 E2B 协议兼容，意味着客户可以无痛迁移——这对 SaaS 厂商来说是机会还是威胁？
