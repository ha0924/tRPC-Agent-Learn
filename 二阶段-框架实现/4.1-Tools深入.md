## 二、FunctionTool 高级用法

### 2.1 自动 Schema 推断

框架最常用的方式——用 Python 类型注解自动生成 JSON Schema：

```python
from trpc_agent_sdk import FunctionTool

def get_weather(city: str, unit: str = "celsius") -> str:
    """获取指定城市的天气信息
    
    Args:
        city: 城市名称，如"北京"、"上海"
        unit: 温度单位，可选 "celsius" 或 "fahrenheit"，默认 celsius
    
    Returns:
        天气信息文本
    """
    # 实际实现...
    return f"{city}今天 28°C，多云"

# 框架自动从类型注解 + docstring 生成 JSON Schema
tool = FunctionTool(get_weather)
```

**自动推断的规则**：

```
函数签名         →  JSON Schema 映射
──────────       ──────────────────
参数名           →  properties 的 key
类型注解 str     →  {"type": "string"}
类型注解 int     →  {"type": "integer"}
类型注解 float   →  {"type": "number"}
类型注解 bool    →  {"type": "boolean"}
类型注解 list    →  {"type": "array"}
类型注解 dict    →  {"type": "object"}
默认值           →  标记为 optional（不在 required 里）
无默认值         →  标记为 required
docstring        →  description 字段（函数描述 + 参数描述）
```

### 2.2 手动 Schema 声明

当自动推断不够精确时（比如枚举值、嵌套对象），手动声明更可控：

```python
tool = FunctionTool(
    name="search_documents",
    description="在知识库中搜索相关文档",
    parameters={
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": "搜索关键词"
            },
            "top_k": {
                "type": "integer",
                "description": "返回结果数量",
                "default": 5,
                "minimum": 1,
                "maximum": 20
            },
            "source": {
                "type": "string",
                "description": "数据源",
                "enum": ["internal_docs", "confluence", "github"]
            }
        },
        "required": ["query"]
    },
    func=search_documents_impl
)
```

### 2.3 手动 vs 自动的选择

```
✅ 用自动推断：
  · 参数简单（str/int/bool/list）
  · docstring 描述清楚
  · 快速原型开发

✅ 用手动声明：
  · 需要 enum 枚举值约束
  · 需要 minimum/maximum 数值范围
  · 参数是嵌套对象（dict 里套 dict）
  · 需要精细控制 description 内容（LLM 靠 description 决定何时调用）
  · 生产环境推荐手动——description 写得好不好直接影响工具调用准确率
```

### 2.4 异步工具

对于需要 I/O 操作的工具（网络请求、数据库查询），应使用 async：

```python
import httpx

async def fetch_stock_price(symbol: str) -> str:
    """获取实时股票价格
    
    Args:
        symbol: 股票代码，如 "AAPL"
    """
    async with httpx.AsyncClient() as client:
        resp = await client.get(f"https://api.example.com/stock/{symbol}")
        data = resp.json()
        return f"{symbol} 当前价格: ${data['price']}"

tool = FunctionTool(fetch_stock_price)
```

**关键点**：
- 框架内部使用 `asyncio` 事件循环，异步工具不会阻塞其他操作
- 如果工具函数是同步的，框架会自动包装为异步执行
- **生产环境推荐所有 I/O 密集型工具都用 async**

## 三、MCPToolset：外部工具服务集成

### 3.1 一阶段回顾

一阶段你已知 MCP 的架构：

```
Agent 框架 ─MCP Client─▶ MCP Server（独立进程）
                          暴露 Tools/Resources/Prompts
```

二阶段要学的是**框架怎么集成 MCP Server**。

### 3.2 连接本地 MCP Server（stdio 模式）

```python
from trpc_agent_sdk import MCPToolset

# 连接一个本地 MCP Server（通过 stdio 管道通信）
mcp_tools = MCPToolset(
    server_params={
        "command": "python",
        "args": ["path/to/mcp_server.py"],
    }
)

# mcp_tools 会自动调用 tools/list 获取所有工具的 schema
# 然后包装为 FunctionTool 列表，可以直接注册到 Agent
```

### 3.3 连接远程 MCP Server（HTTP+SSE 模式）

```python
mcp_tools = MCPToolset(
    server_params={
        "url": "https://mcp-server.example.com/sse",
        "transport": "sse",
    }
)
```

### 3.4 多 MCP Server 组合

一个 Agent 可以同时连接多个 MCP Server，工具来自不同服务：

```python
from trpc_agent_sdk import LlmAgent, MCPToolset

# 文件系统工具
fs_tools = MCPToolset(server_params={"command": "mcp-server-filesystem", "args": ["/workspace"]})

# 数据库工具
db_tools = MCPToolset(server_params={"command": "mcp-server-postgres", "args": ["--conn", db_url]})

# 搜索工具
search_tools = MCPToolset(server_params={"url": "https://search-mcp.example.com/sse", "transport": "sse"})

agent = LlmAgent(
    name="multi_tool_agent",
    model=model,
    instruction="你是一个多功能助手...",
    tools=[
        *fs_tools.get_tools(),      # 文件系统工具
        *db_tools.get_tools(),       # 数据库工具
        *search_tools.get_tools(),   # 搜索工具
    ]
)
```

### 3.5 MCPToolset 的连接生命周期

```
┌────────────────────────────────────────────────┐
│  MCPToolset 的连接管理                           │
├────────────────────────────────────────────────┤
│                                                 │
│  1. 初始化时：                                   │
│     · 启动 MCP Server 进程（stdio）或建立连接    │
│     · 发送 initialize 请求，协商协议版本         │
│     · 调用 tools/list 获取工具列表               │
│     · 把每个工具的 schema 包装为 FunctionTool    │
│                                                 │
│  2. 运行时：                                     │
│     · LLM 决定调用某工具 → 框架路由到对应 MCP    │
│     · MCP Client 发送 tools/call 请求           │
│     · 接收执行结果 → 回传给 LLM                  │
│                                                 │
│  3. 清理时：                                     │
│     · 关闭连接                                   │
│     · stdio 模式下终止子进程                     │
│                                                 │
│  ⚠️ 注意事项：                                    │
│     · MCP Server 进程崩溃 → 需要重连机制         │
│     · 远程 Server 超时 → 需要超时配置            │
│     · 多 Server 时注意工具名冲突                 │
└────────────────────────────────────────────────┘
```

## 四、Streaming Tool（流式工具）

### 4.1 什么时候需要流式工具

```
普通工具：执行完才返回全部结果
  用户等 5 秒 → 突然出来一大段 → 体验差

流式工具：边执行边返回增量结果
  用户实时看到进度 → 体验好

适用场景：
  · 长文本生成（翻译一篇长文章，边翻边出）
  · 数据流处理（逐行处理 CSV，实时显示进度）
  · 搜索结果逐步返回
  · 代码执行实时输出 stdout
```

### 4.2 流式工具的实现方式

```python
from trpc_agent_sdk import FunctionTool
from typing import AsyncGenerator

async def stream_translate(text: str, target_lang: str) -> AsyncGenerator[str, None]:
    """逐句翻译长文本，流式返回每一句的翻译结果
    
    Args:
        text: 需要翻译的原文
        target_lang: 目标语言，如 "en"、"zh"
    """
    sentences = text.split("。")
    for i, sentence in enumerate(sentences):
        translated = await call_translation_api(sentence, target_lang)
        yield f"[{i+1}/{len(sentences)}] {translated}\n"

tool = FunctionTool(stream_translate, is_streaming=True)
```

### 4.3 流式工具和 AG-UI 事件的关系

```
流式工具的每次 yield：
  → 框架转换为 TOOL_CALL_RESULT 事件（增量）
  → 通过 SSE 推给前端
  → 前端实时渲染

完整事件序列：
  TOOL_CALL_START       → 工具开始执行
  TOOL_CALL_ARGS        → 参数（可能也是流式的）
  TOOL_CALL_RESULT (1)  → 第一个增量结果
  TOOL_CALL_RESULT (2)  → 第二个增量结果
  ...
  TOOL_CALL_END         → 工具执行完毕
```

## 五、Agent-as-Tool（Agent 包装为工具）

### 5.1 核心思想

一阶段 3.2 学过 TeamAgent 的"方式 B：把每个成员封装成 Tool"——这就是 Agent-as-Tool 的本质。

```
普通 Tool：
  LLM 调用 → 执行一个函数 → 返回结果

Agent-as-Tool：
  LLM 调用 → 启动一个完整的子 Agent → 子 Agent 自己做 LLM 推理 + 工具调用 → 返回最终结果

对调用方的 LLM 来说，它只看到一个"工具"，
不知道（也不需要知道）这个工具内部是另一个 Agent。
```

### 5.2 使用方式

```python
from trpc_agent_sdk import LlmAgent, AgentTool

# 专门做研究的子 Agent
researcher = LlmAgent(
    name="researcher",
    model=model,
    instruction="你是一个研究助手，擅长信息检索和总结",
    tools=[search_tool, web_fetch_tool]
)

# 专门写代码的子 Agent
coder = LlmAgent(
    name="coder",
    model=model,
    instruction="你是一个编程助手，擅长写高质量代码",
    tools=[code_executor_tool, file_tool]
)

# 主 Agent 把子 Agent 当工具用
main_agent = LlmAgent(
    name="coordinator",
    model=model,
    instruction="你是一个协调者，根据用户需求调用合适的专家",
    tools=[
        AgentTool(researcher, description="调用研究员做信息检索和总结"),
        AgentTool(coder, description="调用程序员写代码或解决技术问题"),
    ]
)
```

### 5.3 Agent-as-Tool vs TeamAgent

```
Agent-as-Tool（工具化）：
  · 主 Agent 用标准 Function Calling 调用子 Agent
  · 子 Agent 执行完返回单次结果
  · 适合：子 Agent 是"一次性任务执行者"
  · 实现简单，和普通 Tool 用法一致

TeamAgent（团队化）：
  · 队长 Agent 可以多轮调度成员
  · 成员之间可能有交互
  · 适合：复杂任务需要多轮协作

实战建议：
  · 80% 场景用 Agent-as-Tool 就够了
  · 只有需要多轮协作时才用 TeamAgent
```

## 六、Long Running Tool（长耗时工具）

### 6.1 问题场景

```
普通工具调用：LLM 等工具执行完 → 继续推理
  工具执行 100ms → 没问题

长耗时工具：执行可能需要几分钟甚至更久
  · 大数据量分析
  · 远程 API 调用（A2A 任务）
  · 代码编译和测试
  · 文件批量处理

如果 LLM 一直等 → 超时 → 失败
```

### 6.2 异步执行模式

```
同步工具：
  LLM → 调用工具 → 等待 → 拿到结果 → 继续推理

异步工具：
  LLM → 调用工具 → 立即返回"任务ID" → LLM 继续做其他事
  → 后续轮次 LLM 调 check_status(task_id) 查进度
  → 完成后 LLM 调 get_result(task_id) 拿结果
```

### 6.3 实现方式

```python
import asyncio
from trpc_agent_sdk import FunctionTool

# 长耗时任务的实际执行
async def _run_analysis(data_path: str) -> dict:
    # 模拟耗时操作
    await asyncio.sleep(60)  # 实际可能是数据分析
    return {"result": "分析完成", "summary": "..."}

# 任务存储
_tasks = {}

async def start_analysis(data_path: str) -> str:
    """启动数据分析任务（异步执行）
    
    Args:
        data_path: 数据文件路径
    
    Returns:
        任务 ID，可用于查询进度
    """
    task_id = f"task_{len(_tasks) + 1}"
    _tasks[task_id] = asyncio.create_task(_run_analysis(data_path))
    return f"分析任务已启动，任务ID: {task_id}，请稍后查询进度"

async def check_analysis(task_id: str) -> str:
    """查询分析任务进度
    
    Args:
        task_id: 任务 ID
    """
    if task_id not in _tasks:
        return "任务不存在"
    task = _tasks[task_id]
    if task.done():
        result = task.result()
        return f"任务完成：{result['summary']}"
    return "任务进行中，请稍后再查"

# 注册为两个配套工具
tools = [
    FunctionTool(start_analysis),
    FunctionTool(check_analysis),
]
```

## 七、ToolFilter（工具拦截器）

### 7.1 什么是 ToolFilter

```
Filter 是 tRPC-Agent 框架的核心扩展机制（详见模块 6）。
ToolFilter 是专门拦截工具调用的 Filter。

拦截时机：
  LLM 决定调用工具
       │
       ▼
  ToolFilter.before()    ← 调用前拦截
       │
       ▼
  工具实际执行
       │
       ▼
  ToolFilter.after()     ← 调用后拦截
       │
       ▼
  结果回传 LLM
```

### 7.2 典型使用场景

```
场景 1：日志记录
  before: 记录"即将调用 xxx 工具，参数为 yyy"
  after:  记录"调用完成，耗时 zms，结果 www"

场景 2：权限控制
  before: 检查当前用户是否有权限调用该工具
         如果无权限 → 短路返回"无权限"，跳过实际执行

场景 3：参数校验/修改
  before: 检查参数合法性，或自动补充默认值

场景 4：结果过滤
  after: 对工具返回结果做脱敏处理（隐藏敏感信息）

场景 5：限流
  before: 检查该工具的调用频率，超限则拒绝

场景 6：缓存
  before: 相同参数的调用，直接返回缓存结果
  after:  把结果写入缓存
```

### 7.3 伪代码示意

```python
from trpc_agent_sdk import ToolFilter

class LoggingToolFilter(ToolFilter):
    async def before(self, tool_name: str, params: dict, context):
        """工具调用前"""
        print(f"[Tool] 即将调用: {tool_name}, 参数: {params}")
        # 返回 None 表示继续执行
        # 返回具体值表示短路（跳过工具执行，直接用这个值作为结果）
        return None

    async def after(self, tool_name: str, params: dict, result, context):
        """工具调用后"""
        print(f"[Tool] 调用完成: {tool_name}, 结果长度: {len(str(result))}")
        # 可以修改 result 后返回
        return result

class PermissionToolFilter(ToolFilter):
    async def before(self, tool_name: str, params: dict, context):
        """权限检查"""
        user = context.get("user")
        if tool_name == "delete_data" and not user.is_admin:
            return "抱歉，您没有权限执行删除操作"  # 短路返回
        return None
```

### 7.4 Filter 链式调用

多个 ToolFilter 按注册顺序依次执行：

```
注册顺序：[LoggingFilter, PermissionFilter, CacheFilter]

执行顺序：
  LoggingFilter.before()
    → PermissionFilter.before()
      → CacheFilter.before()
        → 工具实际执行
      → CacheFilter.after()
    → PermissionFilter.after()
  → LoggingFilter.after()

任何 before() 返回非 None → 短路，后续 Filter 和工具都不执行
```

## 八、工具调用的完整内部流程

### 8.1 从 LLM 输出到工具执行的全链路

```
┌─────────────────────────────────────────────────────────┐
│  工具调用的完整内部流程                                    │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ① LLM 输出 function_call                               │
│     {"name": "get_weather", "arguments": {"city": "北京"}}│
│                                                          │
│  ② Runner 解析 function_call                             │
│     · 从 LLM response 提取工具名和参数                   │
│     · 可能有多个并行工具调用（parallel function calling） │
│                                                          │
│  ③ 匹配工具实例                                          │
│     · 在已注册的 tools 列表里找 name == "get_weather"     │
│     · 找不到 → 返回错误信息给 LLM                        │
│                                                          │
│  ④ ToolFilter.before() 链式执行                          │
│     · 日志、权限、缓存等前置拦截                         │
│     · 任何一个返回非 None → 短路跳过 ⑤                   │
│                                                          │
│  ⑤ 工具函数执行                                          │
│     · 反序列化参数（JSON → Python 类型）                  │
│     · 调用实际函数 get_weather(city="北京")               │
│     · 捕获异常 → 转为错误信息                            │
│                                                          │
│  ⑥ ToolFilter.after() 链式执行                           │
│     · 日志、结果过滤等后置处理                           │
│                                                          │
│  ⑦ 结果格式化                                            │
│     · 序列化为字符串（LLM 只能接收文本）                  │
│     · 构造 tool_result message                           │
│                                                          │
│  ⑧ 回传 LLM                                             │
│     · 把 tool_result 追加到 messages                     │
│     · 再次调用 LLM → LLM 看到工具结果后继续推理          │
│                                                          │
│  ⑨ 事件发射                                              │
│     · 框架同时发射 AG-UI 事件                             │
│     · TOOL_CALL_START / TOOL_CALL_RESULT / TOOL_CALL_END │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 8.2 并行工具调用（Parallel Function Calling）

```
部分 LLM（如 GPT-4）支持一次输出多个工具调用：

LLM 输出：
  function_call_1: get_weather(city="北京")
  function_call_2: get_weather(city="上海")
  function_call_3: get_stock("AAPL")

框架处理：
  · 解析出 3 个工具调用
  · 并行执行（asyncio.gather）
  · 等全部完成后，把 3 个结果一起回传 LLM
  · LLM 在一个上下文里看到 3 个结果，综合生成回答

优势：减少 LLM 调用轮次，降低延迟
```

## 九、工具设计最佳实践

### 9.1 description 是最重要的字段

```
LLM 判断"何时调用这个工具"的唯一依据 = description

差的 description：
  "查天气"
  → LLM 不知道能查哪些城市、返回什么格式、什么时候该用

好的 description：
  "获取中国各城市的实时天气信息，包括温度、湿度、天气状况。
   适合用户询问'天气怎么样'、'需不需要带伞'等场景。
   参数 city 接受中文城市名（如'北京'），不支持国际城市。"
  → LLM 能准确判断何时调用、如何传参
```

### 9.2 工具数量的 Trade-off

```
工具太少（< 3）：Agent 能力受限
工具太多（> 15）：LLM 工具选择准确率下降

经验值：
  · 简单 Agent：3~5 个工具
  · 中型 Agent：5~10 个工具
  · 超过 10 个：考虑用 Agent-as-Tool 分层

如果工具 > 15 个 → 用 ToolFilter 动态筛选：
  根据用户意图，只把最相关的 5~8 个工具传给 LLM
```

### 9.3 工具返回值的设计

```
差的返回值：
  return json.dumps(huge_database_result)  # 返回 5000 行 JSON
  → 吃掉大量 token、LLM 被信息淹没

好的返回值：
  return f"共找到 {count} 条记录。前 3 条：\n{summary}"
  → 精简、LLM 能理解、token 成本低

原则：
  · 返回值应该是 LLM 可理解的自然语言或简短结构化数据
  · 大结果集做摘要或分页
  · 包含足够信息让 LLM 做下一步决策
```

## 十一、知识沉淀小结

| 类别 | 关键点 |
|------|--------|
| **FunctionTool** | 自动 schema 推断 vs 手动声明；异步工具用 async；生产环境推荐手动声明 |
| **MCPToolset** | 连接管理（stdio/SSE）；多 Server 组合；注意工具名冲突 |
| **Streaming Tool** | 用 AsyncGenerator yield 增量结果；和 AG-UI 事件联动 |
| **Agent-as-Tool** | 子 Agent 包装为工具；80% 场景比 TeamAgent 更简单 |
| **Long Running Tool** | 异步执行模式：启动 → 查进度 → 取结果；配套两个工具 |
| **ToolFilter** | before/after 拦截；链式调用；短路机制；日志/权限/缓存 |
| **工具调用流程** | LLM 输出 → Runner 解析 → 匹配工具 → Filter → 执行 → Filter → 回传 |
| **最佳实践** | description 是关键；工具数量 5~10；返回值要精简 |
