## 二、Session 后端实现

### 2.1 三种后端的 API 对比

```python
# ========== InMemory 后端（开发/测试）==========
from trpc_agent_sdk.session import InMemorySessionBackend

session_backend = InMemorySessionBackend()
# 不需要任何配置，数据存在内存里，进程重启就丢失

# ========== Redis 后端（生产/高并发）==========
from trpc_agent_sdk.session import RedisSessionBackend

session_backend = RedisSessionBackend(
    redis_url="redis://localhost:6379/0",
    key_prefix="agent_session:",     # Redis key 前缀
    ttl=3600,                        # 会话过期时间（秒）
)

# ========== SQL 后端（持久化/审计）==========
from trpc_agent_sdk.session import SQLSessionBackend

session_backend = SQLSessionBackend(
    db_url="postgresql://user:pass@localhost:5432/agent_db",
    table_name="agent_sessions",     # 数据库表名
)
```

### 2.2 Session 的核心 API

```python
# Session 对象的核心方法

# 1. 加载会话（Runner 自动调用）
session = await session_backend.load(session_id="sess_001", user_id="user_001")

# 2. 追加消息
session.append_message({"role": "user", "content": "你好"})
session.append_message({"role": "assistant", "content": "你好！有什么可以帮你？"})

# 3. 获取完整消息历史
messages = session.get_messages()
# → [{"role": "user", ...}, {"role": "assistant", ...}]

# 4. 获取/设置状态
session.set_state("intent", "技术咨询")
intent = session.get_state("intent")

# 5. 保存（Runner 每轮结束自动调用）
await session_backend.save(session)

# 6. Token 用量统计
usage = session.get_token_usage()
# → {"total_input_tokens": 1200, "total_output_tokens": 450}
```

## 三、Session Summary（会话摘要）

### 3.1 为什么需要摘要

```
问题：
  多轮对话 → 消息历史越来越长 → 每次 LLM 调用的 input tokens 越来越多
  → token 成本爆炸 + 可能超出上下文窗口

解决：
  对早期的消息做"摘要"，用几百 token 的摘要替代几千 token 的原文
```

### 3.2 摘要机制

```
┌─────────────────────────────────────────────────────┐
│  Session Summary 的工作方式                           │
├─────────────────────────────────────────────────────┤
│                                                      │
│  触发条件（满足任一）：                                │
│    · 消息总 token 数超过阈值（如 4000 tokens）        │
│    · 消息条数超过阈值（如 20 条）                     │
│                                                      │
│  摘要过程：                                           │
│    1. 取前 N 条消息（保留最近 M 条不动）               │
│    2. 把这 N 条消息喂给 LLM，生成摘要                 │
│    3. 用摘要替换原始的 N 条消息                       │
│    4. 最终 messages = [摘要] + [最近 M 条原文]         │
│                                                      │
│  示意：                                               │
│    原始：[msg1, msg2, ..., msg15, msg16, ..., msg20] │
│    摘要后：[summary_of_1_to_15, msg16, ..., msg20]   │
│                                                      │
│  关键配置项：                                         │
│    · max_tokens_before_summary: 触发摘要的 token 阈值│
│    · keep_recent_messages: 保留最近多少条不摘要       │
│    · summary_model: 用什么模型做摘要（可以用便宜的）  │
│    · summary_prompt: 摘要时的 system prompt           │
└─────────────────────────────────────────────────────┘
```

### 3.3 摘要的 Trade-off

```
摘要太激进（keep_recent 太小）：
  · 丢失上下文细节
  · 用户追问"你刚才说的第二点"→ Agent 不记得了

摘要太保守（keep_recent 太大）：
  · 省不了多少 token
  · 还是会超出窗口

经验值：
  · keep_recent_messages = 6~10
  · max_tokens_before_summary = 模型窗口的 50%
  · summary_model 可以用比主模型便宜的（如 GPT-3.5 做摘要）
```

## 四、Memory Service 框架 API

### 4.1 Memory 的三个核心接口

```python
from trpc_agent_sdk.memory import MemoryService

memory = MemoryService(
    embedding_model=embedding_model,  # 向量化模型
    vector_store=vector_store,         # 向量存储后端
)

# ========== 1. 写入记忆 ==========
await memory.write(
    user_id="user_001",
    content="该用户是 Go 开发者，主要使用 tRPC 框架",
    metadata={"type": "preference.language", "source": "explicit"}
)

# ========== 2. 搜索/召回记忆 ==========
results = await memory.search(
    user_id="user_001",
    query="这个用户用什么编程语言？",
    top_k=5,                    # 返回最相关的 5 条
    score_threshold=0.7,        # 相似度阈值（0~1），低于此分数的不返回
)
# results: [
#   {"content": "该用户是 Go 开发者...", "score": 0.92, "metadata": {...}},
#   ...
# ]

# ========== 3. 读取所有记忆（按 user_id）==========
all_memories = await memory.read(user_id="user_001")
```

### 4.2 Memory 的配置项

```
关键配置：

embedding_model：
  · 和 Knowledge 必须用同一个 embedding 模型
  · 常用：text-embedding-3-small / BGE / M3E

vector_store：
  · 开发：ChromaDB（本地文件）
  · 生产：Milvus / Qdrant / PGVector

top_k：
  · Memory 一般 3~5 就够（用户画像条数少）
  · 和 Knowledge 的 top_k（5~10）不同

score_threshold：
  · 太低（0.5）→ 召回不相关记忆，污染 Prompt
  · 太高（0.9）→ 漏掉相关记忆
  · 建议 0.7 起步，根据实测调整

Memory 和 Knowledge 的检索差异：
  Memory：  按 user_id 过滤 → 在过滤后的子集做语义检索
  Knowledge：不按用户过滤 → 全局语义检索
```

### 4.3 隐式写入（自动沉淀）

```
除了显式调用 memory.write()，框架还支持自动沉淀：

配置方式：
  agent = LlmAgent(
      ...
      memory_service=memory,
      auto_memory_extract=True,          # 开启自动沉淀
      memory_extract_interval="session", # 每个 session 结束后提取
  )

内部流程：
  Session 结束 → 框架把整段对话喂给 LLM
  → LLM 按 memory_extract_prompt 提取值得记忆的信息
  → 自动调用 memory.write() 写入
  → 写入前会做冲突检测（向量召回相似记忆 + LLM 判断是否矛盾）
```

## 五、Knowledge 组件链

### 5.1 组件链全景

```
原始文档
   │
   ▼
┌──────────────┐
│ DocumentLoader│  ← 加载文档（PDF/Word/Markdown/网页）
└──────┬───────┘
       │ 纯文本
       ▼
┌──────────────┐
│ TextSplitter  │  ← 切分文本（按段落/标题/递归）
└──────┬───────┘
       │ chunks[]
       ▼
┌──────────────┐
│   Embedder    │  ← 向量化（调 Embedding 模型）
└──────┬───────┘
       │ vectors[]
       ▼
┌──────────────┐
│ VectorStore   │  ← 存储（Chroma/Milvus/FAISS）
└──────────────┘

用户查询
   │
   ▼
┌──────────────┐
│  Retriever    │  ← 检索（向量检索 + 可选 Rerank）
└──────┬───────┘
       │ top_k chunks
       ▼
┌──────────────┐
│PromptTemplate│  ← 拼接上下文（把检索结果塞进 Prompt）
└──────────────┘
```

### 5.2 各组件的 API 示意

```python
from trpc_agent_sdk.knowledge import (
    DocumentLoader,
    MarkdownLoader,
    PDFLoader,
    RecursiveTextSplitter,
    Embedder,
    ChromaVectorStore,
    Retriever,
    KnowledgeBase,
)

# ========== 1. 文档加载 ==========
loader = MarkdownLoader()
documents = loader.load("docs/trpc_guide.md")
# documents: [Document(content="...", metadata={"source": "trpc_guide.md", ...})]

# PDF 加载器
pdf_loader = PDFLoader()
pdf_docs = pdf_loader.load("docs/trpc_api.pdf")

# ========== 2. 文本切分 ==========
splitter = RecursiveTextSplitter(
    chunk_size=500,          # 每个 chunk 最大 500 字符
    chunk_overlap=50,        # 相邻 chunk 重叠 50 字符
    separators=["\n## ", "\n### ", "\n\n", "\n", "。", ""],  # 递归切分分隔符
)
chunks = splitter.split(documents)
# chunks: [Chunk(content="...", metadata={...}), ...]

# ========== 3. 向量化 + 存储 ==========
embedder = Embedder(model="text-embedding-3-small")
vector_store = ChromaVectorStore(collection_name="trpc_knowledge")

# 批量向量化并存储
await vector_store.add(chunks, embedder=embedder)

# ========== 4. 检索器 ==========
retriever = Retriever(
    vector_store=vector_store,
    embedder=embedder,
    top_k=5,
    score_threshold=0.7,
    # reranker=BGEReranker(),  # 可选：精排
)

# ========== 5. 组装为 KnowledgeBase ==========
kb = KnowledgeBase(
    retriever=retriever,
    prompt_template="""
参考资料：
{context}

请基于以上参考资料回答用户问题。如果参考资料中没有相关信息，请直接说"我不知道"。
""",
)
```

### 5.3 Knowledge 和 Agent 的集成

```python
agent = LlmAgent(
    name="knowledge_agent",
    model=model,
    instruction="你是 tRPC 技术助手",
    knowledge_base=kb,    # 直接挂载知识库
)

# Agent 每次收到用户问题时：
# 1. 自动调 retriever.search(query) 检索相关文档
# 2. 用 prompt_template 把检索结果拼进 Prompt
# 3. 发给 LLM 生成回答
```

### 5.4 自定义组件

```python
# 自定义 DocumentLoader（比如加载数据库中的知识）
class DBDocumentLoader(DocumentLoader):
    def __init__(self, db_connection):
        self.db = db_connection
    
    async def load(self, query: str) -> list[Document]:
        rows = await self.db.execute(query)
        return [
            Document(
                content=row["content"],
                metadata={"id": row["id"], "source": "database"}
            )
            for row in rows
        ]

# 自定义 VectorStore（比如用 Milvus）
class MilvusVectorStore(VectorStore):
    async def add(self, chunks, embedder):
        vectors = await embedder.embed_batch([c.content for c in chunks])
        await self.milvus_client.insert(vectors, [c.content for c in chunks])
    
    async def search(self, query_vector, top_k):
        results = await self.milvus_client.search(query_vector, top_k=top_k)
        return results
```

## 六、知识沉淀小结

| 类别 | 关键点 |
|------|--------|
| **Session 后端** | InMemory（开发）/ Redis（高并发）/ SQL（审计）；API 统一 |
| **Session Summary** | token 超阈值触发；保留最近 N 条不摘要；可用便宜模型做摘要 |
| **Memory API** | write（写入）/ search（语义召回）/ read（全量读取）；按 user_id 隔离 |
| **Memory 配置** | top_k 3~5；score_threshold 0.7 起步；支持自动沉淀 |
| **Knowledge 组件链** | Loader → Splitter → Embedder → VectorStore → Retriever → PromptTemplate |
| **集成方式** | agent.knowledge_base = kb，自动检索拼接 |
| **自定义** | 每个组件都可继承基类自定义实现 |
